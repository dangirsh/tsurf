# Plan 34-01: Enable HA MCP Server + Tailscale Serve HTTPS Proxy

## Goal

Expose Home Assistant's built-in MCP Server integration over Tailscale Serve
HTTPS so the Claude Android app (voice mode) can control lights and query the
CO2 sensor. Only one NixOS file changes (`modules/home-assistant.nix`). The HA
MCP integration and entity filtering are UI-only steps requiring human
interaction. No new secrets, no new modules, no new ports.

## Decisions Made (from Research + User Decisions)

- **Access model**: Tailscale Serve HTTPS at `https://neurosys.<tailnet>.ts.net`
  (phone has Tailscale installed). No public internet exposure.
- **Entity scope**: Lights + CO2 sensor only. Filtered via HA Expose UI.
- **Tailscale Serve**: Declarative NixOS systemd oneshot wrapping
  `tailscale serve --bg`. Survives reboots and deploys.
- **HA token**: `ha-token` already exists in sops (commit 81efc66). No new
  secret needed.
- **MCP setup**: UI-only config flow (HA 2025.11.3). Cannot be enabled
  declaratively. Documented as human checkpoint.
- **Auth**: Bearer token (Option B). HA long-lived token pasted into Claude
  connector. Simpler than OAuth for Tailscale-only endpoint.
- **CO2 sensor**: Prerequisite -- must be paired in HA before MCP can surface it.
  Lights work immediately.
- **Single file change**: All NixOS config goes in `modules/home-assistant.nix`
  (< 20 lines added, no new module justified).

## must_haves

1. `modules/home-assistant.nix` has `"mcp_server"` in `extraComponents`
2. `modules/home-assistant.nix` has `use_x_forwarded_for = true` and `trusted_proxies = ["127.0.0.1"]` in the `http` block
3. `systemd.services.tailscale-serve-ha` exists as a oneshot service proxying HTTPS 443 to 127.0.0.1:8123
4. The tailscale-serve-ha service starts after `tailscaled.service` and is in `multi-user.target`
5. `nix flake check` passes for all configurations (neurosys + ovh)
6. After deploy: `tailscale serve status` on neurosys shows HTTPS 443 -> 127.0.0.1:8123
7. After HA UI setup: `curl -H "Authorization: Bearer <token>" https://neurosys.<tailnet>.ts.net/api/mcp` returns a valid MCP response
8. Claude Android app can toggle a light via voice
9. Claude Android app can read CO2 sensor via voice (after sensor is paired)

    <title>Add tailscale-serve-ha systemd service</title>
    <description>
      Edit `/data/projects/neurosys/modules/home-assistant.nix`.

      Add the following systemd service definition at the end of the module
      (inside the outer attribute set, after the `system.activationScripts`
      block):

      ```nix
      # --- Tailscale Serve: HTTPS proxy to HA for MCP ---
      # @decision HA-05: Declarative systemd oneshot for tailscale serve --bg
      # @rationale: --bg persistence survives reboots (config stored in
      #   /var/lib/tailscale), but a systemd wrapper ensures the config is
      #   applied on every deploy and follows NixOS declarative convention.
      #   Oneshot + RemainAfterExit = runs once, stays "active".
      systemd.services.tailscale-serve-ha = {
        description = "Tailscale Serve: HTTPS proxy to Home Assistant MCP";
        after = [ "tailscaled.service" ];
        wants = [ "tailscaled.service" ];
        wantedBy = [ "multi-user.target" ];

        serviceConfig = {
          Type = "oneshot";
          RemainAfterExit = true;
          # Wait for Tailscale to be fully authenticated before configuring serve.
          # tailscaled.service may report ready before auth completes.
          ExecStartPre = "${pkgs.bash}/bin/bash -c 'for i in $(seq 1 30); do ${pkgs.tailscale}/bin/tailscale status --json 2>/dev/null | ${pkgs.jq}/bin/jq -e \".BackendState == \\\"Running\\\"\" >/dev/null 2>&1 && exit 0; sleep 2; done; echo \"tailscale not ready after 60s\"; exit 1'";
          ExecStart = "${pkgs.tailscale}/bin/tailscale serve --bg --https=443 http://127.0.0.1:8123";
          ExecStop = "${pkgs.tailscale}/bin/tailscale serve off";
        };
      };
      ```

      **Notes on the ExecStartPre wait loop:**
      - `tailscaled.service` may report as "started" before Tailscale
        authenticates to the control plane. The `tailscale serve` command
        requires authentication to configure the serve endpoint.
      - The loop polls `tailscale status --json` up to 30 times (60 seconds
        max, 2s intervals) for `BackendState == "Running"`.
      - If Tailscale is not ready after 60s, the service fails (which is
        correct -- serve cannot work without authentication).

      **Note on pkgs scope:**
      The module already receives `pkgs` in its argument set
      (`{ config, pkgs, ... }`). The `pkgs.bash`, `pkgs.tailscale`, and
      `pkgs.jq` references are valid.

      Verify: No syntax errors (checked via `nix flake check` in task 34-01-C).

## Risks and Mitigations

### Tailscale Serve startup race (MEDIUM)

`tailscaled.service` may report ready before authentication completes. The
`ExecStartPre` wait loop (30 retries x 2s = 60s timeout) mitigates this by
polling `tailscale status --json` for `BackendState == "Running"`. If Tailscale
auth is slow on boot, the service fails and must be manually restarted.

Fallback: the `--bg` flag on `tailscale serve` means once it runs successfully
even once, the config persists across reboots independently of the systemd
service. The service is a belt-and-suspenders declarative wrapper.

### MCP integration is UI-only (LOW)

The MCP Server integration cannot be enabled via NixOS config. If HA state is
lost, it must be re-enabled manually. Mitigated by: `/var/lib/hass` is
persisted via impermanence and backed up via restic.

### Port 443 conflict with nginx (LOW)

nginx serves HTTPS on port 443 on the public interface (`0.0.0.0`). Tailscale
Serve operates on the `tailscale0` virtual NIC. They bind to different
interfaces, so no conflict is expected. Verified in research. If conflict
occurs, change Tailscale Serve to a different port (e.g., `--https=8443`).

### CO2 sensor not yet paired (EXPECTED)

No CO2 entity exists in HA. This is a known prerequisite documented in the
plan. Lights work immediately. CO2 requires: (1) physical sensor connected,
(2) ESPHome/Hue pairing, (3) entity exposed in HA UI.

### Claude connector auth on Tailscale-only endpoint (LOW)

Bearer token auth does not require claude.ai servers to reach the HA instance
(unlike OAuth). The token is simply included in MCP requests from the Claude
app, which runs on the user's phone (on the tailnet). This is confirmed in
research as the correct approach.
