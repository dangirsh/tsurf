    **Cross-host access:** The homepage runs on Contabo. For local agents (neurosys-dev, conway-automaton, claw-swap-dev), it queries `http://localhost:<port>`. For OVH agents (ovh-dev), it queries `http://neurosys-prod:<port>` via Tailscale MagicDNS. This requires the socat proxy on OVH to listen on `0.0.0.0` (or the Tailscale interface), not just localhost.

    **Socat bind address:** The default socat `TCP-LISTEN` binds to all interfaces. Since the ports are in `internalOnlyPorts` (not on public firewall) and only reachable via Tailscale (trustedInterfaces), binding to all interfaces is safe. Verify this in the agentd.nix socat service definition.

    **Position in services list:** Insert the "Agents" section at the top of the services list (before "Infrastructure") to give it prominence on the dashboard. The agents are the primary inhabitants of the server -- they deserve top billing.

    **Socket readiness race:** The socat proxy may start before agentd has created its Unix socket. The `after = ["agentd-${name}.service"]` dependency waits for service start, not socket readiness. This is handled by the socat proxy's `Restart = "on-failure"` with `RestartSec = 3` (set in plan 01, Task 3). The proxy will fail, wait 3 seconds, then retry until the socket becomes available. This is the intended behavior — no additional socket-activation or readiness-check logic is needed.

    **Graceful degradation:** If an agent is not running, the socat proxy will fail to connect to the Unix socket and the customapi widget will show an error state. This is acceptable -- it indicates the agent is down. The homepage widget handles HTTP errors gracefully with a "Service Unavailable" display.

    **Note on customapi mappings:** The agentd API returns an array `[{"name":"...","running":true,"session":"...","restarts":0}]`. The `0.running` and `0.restarts` field paths access the first (and only) agent in the response. Verify the exact response format from agentd docs/source.
