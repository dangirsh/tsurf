## Objective

Reconfigure the automaton's NixOS module based on the diagnostic findings from Plan 51-01. This plan addresses the root causes that prevent the agent from being productive:

1. **Replace the abstract genesis prompt** with a concrete, actionable first task
2. **Increase maxTokensPerTurn** from 4096 to 16384 (the system prompt alone is enormous)
3. **Set maxChildren to at least 1** so the orchestrator can delegate, OR remove orchestrator language from the system prompt context
4. **Configure the social relay URL** so the agent can participate in the Conway marketplace
5. **Adjust the SOUL.md** to reflect achievable goals given the infrastructure constraints
6. **Tune heartbeat config** for productive monitoring instead of no-op checks

**Important:** The exact changes depend on Plan 51-01 diagnostic findings. If the Conway API key is a placeholder, some configurations (social relay, sandbox creation) are moot until a valid key is provisioned. The executor must adapt based on 51-01 results.

<task id="C" title="Decide on maxChildren and orchestrator mode">
<description>
The research identified a conflict: the system prompt instructs the agent to be a "PARENT ORCHESTRATOR" and delegate via `create_goal`, but `maxChildren = 0` makes delegation impossible. This wastes tokens on reasoning the agent cannot execute.

**Decision point (requires human input from 51-01 review):**

**Option 1: Enable child agents (maxChildren = 2)**
```nix
    maxChildren = 2;
```
This lets the agent spawn up to 2 worker agents for task delegation. Workers share the same inference proxy and BYOK key. Each worker is a separate process under the `automaton` user.

**Pros:** Matches the designed architecture. Agent can delegate specialized tasks.
**Cons:** Each child consumes inference tokens. Without revenue, this accelerates spend.

**Option 2: Keep maxChildren = 0, simplify expectations**
Leave `maxChildren = 0` and adjust the genesis prompt to not reference delegation or orchestrator patterns.

**Pros:** Lower cost. Simpler mental model.
**Cons:** The system prompt still has orchestrator instructions (baked into the automaton binary). The agent will see orchestrator tools in its tool list and be confused about why they don't work.

**Recommended: Option 1 (maxChildren = 2)** only if the wallet is funded per 51-03, or the user commits to funding it. Each child agent consumes inference tokens via BYOK, so unfunded operation with children accelerates spend with no revenue pathway. If the wallet remains unfunded, default to Option 2 until funding is confirmed.

The orchestrator architecture is deeply embedded in the system prompt and cannot be easily removed without code changes.

**checkpoint:human-decision** — Ask user which option they prefer. Default to Option 1 if no preference stated.

**Files to modify (if Option 1):**
- `/data/projects/neurosys/modules/automaton.nix` — change `maxChildren = 0` to `maxChildren = 2`
- `/data/projects/private-neurosys/modules/automaton.nix` — same change
</description>
<verification>
1. `maxChildren` value is set according to the chosen option
2. If Option 2, genesis prompt (from Task A) does not reference delegation or child agents
</verification>

<task id="H" title="Apply configuration to live server">
<description>
After validation, apply the config changes to the live server. There are two aspects:

**1. Deploy the new NixOS configuration:**
The automaton.json is only written on first activation (if file doesn't exist). Since the file already exists on a running server, the deploy alone won't update the genesis prompt, SOUL.md, or heartbeat config.

**2. Update the live config files:**
After deploy, SSH to the server and update the runtime config using a single atomic jq pipeline (all field updates in one pass, one temp file):

```bash
# All automaton.json updates in a single atomic pipeline
# Substitute NEW_PROMPT_HERE with the chosen genesis prompt from Task A
CFG=/var/lib/automaton/.automaton/automaton.json
ssh root@neurosys "jq --arg prompt 'NEW_PROMPT_HERE' '
  .genesisPrompt = \$prompt |
  .maxTokensPerTurn = 16384 |
  .maxChildren = 2 |
  .socialRelayUrl = \"wss://social.conway.tech\"
' $CFG > ${CFG}.tmp && mv ${CFG}.tmp $CFG && chown automaton:automaton $CFG && chmod 0600 $CFG"

# Replace SOUL.md
ssh root@neurosys 'rm /var/lib/automaton/.automaton/SOUL.md'
# (let activation script recreate from new template on next deploy, or write directly)

# Replace heartbeat.yml
ssh root@neurosys 'rm /var/lib/automaton/.automaton/heartbeat.yml'
# (let activation script recreate from new template on next deploy)

# Restart the service
ssh root@neurosys 'systemctl restart conway-automaton.service'
```

**Note:** The `${CFG}.tmp` temp file is in the same directory as the original (`/var/lib/automaton/.automaton/`), avoiding `/tmp/` entirely. It is immediately renamed and does not persist.

**checkpoint:human-action** — Deploying config changes to a live server requires user confirmation.
