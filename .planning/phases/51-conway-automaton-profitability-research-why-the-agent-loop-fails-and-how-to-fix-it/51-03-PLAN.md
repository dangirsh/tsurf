# Plan 51-03: Infrastructure — Fund Wallet and Verify Conway API Key

```yaml
wave: 2
depends_on: [51-01]
files_modified:
  - secrets/neurosys.yaml  # sops-encrypted; conway-api-key may be in private overlay's secrets file
autonomous: false
```

## Objective

Address the financial bootstrap problem and Conway Cloud access. Without funding and a valid API key, the agent cannot participate in any revenue-generating activity regardless of how well the config is tuned.

This plan runs in parallel with 51-02 (both depend only on 51-01 diagnostics). The changes here are secret management and external actions, not NixOS module changes.

**Key constraint:** This plan requires human action — wallet funding requires a crypto transaction from the user's wallet, and Conway API key provisioning may require account creation on conway.tech.

---

## Tasks

<task id="A" title="Verify or provision Conway API key">
<description>
Based on 51-01 diagnostic findings, determine the Conway API key status:

**If API key is placeholder (`cnwy_k_PLACEHOLDER` or empty):**

1. The user needs to obtain a real Conway API key from https://conway.tech
2. Once obtained, update the sops-encrypted secret:
   ```bash
   sops --set '["conway-api-key"] "cnwy_k_REAL_KEY_HERE"' secrets/neurosys.yaml
   ```
3. The activation script in `automaton.nix` will inject the key into `automaton.json` on next deploy.

**If API key exists but may be invalid (per 51-01 Task D):**

1. Test the key directly:
   ```bash
   CONWAY_KEY=$(sops -d --extract '["conway-api-key"]' secrets/neurosys.yaml)
   curl -s https://api.conway.tech/v1/credits -H "Authorization: Bearer $CONWAY_KEY" | jq .
   ```
2. If 401/403: the key has expired or been revoked. User must regenerate from Conway dashboard.
3. If 200: key is valid. Record the credit balance.

**If API key is valid:**
No action needed. Record the credit balance for Plan 51-04 validation.

**checkpoint:human-action** — Conway API key provisioning requires user account access.
</description>
<verification>
1. Conway API key in sops is not a placeholder
2. API call to `https://api.conway.tech/v1/credits` returns 200 with the new key
</verification>
</task>

<task id="B" title="Determine wallet address and funding needs">
<description>
From 51-01 Task C, the wallet address and USDC balance are known.

**If wallet is unfunded ($0 USDC):**

The user needs to send USDC (on Base mainnet) to the wallet address. Recommended initial funding:

- **Minimum viable:** $10 USDC — enough for ~2000 Conway credits ($5 minimum topup) and some x402 micro-payments
- **Comfortable:** $50 USDC — enough for sustained experimentation (sandbox creation, domain registration, multiple topups)
- **Full experiment:** $200 USDC — sufficient for 30 days of active operation with child agents and sandbox hosting

The agent's treasury policy caps:
- Max single transfer: $50
- Max hourly transfer: $100
- Max daily transfer: $500
- Minimum reserve: $50
- Max x402 payment: $5
- Max inference daily: $50

These limits prevent runaway spending. The agent cannot drain the wallet faster than the daily cap.

**Action for user:**
1. Record wallet address from 51-01 Task C
2. Send USDC on Base mainnet to that address
3. Verify receipt:
   ```bash
   # Check balance after funding (same method as 51-01 Task C)
   ADDR="0x..."  # wallet address from 51-01

   # Guard: validate address format before making RPC call
   if ! echo "$ADDR" | grep -qE '^0x[0-9a-fA-F]{40}$'; then
     echo "ERROR: Invalid wallet address: '$ADDR' — cannot verify funding"
     exit 1
   fi

   PADDED=$(echo $ADDR | sed 's/0x//' | awk '{printf "%064s", $0}' | tr ' ' '0')
   curl -s -X POST https://mainnet.base.org \
     -H "Content-Type: application/json" \
     -d "{\"jsonrpc\":\"2.0\",\"method\":\"eth_call\",\"params\":[{\"to\":\"0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913\",\"data\":\"0x70a08231000000000000000000000000${PADDED}\"},\"latest\"],\"id\":1}" \
     | jq -r '.result'
   ```

**checkpoint:human-action** — Wallet funding requires a crypto transaction from the user's wallet.
</description>
<verification>
1. Wallet USDC balance is greater than $0 (funding confirmed)
2. Exact balance is recorded for Plan 51-04 baseline
</verification>
</task>

<task id="C" title="Verify sops secret injection works on deploy">
<description>
After updating the Conway API key in sops (if Task A required changes), verify the deployment pipeline correctly injects it.

**Test the activation script logic:**
The activation script in `automaton.nix` reads from `config.sops.secrets."conway-api-key".path` and injects into `automaton.json` via jq. It skips injection if the key is empty or equals `cnwy_k_PLACEHOLDER`.

After deploy:
```bash
ssh root@neurosys 'jq -r .conwayApiKey /var/lib/automaton/.automaton/automaton.json | head -c 10'
```

Should show the first 10 characters of the real key (starts with `cnwy_k_`).

**Do NOT display the full key.** Just confirm it starts with `cnwy_k_` and is not a placeholder.

If injection fails, check:
1. `ssh root@neurosys 'cat /run/secrets/conway-api-key | head -c 10'` — is the sops secret decrypted?
2. Is the sopsFile path correct in the module? (Should be `../secrets/neurosys.yaml` from modules/ directory)
</description>
<verification>
1. `conwayApiKey` in live automaton.json starts with `cnwy_k_` and is not `cnwy_k_PLACEHOLDER`
2. Sops secret is decrypted at `/run/secrets/conway-api-key`
</verification>
</task>

**Do NOT trigger a topup manually.** Just verify the pathway works (API key valid + USDC available = topup should succeed when agent tries).

Record: Conway credit balance, USDC balance, expected topup tier.
</description>
