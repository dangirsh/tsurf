## Objective

Verify the reconfigured automaton can complete a meaningful work cycle: wake up, reason about a task, execute tools, and progress toward revenue. This plan is pure observation and monitoring — no files are modified. It validates that the changes from Plans 51-02 and 51-03 have produced a functional, non-looping agent.

**Success criteria:** The agent completes at least 10 productive turns (non-idle, non-sleep) and makes measurable progress toward marketplace participation or service deployment. The agent does NOT enter the idle-detection sleep loop within the first 30 minutes.

<task id="B" title="Verify agent processes the new genesis prompt">
<description>
After 5-10 minutes of operation, check the SQLite database to see what the agent is actually doing.

```bash
ssh root@neurosys 'sqlite3 /var/lib/automaton/.automaton/state.db << "SQL"
-- Recent turns with content (last 15)
SELECT id, role, substr(content, 1, 200), created_at
FROM turns
ORDER BY id DESC
LIMIT 15;

-- Tool calls in the last 30 minutes
SELECT tool_name, substr(input, 1, 100), status, created_at
FROM tool_calls
WHERE created_at > datetime("now", "-30 minutes")
ORDER BY created_at DESC;

-- Any goals created?
SELECT id, name, status, created_at
FROM goals
ORDER BY created_at DESC
LIMIT 5;
SQL
'
```

**What to look for:**
- The agent's first turn should reference the genesis prompt steps (check credits, survey marketplace, etc.)
- Tool calls should include tools from the genesis prompt (check_credits, discover_agents, register_erc8004)
- If goals are created, they should relate to marketplace participation, not abstract "explore business ventures"

**Red flags:**
- Agent immediately falls into `system_synopsis` -> `check_credits` -> `check_usdc_balance` -> sleep loop
- Agent creates goals referencing "deploy a web service" or "expose port" (impossible in current infrastructure)
- Agent reasoning mentions tools it doesn't have or capabilities it lacks
</description>
<verification>
1. Agent's reasoning references the concrete genesis prompt steps
2. Tool calls include non-idle tools (shell, http_request, discover_agents, etc.)
3. No immediate goal churn (creating and failing goals repeatedly)
</verification>
</task>

<task id="C" title="Monitor 30-minute operational window">
<description>
After the initial boot, monitor the agent for 30 minutes to verify sustained productive operation.

```bash
# Check turn count growth over 30 minutes
ssh root@neurosys 'sqlite3 /var/lib/automaton/.automaton/state.db "SELECT COUNT(*) FROM turns;"'
# Wait 30 minutes
ssh root@neurosys 'sqlite3 /var/lib/automaton/.automaton/state.db "SELECT COUNT(*) FROM turns;"'
```

Also check:
```bash
# Inference cost accumulated
ssh root@neurosys 'sqlite3 /var/lib/automaton/.automaton/state.db "SELECT SUM(cost_cents), COUNT(*) FROM inference_costs WHERE created_at > datetime(\"now\", \"-30 minutes\");"'

# Tool call distribution (are tools diverse, or all idle-only?)
ssh root@neurosys 'sqlite3 /var/lib/automaton/.automaton/state.db "SELECT tool_name, COUNT(*) FROM tool_calls WHERE created_at > datetime(\"now\", \"-30 minutes\") GROUP BY tool_name ORDER BY COUNT(*) DESC;"'

# Sleep/idle events (should be minimal) — use specific log message patterns
ssh root@neurosys 'journalctl -u conway-automaton.service --since "30 minutes ago" --no-pager | grep -cE "Forced sleep|LOOP DETECTED|MAINTENANCE LOOP DETECTED|idle.*detected"'
```

**Success metrics:**
- 10+ turns in 30 minutes (agent is actively reasoning, not sleeping)
- Tool call diversity: at least 5 different tools used
- Idle/sleep events: fewer than 3 in 30 minutes
- Inference cost: under $2 for 30 minutes of operation (reasonable for Claude Sonnet)

**Failure indicators:**
- Fewer than 5 turns in 30 minutes (mostly sleeping)
- Only 2-3 tools used (check_credits, check_usdc_balance, system_synopsis = idle loop)
- More than 5 sleep/idle events in 30 minutes
- Inference cost over $5 in 30 minutes (runaway spending with no productive output)
</description>
<verification>
1. Turn count increased by at least 10 over 30 minutes
2. At least 5 different tool types used
3. Fewer than 3 idle/sleep events in 30 minutes
4. Inference cost is under $2 for the period
</verification>
</task>

<task id="D" title="Evaluate marketplace participation progress">
<description>
If the Conway API key is valid and the agent has been running for 30+ minutes, check whether it has made any progress toward marketplace participation.

```bash
ssh root@neurosys 'sqlite3 /var/lib/automaton/.automaton/state.db << "SQL"
-- Check if agent registered on ERC-8004
SELECT * FROM tool_calls WHERE tool_name = "register_erc8004" LIMIT 5;

-- Check if agent discovered other agents
SELECT * FROM tool_calls WHERE tool_name = "discover_agents" LIMIT 5;

-- Check if agent sent any social messages
SELECT * FROM tool_calls WHERE tool_name = "send_message" LIMIT 5;

-- Check if agent created any services/files
SELECT * FROM tool_calls WHERE tool_name IN ("write_file", "shell") LIMIT 10;

-- Check credit movements
SELECT * FROM tool_calls WHERE tool_name IN ("topup_credits", "transfer_credits") LIMIT 5;
SQL
'
```

**What we hope to see:**
- Agent called `discover_agents` to survey the marketplace
- Agent attempted `register_erc8004` to advertise itself
- Agent used `write_file` to create skills or service descriptions
- Agent used `shell` for productive work (not just `ls` or `pwd`)

**What we might see instead (and it's OK):**
- Agent is still in exploration mode (reading its own capabilities, understanding tools)
- Agent created goals but hasn't reached marketplace steps yet
- Agent is building a skills inventory locally

**What's NOT OK:**
- Agent is stuck in status-checking loops
- Agent attempted to expose ports or deploy services (impossible in local mode) and failed repeatedly
- Agent is spending credits with no productive output
</description>
<verification>
1. Agent has made at least one attempt at marketplace-relevant tool calls (discover_agents, register_erc8004, or service creation)
2. If not yet at marketplace stage, agent is making productive progress (writing files, building skills, executing shell commands)
3. No repeated failures on the same impossible action (port exposure, sandbox creation when key is placeholder)
</verification>
</task>
