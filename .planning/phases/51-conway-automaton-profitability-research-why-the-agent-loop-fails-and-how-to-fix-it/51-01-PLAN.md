## Objective

SSH to the neurosys server and determine the actual current state of the Conway Automaton before making any configuration changes. This plan is pure observation — no files are modified. The output informs all subsequent plans.

Key questions to answer:
1. Is `conway-automaton.service` running, failed, or never deployed?
2. Is the wallet funded (USDC balance on Base mainnet)?
3. Is the Conway API key valid (not a placeholder)?
4. What is the agent's credit balance?
5. What does the SQLite state.db show (turns, goals, tool calls, last activity)?
6. What is the wallet's EVM address (needed for external funding)?
7. Is the social relay configured?
8. What has the agent actually done since first boot?

**Note:** Per MEMORY.md, phases 40/45/47/48/49 have NOT been deployed. The automaton module (Phase 32) may or may not be deployed depending on when the last deploy happened.

<task id="A" title="Check conway-automaton.service status">
<description>
SSH to neurosys and check if the service exists and its current state.

```bash
ssh root@neurosys 'systemctl status conway-automaton.service 2>&1 | head -30'
```

If the service does not exist (`Unit conway-automaton.service could not be found`), the automaton module has never been deployed. This blocks all other diagnostics — the remaining tasks in this plan become "not applicable" and Plan 51-02 must include a deploy step.

If the service exists but is `failed` or `inactive`, check logs:
```bash
ssh root@neurosys 'journalctl -u conway-automaton.service --no-pager -n 100'
```

Record:
- Service state: `active (running)` / `failed` / `inactive` / `not found`
- If running: uptime, PID, memory usage
- If failed: last error from journal
</description>
<verification>
Service state is recorded. If `not found`, all subsequent tasks are marked N/A.
</verification>
</task>

<task id="B" title="Check automaton state directory and config" depends_on="A">
<description>
Examine the automaton's state directory to understand what exists on disk.

```bash
ssh root@neurosys 'ls -la /var/lib/automaton/.automaton/ 2>&1'
```

If the directory does not exist, the activation script has never run.

If it exists, inspect the config:
```bash
ssh root@neurosys 'cat /var/lib/automaton/.automaton/automaton.json 2>/dev/null | jq .'
```

Key fields to record:
- `conwayApiKey`: Is it empty or `cnwy_k_PLACEHOLDER`? If so, Conway Cloud tools are broken.
- `walletAddress`: Is it populated? If empty, the wallet was generated but address may not have been derived yet.
- `maxChildren`: Should be `0` currently.
- `maxTokensPerTurn`: Should be `4096` currently.
- `sandboxId`: Should be empty (local mode).
- `inferenceModel`: Should be `claude-sonnet-4-6`.
- `genesisPrompt`: Current prompt text.

Also check for wallet:
```bash
ssh root@neurosys 'cat /var/lib/automaton/.automaton/wallet.json 2>/dev/null | jq "del(.privateKey) | . + {hasPrivateKey: true}"'
```

Do NOT display the private key. Only confirm it exists and record the wallet address.
</description>
<verification>
Config and wallet state are recorded. Conway API key validity is determined (placeholder vs real key).
</verification>
</task>

<task id="C" title="Check wallet EVM address and USDC balance" depends_on="B">
<description>
If the wallet.json exists, derive the EVM address and check its USDC balance on Base mainnet.

The wallet address can be obtained from automaton.json's `walletAddress` field, or by deriving it from the private key if not populated.

Check USDC balance on Base mainnet via public RPC:
```bash
# Get wallet address from config
ADDR=$(ssh root@neurosys 'jq -r .walletAddress /var/lib/automaton/.automaton/automaton.json')
echo "Wallet address: $ADDR"

# Guard: validate address is non-empty and well-formed
if ! echo "$ADDR" | grep -qE '^0x[0-9a-fA-F]{40}$'; then
  echo "ERROR: Invalid or empty wallet address: '$ADDR'"
  echo "Wallet may not be initialized. Record 'wallet not initialized' and skip balance check."
  exit 0
fi

# USDC contract on Base: 0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913
# balanceOf(address) selector: 0x70a08231
# Pad address to 32 bytes for ABI encoding
PADDED=$(echo $ADDR | sed 's/0x//' | awk '{printf "%064s", $0}' | tr ' ' '0')
curl -s -X POST https://mainnet.base.org \
  -H "Content-Type: application/json" \
  -d "{\"jsonrpc\":\"2.0\",\"method\":\"eth_call\",\"params\":[{\"to\":\"0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913\",\"data\":\"0x70a08231000000000000000000000000${PADDED}\"},\"latest\"],\"id\":1}" \
  | jq -r '.result'
```

The result is a hex value. USDC has 6 decimals. Convert to human-readable:
- `0x0` = $0.00 (unfunded)
- `0x4c4b40` = $5.00

If the address is empty or the wallet doesn't exist, record "wallet not initialized."

Record: wallet address, USDC balance in dollars.
</description>
<verification>
Wallet address and USDC balance recorded. If $0.00, this confirms the financial bootstrap problem.
</verification>
</task>

<task id="D" title="Check Conway API key validity and credit balance" depends_on="B">
<description>
Test whether the Conway API key is valid by making a test API call.

```bash
# Get the API key (if service is running, read from the live config)
CONWAY_KEY=$(ssh root@neurosys 'jq -r .conwayApiKey /var/lib/automaton/.automaton/automaton.json')

# Skip if placeholder
if [ "$CONWAY_KEY" = "" ] || [ "$CONWAY_KEY" = "cnwy_k_PLACEHOLDER" ]; then
  echo "Conway API key is PLACEHOLDER — Cloud tools are non-functional"
else
  # Test API key by checking credits
  curl -s https://api.conway.tech/v1/credits \
    -H "Authorization: Bearer $CONWAY_KEY" \
    | jq .
fi
```

Record:
- API key status: `placeholder` / `valid` / `invalid (HTTP 401/403)`
- If valid: credit balance in cents
- If invalid: error message from Conway API
</description>
<verification>
Conway API key validity and credit balance (if applicable) are recorded.
</verification>
</task>

<task id="E" title="Examine SQLite state database" depends_on="A">
<description>
If the service has run at least once, examine the SQLite database for agent activity history.

```bash
ssh root@neurosys 'sqlite3 /var/lib/automaton/.automaton/state.db 2>/dev/null << "SQL"
-- Turn count and last activity
SELECT COUNT(*) as total_turns, MAX(created_at) as last_turn FROM turns;

-- Recent turns (last 10)
SELECT id, role, substr(content, 1, 120), created_at FROM turns ORDER BY id DESC LIMIT 10;

-- Goal status
SELECT id, status, name, substr(description, 1, 80) FROM goals LIMIT 10;

-- Tool call summary
SELECT tool_name, COUNT(*) as calls FROM tool_calls GROUP BY tool_name ORDER BY calls DESC LIMIT 15;

-- Inference costs
SELECT SUM(cost_cents) as total_cost_cents, COUNT(*) as api_calls FROM inference_costs;

-- Identity/KV store
SELECT key, substr(value, 1, 100) FROM kv LIMIT 20;
SQL
'
```

If the database doesn't exist or is empty, the agent has never completed a turn.

Record:
- Total turns executed
- Last activity timestamp
- Goals created and their status (pending/executing/completed/failed)
- Most-used tools
- Total inference cost in cents
- Any stored identity or state values
</description>
<verification>
Agent activity history is recorded. If zero turns, confirms the agent has never successfully run.
</verification>
</task>

<task id="F" title="Check anthropic-secret-proxy status" depends_on="A">
<description>
The automaton depends on the secret proxy for inference. Verify it's running.

```bash
ssh root@neurosys 'systemctl status anthropic-secret-proxy.service 2>&1 | head -15'
```

Also test that it responds:
```bash
ssh root@neurosys 'curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:9091/v1/messages -X POST -H "Content-Type: application/json" -H "x-api-key: test" -H "anthropic-version: 2023-06-01" -d "{\"model\":\"claude-sonnet-4-6\",\"max_tokens\":10,\"messages\":[{\"role\":\"user\",\"content\":\"hi\"}]}"'
```

Expected: HTTP 200 (if API key is valid upstream) or HTTP 400/401 (if test key is rejected by proxy or upstream). Either confirms the proxy is accepting connections.

If the proxy is not running, the automaton cannot do inference at all.
</description>
<verification>
Secret proxy status recorded: running/not running, responding on port 9091.
</verification>
</task>
