<objective>
Add deploy-rs magic rollback safety to the neurosys deployment pipeline.

Purpose: A Tailscale-only server has zero fallback if a deploy breaks networking -- SSH is the only access path, and it routes through Tailscale. deploy-rs creates an inotify canary on the target after activation; if the deployer cannot SSH back to confirm within 120 seconds, the target automatically rolls back to the previous NixOS generation. This turns "permanent lockout" into "brief blip".

Output: deploy-rs integrated in flake.nix, deploy.sh evolved to use it, recovery runbook updated with rollback documentation.
</objective>

**deploy.sh changes:**

Replace the `nixos-rebuild switch` invocation with deploy-rs. The deploy-rs binary comes from the flake input itself for version consistency (the lib used for activation scripts must match the CLI binary).

1. In the "Build + deploy" section, replace the `nix shell nixpkgs#nixos-rebuild -c nixos-rebuild switch` call:

   For local mode (default):
   ```bash
   nix run ".#deploy-rs" -- \
     "$FLAKE_DIR#neurosys" \
     --confirm-timeout 120
   ```

   Wait -- deploy-rs flake does not export `apps` or `packages` by default in a way that `nix run .#deploy-rs` works from the neurosys flake. Instead, use the deploy-rs input's package directly:
   ```bash
   nix run "github:serokell/deploy-rs" -- \
     "$FLAKE_DIR#neurosys" \
     --confirm-timeout 120
   ```

   However, for reproducibility (pin to the flake.lock version), the better approach is:
   ```bash
   nix shell "${FLAKE_DIR}#deploy-rs-package" -c deploy \
     "$FLAKE_DIR#neurosys" \
     --confirm-timeout 120
   ```

   Actually, the simplest correct approach: add a `packages.x86_64-linux.deploy-rs` passthrough in flake.nix that exposes the deploy-rs package from the input, OR just use `nix run` with the locked input. Since the flake.lock pins deploy-rs, `nix run github:serokell/deploy-rs` will use the locked version when run from within the flake directory.

   Simpler approach that works: use the deploy-rs input's defaultApp. Since we have deploy-rs in inputs and it's locked, running from the flake dir picks up the lock:
   ```bash
   nix run "${FLAKE_DIR}#checks.x86_64-linux" is wrong...
   ```

   Final approach (proven pattern from research): just use `nix run github:serokell/deploy-rs --`. When run from a directory with a flake.lock that contains deploy-rs, Nix resolves to the locked revision. This is the standard deploy-rs usage pattern.

   For local mode:
   ```bash
   echo "==> Deploying to $TARGET with magic rollback (confirm timeout: 120s)..."
   nix run github:serokell/deploy-rs -- \
     "$FLAKE_DIR#neurosys" \
     --confirm-timeout 120
   ```

   For remote mode, use `--remote-build`:
   ```bash
   echo "==> Deploying via remote build on $TARGET with magic rollback..."
   nix run github:serokell/deploy-rs -- \
     "$FLAKE_DIR#neurosys" \
     --confirm-timeout 120 \
     --remote-build
   ```

   Note: The `--target` flag from deploy.sh becomes irrelevant for the deploy-rs call since the hostname is configured in `deploy.nodes.neurosys.hostname` in flake.nix. If the user overrides `--target`, print a warning that deploy-rs uses the flake-configured hostname. Keep `$TARGET` for the container health check SSH commands (those still use direct SSH).

2. Add a `--first-deploy` flag to deploy.sh that passes `--magic-rollback false` to deploy-rs. This is needed exactly once for the initial migration from nixos-rebuild to deploy-rs. Add it to the argument parser and usage text.

3. Add a `--no-magic-rollback` flag for intentional networking changes (per 25-RESEARCH.md anti-pattern guidance). Passes `--magic-rollback false` to deploy-rs.

4. Update the @decision comments at the top of deploy.sh to reflect the deploy-rs evolution.

5. Keep everything else: flake input update, locking (both local and remote -- defense in depth per research Open Question 3), container health check, reporting. The container health check still uses SSH to $TARGET.

6. Update the rollback guidance in the failure report section. Currently it prints `ssh $TARGET nixos-rebuild switch --rollback`. Update to note that deploy-rs handles automatic rollback for connectivity failures, and manual rollback (if needed for container issues) uses `ssh $TARGET nixos-rebuild switch --rollback` as before.
