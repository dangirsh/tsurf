# Plan 32-01: Package Conway Automaton as NixOS Derivation

## Goal

Add the Conway Automaton (`@conway/automaton`) as a flake input and package it
using `buildNpmPackage`. The package must compile TypeScript, build the
`better-sqlite3` native addon, and patch the hardcoded Anthropic API URL to
read `ANTHROPIC_BASE_URL` from the environment. Validate with `nix build` and
`nix flake check`.

## Decisions Made (from Research)

- **Packaging strategy**: `buildNpmPackage` with lockfile conversion from
  `pnpm-lock.yaml` (consistent with the claw-swap pattern). The existing
  `package-lock.json` in the repo is a stub (typescript-only) and cannot be
  used. Generate a proper npm lockfile via `pnpm import` in `postPatch`.
- **Anthropic URL patch**: Modify `src/conway/inference.ts` in `postPatch` to
  read `process.env.ANTHROPIC_BASE_URL` instead of the hardcoded
  `https://api.anthropic.com`. This is a ~5 line change that enables the
  existing secret proxy integration.
- **Node.js version**: Use `nodejs_22` (NixOS 25.11 provides it; automaton
  CI tests Node 20 and 22).
- **Native addon**: `better-sqlite3` requires `python3`, `node-gyp`, and
  `makeCacheWritable = true`, followed by explicit `npm rebuild better-sqlite3`
  in `buildPhase`. This is a known pattern from claw-swap (see MEMORY.md).
- **ESM module type**: The automaton uses `"type": "module"` and targets ES2022
  with NodeNext module resolution. The entry point is `dist/index.js`.
- **Flake input**: `flake = false` since the automaton repo is not a Nix flake.
  The source is fetched and passed to `buildNpmPackage` via `src`.

## must_haves

1. `flake.nix` has `inputs.automaton` pointing to `github:Conway-Research/automaton` with `flake = false`
2. `packages/automaton.nix` exists and uses `buildNpmPackage`
3. The Anthropic API URL in `src/conway/inference.ts` is patched to read `ANTHROPIC_BASE_URL` env var
4. `better-sqlite3` native addon compiles successfully (no runtime errors)
5. `nix build .#automaton` succeeds and produces a store path containing `lib/node_modules/@conway/automaton/dist/index.js`
6. `nix flake check` passes for both `nixosConfigurations.neurosys` and `nixosConfigurations.ovh`
7. The built package can be invoked: `node <store-path>/lib/node_modules/@conway/automaton/dist/index.js --help` prints usage

      2. Use `buildNpmPackage` with the automaton source
      3. In `postPatch`, perform three operations:
         a. Generate a proper `package-lock.json` from `pnpm-lock.yaml`:
            ```nix
            postPatch = ''
              # Generate npm lockfile from pnpm lockfile for buildNpmPackage
              ${pnpm}/bin/pnpm import || true
              # If pnpm import generates package-lock.json, great.
              # If not, try: npx @pnpm/make-dedicated-lockfile && npm install --package-lock-only
              # The existing package-lock.json is a stub with only typescript.

              # Patch hardcoded Anthropic API URL to read from environment
              substituteInPlace src/conway/inference.ts \
                --replace-fail \
                  '"https://api.anthropic.com/v1/messages"' \
                  '`${process.env.ANTHROPIC_BASE_URL || "https://api.anthropic.com"}/v1/messages`'
            '';
            ```
         b. Patch the Anthropic API base URL in `src/conway/inference.ts`
            (included in the substituteInPlace above)

      4. Set build configuration:
         ```nix
         nodejs = nodejs_22;
         makeCacheWritable = true;
         npmDepsHash = ""; # Must be computed — see verification step
         nativeBuildInputs = [ python3 nodejs_22 ];
         ```

      5. Set the build phase to compile TypeScript and rebuild native addons:
         ```nix
         buildPhase = ''
           runHook preBuild
           npx tsc
           npm rebuild better-sqlite3
           runHook postBuild
         '';
         ```

      6. Set the install phase:
         ```nix
         installPhase = ''
           runHook preInstall
           mkdir -p $out/lib/node_modules/@conway/automaton
           cp -r dist node_modules package.json $out/lib/node_modules/@conway/automaton/
           # Copy constitution.md for service activation script to reference
           cp constitution.md $out/lib/node_modules/@conway/automaton/

           mkdir -p $out/bin
           cat > $out/bin/automaton <<WRAPPER
           #!/bin/sh
           exec ${nodejs_22}/bin/node $out/lib/node_modules/@conway/automaton/dist/index.js "\$@"
           WRAPPER
           chmod +x $out/bin/automaton
           runHook postInstall
         '';
         ```

      7. Add meta information:
         ```nix
         meta = with lib; {
           description = "Conway Automaton - autonomous AI agent runtime";
           homepage = "https://github.com/Conway-Research/automaton";
           platforms = [ "x86_64-linux" ];
         };
         ```

      **IMPORTANT pnpm-to-npm lockfile conversion:**
      The `pnpm import` command may not work in the Nix sandbox because it
      needs network access. Alternative approaches if `pnpm import` fails:

      - **Option A (preferred):** Run `pnpm import` locally on the automaton
        source, capture the generated `package-lock.json`, and include it as
        a separate source or patch file. This is the most reliable approach.

      - **Option B:** Use `npmConfigHook` and generate the lockfile outside
        the build. Run `cd /tmp/automaton-src && pnpm import` locally, then

        `node_modules/` + `dist/` into the final derivation.

      The implementer should try Option A first. If `pnpm import` cannot be

## Risks and Mitigations

### pnpm lockfile conversion (HIGH)

The `pnpm-lock.yaml` to `package-lock.json` conversion is the highest-risk
step. The `pnpm import` command may fail in the Nix sandbox (no network) or
produce an incompatible lockfile. Mitigation: generate the lockfile locally
and store it as `packages/automaton-package-lock.json`. If all npm-based
approaches fail, fall back to a `mkDerivation` + fixed-output derivation
approach that runs `pnpm install --frozen-lockfile` directly.

### better-sqlite3 native addon (MEDIUM)

Known pattern from claw-swap. Requires `python3` + `makeCacheWritable`.
If it fails, check that node-gyp can find Python and the Node.js headers
match the runtime version (nodejs_22).

### Upstream source changes (LOW)

The automaton repo is pinned by `flake.lock`. If the API surface changes
between when the research was done and when the implementer runs the build,
the TypeScript patch may not apply cleanly. The `--replace-fail` flag in
`substituteInPlace` will catch this.

## Notes for Implementer

- The automaton uses pnpm workspaces (`pnpm-workspace.yaml` defines
  `packages: ["packages/*"]`). The root package is `@conway/automaton` and
  there is a sub-package at `packages/cli/`. The `buildNpmPackage` approach
  builds from the root.
- `dontNpmBuild = true` is NOT set because we provide a custom `buildPhase`.
  If `buildNpmPackage` tries to run `npm run build` automatically AND our
  custom `buildPhase`, set `dontNpmBuild = true` and handle everything in
  the custom `buildPhase`.
- The `constitution.md` at the repo root is the immutable behavioral rules
  file. It must be copied into the package so the service module can reference
  it during state pre-seeding.
- If `pnpm import` works in the sandbox: great, no need for
  `automaton-package-lock.json`. If it does not work: the local generation
  approach is well-established (claw-swap uses the same pattern with workspace
  lockfile copying).
