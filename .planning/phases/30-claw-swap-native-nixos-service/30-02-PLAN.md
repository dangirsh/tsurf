# Plan 30-02: Deploy + Data Migration on OVH

## Goal

Deploy the Phase 30 config to OVH (production) with zero data loss. Stop the
Docker postgres container, dump its data, deploy native postgresql via
NixOS switch, restore the dump, and verify `https://claw-swap.com` returns
200. The Contabo (neurosys) host deploys the new config as a secondary
validation.

## Prerequisites

- Plan 30-01 completed: claw-swap and neurosys repos both committed and pushed
- neurosys flake.lock updated with new claw-swap commit
- SSH access to `root@neurosys-prod` (OVH via Tailscale) and `root@neurosys`

## must_haves

1. Live data from Docker postgres preserved in native postgresql (no data loss)
2. `curl -s -o /dev/null -w "%{http_code}" https://claw-swap.com` returns `200`
3. `systemctl is-active claw-swap-app` returns `active` on OVH
4. `systemctl is-active postgresql` returns `active` on OVH
5. Docker containers `claw-swap-db` and `claw-swap-app` no longer present on OVH
6. `nix flake check` still passes after flake.lock update (from 30-01-I)
7. Neurosys (Contabo) deploy also succeeds without regression

## Tasks

<tasks>
  <task id="30-02-A" wave="1">
    <title>Pre-migration: dump data from Docker postgres on OVH</title>
    <description>
      SSH to OVH: `ssh root@neurosys-prod`

      1. Stop the Docker app container to prevent writes during dump:
         ```bash
         docker stop claw-swap-app
         ```

      2. Dump all data from the Docker postgres container:
         ```bash
         docker exec claw-swap-db pg_dumpall -U claw > /var/lib/claw-swap/backup.sql
         ```

      3. Verify the dump is non-empty:
         ```bash
         wc -l /var/lib/claw-swap/backup.sql
         ```
         Expect at least a few hundred lines (schema + data).

      4. Note: the Docker postgres container stays running during this step
         so the dump is clean. The app container is stopped to freeze writes.

      5. Copy the dump to a safe location in case the path disappears after
         impermanence wipes `/var/lib/claw-swap` (after migration):
         ```bash
         cp /var/lib/claw-swap/backup.sql /persist/backup-claw-swap-$(date +%Y%m%d).sql
         ```
         The `/persist` subvolume survives reboots and NixOS switches.
    </description>
  </task>

  <task id="30-02-B" wave="2">
    <title>Deploy Phase 30 config to OVH</title>
    <description>
      From the local machine (`/data/projects/neurosys`):

      ```bash
      ./scripts/deploy.sh --node ovh
      ```

      What happens during this deploy:
      - `virtualisation.oci-containers` claw-swap-db and claw-swap-app units
        are removed from systemd; Docker stops and removes those containers
      - `services.postgresql` starts a fresh, empty postgresql_16 instance at
        `/var/lib/postgresql`
      - `claw-swap-app.service` starts but FAILS (expected: no DB data yet,
        or schema mismatch); this is acceptable at this step
      - `users.users.claw-swap` and `users.groups.claw-swap` are created

      After the switch completes:

      On OVH, verify postgresql is running:
      ```bash
      ssh root@neurosys-prod systemctl is-active postgresql
      ```

      Verify the `claw_swap` database and `claw` role exist:
      ```bash
      ssh root@neurosys-prod "sudo -u postgres psql -c '\l'"
      ssh root@neurosys-prod "sudo -u postgres psql -c '\du'"
      ```

      Note: `claw-swap-app.service` may be failing at this point — that is
      expected until data is restored. Do not escalate this as a blocker.
    </description>
  </task>

  <task id="30-02-C" wave="3">
    <title>Restore data into native postgresql on OVH</title>
    <description>
      SSH to OVH: `ssh root@neurosys-prod`

      The dump was saved to `/persist/backup-claw-swap-YYYYMMDD.sql`.

      Restore it:
      ```bash
      sudo -u postgres psql < /persist/backup-claw-swap-$(date +%Y%m%d).sql
      ```

      The dump includes `CREATE ROLE`, `CREATE DATABASE`, and `COPY` statements.
      Some may produce "already exists" warnings for `claw` and `claw_swap` —
      these are safe to ignore (the NixOS `ensureDatabases`/`ensureUsers` already
      created them).

      After restore, verify the data:
      ```bash
      sudo -u postgres psql claw_swap -c '\dt'
      sudo -u postgres psql claw_swap -c 'SELECT count(*) FROM users;'
      ```
      (Replace `users` with the actual table name if different. The Drizzle
      schema has tables matching the migration files in `src/db/migrations/`.)
    </description>
  </task>

  <task id="30-02-D" wave="4">
    <title>Start claw-swap-app and verify it connects to postgresql</title>
    <description>
      On OVH:

      ```bash
      ssh root@neurosys-prod "systemctl restart claw-swap-app && systemctl status claw-swap-app"
      ```

      Check the journal for startup errors:
      ```bash
      ssh root@neurosys-prod "journalctl -u claw-swap-app -n 50 --no-pager"
      ```

      Expected: the app starts, logs "Server running on port 3000" (or similar),
      and no DB connection errors appear.

      If the app fails:
      - Check the DATABASE_URL is correctly set to the Unix socket path
      - Verify the `claw-swap` user is in the `postgres` group:
        `id claw-swap` should show `postgres` in groups
      - Verify the socket exists: `ls -la /run/postgresql/`
      - Check pg_hba.conf for the `local claw_swap claw trust` rule:
        `sudo -u postgres psql -c "SHOW hba_file;"` then inspect the file
    </description>
  </task>

## Rollback Plan

If the deploy fails and the site is broken:

1. Restore the Docker app container manually:
   ```bash
   ssh root@neurosys-prod "docker start claw-swap-app"
   ```
   (The Docker postgres container `claw-swap-db` was kept running during the
   migration window. If it was also stopped, restore the dump into it.)

2. If deploy-rs magic rollback already activated, the previous NixOS generation
   is active — but the Docker containers no longer have systemd units to start
   them. Start manually: `docker start claw-swap-db && sleep 5 && docker start claw-swap-app`.

3. Investigate the failure in the new generation's journal:
   `journalctl -u claw-swap-app -b 0 -n 100 --no-pager`

## Notes for Implementer

- This plan is marked `autonomous: false` because it involves live data migration
  and requires manual confirmation at each stage before proceeding. Do not auto-
  advance through the waves without verifying each step.
- The dump-then-restore sequence must happen in order: dump (A) → deploy (B) →
  restore (C) → start app (D) → smoke test (E). Never skip C before D.
- The backup SQL path uses `/persist/` (BTRFS persisted subvolume) so it
  survives any impermanence-triggered root wipe during the NixOS switch.
- If `pg_dumpall` fails in task A (e.g., Docker container not responsive), do
  NOT proceed to deploy. Fix the dump first. The Docker DB is the source of
  truth until restoration is verified.
