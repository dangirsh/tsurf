# Phase 1: Flake Scaffolding + Pre-Deploy - Research

**Researched:** 2026-02-13
**Domain:** NixOS flake scaffolding, disko partitioning, GRUB hybrid boot, sops-nix bootstrap with age/SSH keys
**Confidence:** HIGH

## Summary

Phase 1 delivers all the configuration scaffolding required so that `nixos-anywhere` can deploy a working NixOS system on the first try. This is purely a local development phase -- no deployment happens yet. The deliverables are: a `flake.nix` with pinned inputs, a disko disk layout for Contabo's `/dev/sda`, a GRUB bootloader config for hybrid BIOS/UEFI, and a working sops-nix secrets pipeline (pre-generated SSH host key, derived age key, `.sops.yaml`, encrypted secrets file).

The critical technical challenge in this phase is the **sops-nix bootstrap chicken-and-egg problem**: sops-nix decrypts secrets using an age key derived from the target host's SSH ed25519 key, but that key does not exist until the host is deployed. The solution is to **pre-generate the SSH host key locally**, derive the age public key from it using `ssh-to-age`, configure `.sops.yaml` with that key, encrypt secrets against it, and later deploy the pre-generated key to the host via nixos-anywhere's `--extra-files` flag (Phase 2). This phase handles everything up to and including local decryption verification; actual deployment is Phase 2.

**Primary recommendation:** Build the flake skeleton with minimal modules (just enough for `nix flake check` to pass), the disko config with hybrid BIOS+UEFI partitions, GRUB boot config, and a complete sops-nix secrets pipeline verified locally. Do NOT attempt to deploy in this phase.

### Pattern 1: Flake Input Wiring with specialArgs

**What:** Pass all flake inputs to modules via `specialArgs = { inherit inputs; };` so any module can access sops-nix, disko, etc. without explicit threading.
**When to use:** Always in flake-based NixOS configs.

```nix
# flake.nix
{
  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-25.11";
    home-manager = {
      url = "github:nix-community/home-manager/release-25.11";
      inputs.nixpkgs.follows = "nixpkgs";
    };
    sops-nix = {
      url = "github:Mic92/sops-nix";
      inputs.nixpkgs.follows = "nixpkgs";
    };
    disko = {
      url = "github:nix-community/disko";
      inputs.nixpkgs.follows = "nixpkgs";
    };
  };

  outputs = { self, nixpkgs, home-manager, sops-nix, disko, ... } @ inputs: {
    nixosConfigurations.acfs = nixpkgs.lib.nixosSystem {
      system = "x86_64-linux";
      specialArgs = { inherit inputs; };
      modules = [
        disko.nixosModules.disko
        sops-nix.nixosModules.sops
        home-manager.nixosModules.home-manager
        ./hosts/acfs
        {
          home-manager.useGlobalPkgs = true;
          home-manager.useUserPackages = true;
          home-manager.extraSpecialArgs = { inherit inputs; };
          home-manager.users.dangirsh = import ./home;
        }
      ];
    };
  };
}
```

**Source:** nixos-anywhere-examples flake.nix, sops-nix README, STACK.md research

### Pattern 2: Hybrid BIOS+UEFI Disko Config

**What:** A GPT partition table with both a 1MB BIOS boot partition (type EF02) and a 512MB EFI System Partition (type EF00), plus a root ext4 partition. This works regardless of whether Contabo boots in BIOS or UEFI mode.
**When to use:** Always for Contabo VPS (or any server where boot mode is uncertain).

```nix
# hosts/acfs/disko-config.nix
{ ... }: {
  disko.devices = {
    disk.main = {
      device = "/dev/sda";
      type = "disk";
      content = {
        type = "gpt";
        partitions = {
          boot = {
            size = "1M";
            type = "EF02";  # BIOS boot partition (for GRUB BIOS)
          };
          ESP = {
            name = "ESP";
            size = "512M";
            type = "EF00";  # EFI System Partition
            content = {
              type = "filesystem";
              format = "vfat";
              mountpoint = "/boot";
              mountOptions = [ "umask=0077" ];
            };
          };
          root = {
            size = "100%";
            content = {
              type = "filesystem";
              format = "ext4";
              mountpoint = "/";
            };
          };
        };
      };
    };
  };
}
```

**Source:** disko example/hybrid.nix (BIOS boot + ESP pattern), ARCHITECTURE.md research, PITFALLS.md Pitfall 3

**Key decisions in this config:**
- **`/dev/sda`**: Confirmed by lsblk output in the server context. Contabo uses `sda` not `vda` on this VPS tier.
- **No swap partition**: 47GB RAM is sufficient. Swap can be added later if needed. Keeps initial layout simple.
- **No LUKS**: VPS has no meaningful threat model for disk encryption (hypervisor has access anyway). Contabo cannot enter passphrase at boot.
- **ext4 (not btrfs/ZFS)**: Simplest, most battle-tested. No need for snapshots or compression on a VPS with SSD.

### Pattern 3: GRUB Hybrid Boot Configuration

**What:** GRUB configured to support both BIOS and UEFI boot. Installs to the disk's MBR (for BIOS) and as a removable EFI binary (for UEFI without NVRAM support).
**When to use:** Always for Contabo VPS.

```nix
# modules/boot.nix
{ config, lib, pkgs, ... }: {
  # GRUB bootloader -- hybrid BIOS/UEFI for Contabo compatibility
  boot.loader.grub = {
    enable = true;
    efiSupport = true;
    efiInstallAsRemovable = true;  # No NVRAM on Contabo VPS
    device = "/dev/sda";           # Install GRUB MBR to this disk (for BIOS boot)
  };

  # Do NOT set boot.loader.efi.canTouchEfiVariables (Contabo has no persistent NVRAM)
  # Do NOT enable systemd-boot (UEFI-only, does not support BIOS fallback)

  boot.loader.grub.configurationLimit = 10;  # Prevent boot menu bloat
}
```

**Source:** NixOS manual sec-bootloader, NixOS Wiki GRUB page, PITFALLS.md Pitfall 3, Pitfall 15

**Why `efiInstallAsRemovable = true`:**
Contabo VPS does not provide persistent NVRAM for EFI variables. Without this flag, GRUB would try to write EFI boot entries that get lost on reboot. `efiInstallAsRemovable` places the EFI binary at the fallback path (`/boot/EFI/BOOT/BOOTX64.EFI`) where all UEFI firmware looks by default.

### Pattern 4: sops-nix Age Key from SSH Host Key

**What:** sops-nix derives its decryption key from the host's SSH ed25519 key at activation time. No separate age key file to manage.
**When to use:** Always for NixOS servers with sops-nix.

```nix
# modules/secrets.nix
{ config, ... }: {
  sops = {
    defaultSopsFile = ../../secrets/acfs.yaml;
    age.sshKeyPaths = [ "/etc/ssh/ssh_host_ed25519_key" ];
    # Do NOT set age.keyFile -- we derive from SSH host key
    # Do NOT set age.generateKey -- we pre-generate and deploy the key
  };

  # Declare individual secrets as needed (Phase 1: minimal placeholder)
  # sops.secrets."example/test" = {};
}
```

**Source:** sops-nix README (`age.sshKeyPaths`), Stapelberg 2025 guide, PITFALLS.md Pitfall 2

### Pattern 5: VirtIO Kernel Modules for Contabo

**What:** Explicitly declare VirtIO kernel modules in initrd to prevent boot hang on KVM-based VPS.
**When to use:** Always for KVM/QEMU virtual machines (Contabo uses KVM).

```nix
# hosts/acfs/hardware.nix
{ config, lib, pkgs, modulesPath, ... }: {
  imports = [
    (modulesPath + "/profiles/qemu-guest.nix")
  ];

  boot.initrd.availableKernelModules = [
    "virtio_pci"
    "virtio_scsi"
    "virtio_blk"
    "virtio_net"
    "ahci"
    "sd_mod"
  ];

  boot.kernelModules = [ "kvm-amd" ];  # AMD EPYC host
}
```

**Source:** NixOS/nixpkgs#76980, PITFALLS.md Pitfall 4. The `qemu-guest.nix` profile provides sensible defaults for virtualized environments. We add explicit modules as a safety net.

### Anti-Patterns to Avoid

- **Monolithic configuration.nix**: Do NOT put everything in one file. Use module-per-concern from the start.
- **Forgetting `git add`**: Nix flakes only see git-tracked files. Every new `.nix` file MUST be `git add`ed before `nix flake check` will see it.
- **Using `nixpkgs-unstable` for server**: Pin to `nixos-25.11` stable. Unstable breaks.
- **Setting `boot.loader.efi.canTouchEfiVariables = true`**: Contabo has no persistent NVRAM. This will fail silently or cause errors.
- **Using systemd-boot**: It is UEFI-only. If Contabo boots in BIOS mode, the system will not boot.
- **Generating age key from `age-keygen` instead of SSH host key**: Creates a separate key to manage. Deriving from SSH host key is simpler -- sops-nix does this automatically.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Disk partitioning | Shell scripts with `parted`/`fdisk` | disko declarative config | Reproducible, version-controlled, nixos-anywhere integration |
| Secret encryption | Manual age/GPG encrypt/decrypt scripts | sops CLI + `.sops.yaml` creation rules | sops handles multi-key encryption, YAML editing, key rotation |
| Age key derivation | Manual `age-keygen` + key distribution | `ssh-to-age` + `sops.age.sshKeyPaths` | Zero extra key management; SSH host key is the single source |
| SSH host key generation | Hope that NixOS generates the right key | `ssh-keygen -t ed25519` pre-generation | Must be deterministic for sops-nix; random generation breaks secrets |
| Boot loader configuration | Manual GRUB install scripts | NixOS `boot.loader.grub` module | Handles hybrid BIOS/UEFI, generation management, rollback |
| flake.lock management | Manual version pinning | `nix flake update` + commit lock file | Nix handles dependency resolution and lock file generation |

**Key insight:** Phase 1 is scaffolding -- every deliverable is a configuration file, not custom code. The NixOS module system, disko, and sops-nix have solved all these problems. The task is wiring them correctly, not building anything novel.

## Common Pitfalls

### Pitfall 1: sops-nix Bootstrap Chicken-and-Egg

**What goes wrong:** Secrets must be encrypted for the host's age key before deployment, but the host's SSH key (from which the age key derives) does not exist until the host is deployed.
**Why it happens:** The age public key in `.sops.yaml` must be known at encryption time. The SSH host key is generated at install time.
**How to avoid:**
1. Pre-generate the SSH host key locally: `ssh-keygen -t ed25519 -f /tmp/host-key/ssh_host_ed25519_key -N ""`
2. Derive age public key: `cat /tmp/host-key/ssh_host_ed25519_key.pub | ssh-to-age`
3. Put age public key in `.sops.yaml` as a recipient
4. Encrypt secrets with `sops --encrypt`
5. Store pre-generated key securely (NOT in the repo)
6. In Phase 2: deploy via `nixos-anywhere --extra-files /tmp/host-key-dir` (where the dir mimics `/etc/ssh/` structure)
**Warning signs:** "failed to decrypt" errors from sops-nix at activation time.
**Confidence:** HIGH (sops-nix docs, Stapelberg guide, PITFALLS.md)

### Pitfall 2: Contabo BIOS-Only Boot

**What goes wrong:** UEFI-only boot config (systemd-boot, or GRUB without `device` set) results in an unbootable system.
**Why it happens:** Contabo's KVM may not pass through UEFI firmware on all VPS tiers. Behavior varies.
**How to avoid:** Use GRUB with both BIOS boot partition (EF02) and ESP (EF00). Set `boot.loader.grub.device = "/dev/sda"` AND `boot.loader.grub.efiSupport = true` AND `boot.loader.grub.efiInstallAsRemovable = true`.
**Warning signs:** nixos-anywhere completes but SSH never comes back; black screen on VNC.
**Confidence:** MEDIUM (Contabo BIOS-only claim is single-source; hedge with hybrid config)

### Pitfall 3: Missing VirtIO Kernel Modules

**What goes wrong:** Stage 1 boot hangs waiting for root device because initrd lacks virtio drivers.
**Why it happens:** `nixos-generate-config` sometimes fails to detect virtio in VM environments. Writing config from scratch means no auto-detection.
**How to avoid:** Always include `virtio_pci`, `virtio_scsi`, `virtio_blk`, `virtio_net` in `boot.initrd.availableKernelModules`. Import `qemu-guest.nix` profile.
**Warning signs:** VNC shows "waiting for device /dev/sda" during boot.
**Confidence:** HIGH (NixOS/nixpkgs#76980, multiple community reports)

### Pitfall 4: `nix flake check` Fails with Unfetchable Secrets

**What goes wrong:** `nix flake check` tries to evaluate the full NixOS configuration, including sops-nix secret declarations. If `sops.defaultSopsFile` points to an encrypted YAML that cannot be parsed by Nix's evaluator, the check may fail.
**Why it happens:** sops-nix decryption happens at activation time, not evaluation time. The encrypted YAML is a valid Nix path reference but its content is opaque during evaluation. The module handles this gracefully -- `nix flake check` evaluates the module structure, not the secret values.
**How to avoid:** Ensure the secrets YAML file exists and is git-tracked. The file does not need to be decryptable during `nix flake check` -- sops-nix is designed to defer decryption. The check verifies configuration structure, not runtime behavior.
**Warning signs:** Evaluation errors mentioning the secrets file path. Usually means the file is missing or not git-added.
**Confidence:** HIGH (sops-nix design)

### Pitfall 5: Forgetting `git add` for New Files

**What goes wrong:** New `.nix` files are invisible to the flake evaluator. `nix flake check` fails with "file not found" or module not existing.
**Why it happens:** Nix flakes only see git-tracked files. Untracked files in the working directory are invisible.
**How to avoid:** Run `git add .` (or specific files) before any `nix flake check` or `nix build`. Make this habitual.
**Warning signs:** "error: getting status of '/nix/store/...': No such file or directory" or similar path errors.
**Confidence:** HIGH (well-documented flake behavior)

## Code Examples

### Complete sops-nix Bootstrap Workflow (Local)

This is the step-by-step workflow for setting up the secrets pipeline locally in Phase 1.

```bash
# Step 1: Pre-generate the SSH host key for the target server
mkdir -p tmp/host-key
ssh-keygen -t ed25519 -f tmp/host-key/ssh_host_ed25519_key -N "" -C "acfs-host-key"

# Step 2: Derive the age public key from the host's SSH public key
nix-shell -p ssh-to-age --run \
  'cat tmp/host-key/ssh_host_ed25519_key.pub | ssh-to-age'
# Output: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5...  (save this)

# Step 3: Also derive YOUR personal age key (for local decryption)
# From your SSH key:
nix-shell -p ssh-to-age --run \
  'ssh-add -L | grep ed25519 | ssh-to-age'
# Or from a file:
nix-shell -p ssh-to-age --run \
  'cat ~/.ssh/id_ed25519.pub | ssh-to-age'

# Step 4: Create .sops.yaml with both keys
cat > .sops.yaml << 'SOPSEOF'
keys:
  - &admin_user age1XXXXXXXXXXXXXX  # Your personal age public key
  - &acfs_host  age1YYYYYYYYYYYYYY  # Host's age public key (from step 2)
creation_rules:
  - path_regex: secrets/acfs\.yaml$
    key_groups:
      - age:
        - *admin_user
        - *acfs_host
SOPSEOF

# Step 5: Create and encrypt a secrets file
# Set up your personal age key for decryption
mkdir -p ~/.config/sops/age/
# If using SSH-derived key:
nix-shell -p ssh-to-age --run \
  'ssh-to-age -private-key < ~/.ssh/id_ed25519 >> ~/.config/sops/age/keys.txt'

# Create the encrypted secrets file
mkdir -p secrets/
nix-shell -p sops --run 'sops secrets/acfs.yaml'
# This opens your editor. Add placeholder content like:
#   example_secret: "placeholder-value-for-testing"

# Step 6: Verify local decryption works
nix-shell -p sops --run 'sops -d secrets/acfs.yaml'
# Should print the plaintext. If it does, the pipeline works.
```

**Source:** sops-nix README, ssh-to-age README, Stapelberg 2025 guide

```yaml

keys:
  # Personal key: allows YOU to edit secrets from your dev machine
  - &admin_dangirsh age1XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
  # Host key: allows the acfs server to decrypt secrets at activation time

  # All secrets for the acfs host are encrypted for both admin and host

      - age:
        - *admin_dangirsh
        - *host_acfs
```

### secrets/acfs.yaml Content Structure

When editing with `sops secrets/acfs.yaml`, the plaintext structure is:

```yaml
# Placeholder secrets for Phase 1 verification
# Real values added in later phases as services are configured
example_secret: "phase1-placeholder"
```

After encryption, the file contains only ciphertext and sops metadata. It is safe to commit.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| NixOS channels | Flakes with `flake.lock` | 2020+ (mainstream 2023+) | Reproducible, lockfile-pinned, composable inputs |
| Manual partitioning | disko declarative partitioning | 2022+ | Reproducible disk layout, nixos-anywhere integration |
| GPG for sops | age encryption | 2021+ | Simpler key management, no expiration, SSH key derivation |
| nixos-infect | nixos-anywhere | 2022+ | Clean install instead of in-place conversion, disko integration |
| Separate age keys | SSH-derived age keys via ssh-to-age | 2022+ | Zero extra key infrastructure |
| systemd-boot for servers | GRUB for VPS (hybrid) | N/A (GRUB was always the right choice for uncertain boot environments) | Works on both BIOS and UEFI |

**Deprecated/outdated patterns to avoid:**
- `nixos-infect`: Legacy, fragile, no declarative disk management
- NixOS channels (non-flake): No lockfiles, not reproducible
- GPG for sops encryption: Complex key management, expired keys break decryption
- Password-protected age keys: sops-nix decrypts at activation; cannot prompt for password

## Open Questions

1. **Contabo boot mode verification**
   - What we know: At least one community source reports BIOS-only on Contabo. The hybrid GRUB config handles both.
   - What's unclear: Whether THIS specific Contabo VPS tier boots BIOS or UEFI.
   - Recommendation: Use hybrid config (already planned). Optionally, SSH into the current Ubuntu system and check `[ -d /sys/firmware/efi ] && echo UEFI || echo BIOS` before Phase 2 deployment. This does NOT block Phase 1 work.
   - **Confidence:** MEDIUM for the specific boot mode; HIGH for the hybrid solution working regardless.

2. **`nix flake check` evaluation depth**
   - What we know: `nix flake check` evaluates flake outputs (including `nixosConfigurations`) for type correctness and basic evaluation. It does NOT build the system or test activation.
   - What's unclear: Whether certain sops-nix or disko options cause evaluation-time failures when the target hardware is not present.
   - Recommendation: Test early. If `nix flake check` fails on disko or sops options, investigate. Both are designed to evaluate cleanly without the target hardware.
   - **Confidence:** HIGH that it will work (both modules defer hardware interaction to activation/deployment time).

3. **Exact SSH authorized key value**
   - What we know: The dangirsh user has one SSH key for access. The actual public key value must be extracted from `~/.ssh/id_ed25519.pub` or `~/.ssh/authorized_keys` on the current server.
   - What's unclear: Whether there are multiple keys that should be authorized.
   - Recommendation: Extract from current server during implementation. This is a data retrieval task, not a research gap.
   - **Confidence:** HIGH (straightforward).

4. **Disk device path stability**
   - What we know: Current Ubuntu system shows `/dev/sda` from `lsblk`. Contabo uses various virtualization backends.
   - What's unclear: Whether nixos-anywhere's kexec environment will present the disk as `/dev/sda` or `/dev/vda`.
   - Recommendation: Use `/dev/sda` based on current evidence. If deployment (Phase 2) fails, check `lsblk` in the kexec environment and update. Also consider using `/dev/disk/by-id/` for more stable naming if available.
