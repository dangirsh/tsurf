<objective>
Bootstrap the complete sops-nix secrets pipeline: pre-generate the SSH host key for the target server, derive age public keys, create `.sops.yaml` with encryption rules, encrypt a secrets file, and verify local decryption works. This solves the chicken-and-egg problem where secrets must be encrypted for the host's key BEFORE the host exists.

Purpose: Without this pipeline, sops-nix cannot decrypt secrets at activation time on the deployed server. The pre-generated host key will be deployed to the server via `nixos-anywhere --extra-files` in Phase 2. The admin key enables local editing of encrypted secrets from the dev machine.

Output: `.sops.yaml`, encrypted `secrets/acfs.yaml`, pre-generated SSH host key pair in `tmp/host-key/`
</objective>

  <action>
Execute the sops-nix bootstrap workflow from the research document. This is a multi-step CLI workflow that must run in order.

**Step 1: Pre-generate SSH host key**
```bash
mkdir -p tmp/host-key
ssh-keygen -t ed25519 -f tmp/host-key/ssh_host_ed25519_key -N "" -C "acfs-host-key"
```
This creates the key pair that will be deployed to the server in Phase 2 via `nixos-anywhere --extra-files`.

**Step 2: Derive the HOST's age public key**
```bash
nix-shell -p ssh-to-age --run 'cat tmp/host-key/ssh_host_ed25519_key.pub | ssh-to-age'
```
Save the output (starts with `age1...`). This is the `&host_acfs` anchor in `.sops.yaml`.

**Step 3: Derive the ADMIN's age public key**
The admin key allows the human user to decrypt and edit secrets locally. Derive from the personal SSH key:
```bash
nix-shell -p ssh-to-age --run 'cat ~/.ssh/id_ed25519.pub | ssh-to-age'
```
Save the output. This is the `&admin_dangirsh` anchor in `.sops.yaml`.

If `~/.ssh/id_ed25519.pub` does not exist or produces an error, try `~/.ssh/authorized_keys` instead (filter for the ed25519 line):
```bash
grep ed25519 ~/.ssh/authorized_keys | head -1 | nix-shell -p ssh-to-age --run 'ssh-to-age'
```
Note: The server's `~/.ssh/id_ed25519` is a parts-agent key, not the human user's key. The human user's key is in `~/.ssh/authorized_keys`. For the admin age key, we need the public key that the human uses to SSH in. Derive from whichever ed25519 public key represents the human admin.

**Step 4: Create .sops.yaml**
Write `.sops.yaml` at the repository root with both age key anchors and a creation rule matching `secrets/acfs.yaml`. Use the EXACT age public key strings from steps 2 and 3 (no placeholders):

```yaml
keys:
  - &admin_dangirsh age1XXXX...  # actual value from step 3
  - &host_acfs      age1YYYY...  # actual value from step 2
creation_rules:
  - path_regex: secrets/acfs\.yaml$
    key_groups:
      - age:
        - *admin_dangirsh
        - *host_acfs
```

**Step 5: Set up admin age private key for local decryption**
The admin needs a private age key in `~/.config/sops/age/keys.txt` to decrypt secrets locally:
```bash
mkdir -p ~/.config/sops/age/
nix-shell -p ssh-to-age --run 'ssh-to-age -private-key < ~/.ssh/id_ed25519 >> ~/.config/sops/age/keys.txt'
```
If `~/.ssh/id_ed25519` is the parts-agent key (not the admin's key), this step may not produce the correct decryption key. In that case, the admin must set up their own age key separately. Document what was done and any limitations.

**Step 6: Create and encrypt secrets/acfs.yaml**
Replace the placeholder `secrets/acfs.yaml` from plan 01 with a real sops-encrypted file:
```bash
nix-shell -p sops --run 'sops secrets/acfs.yaml'
```
This opens an editor. Add a placeholder secret:
```yaml
example_secret: "phase1-placeholder-for-testing"
```
Save and close. sops will encrypt the file in-place.

If the editor-based workflow is not possible in the CLI environment, create the encrypted file directly:
```bash
echo 'example_secret: "phase1-placeholder-for-testing"' | nix-shell -p sops --run 'sops --encrypt --input-type yaml --output-type yaml /dev/stdin' > secrets/acfs.yaml
```

**Step 7: git add .sops.yaml secrets/acfs.yaml**
Both files must be tracked for `nix flake check` and for sops-nix to find them.

IMPORTANT: The `tmp/host-key/` directory should NOT be committed to git. It contains private key material. Verify that `.gitignore` or the project's ignore rules exclude `tmp/`. If not, add `tmp/` to `.gitignore`.
