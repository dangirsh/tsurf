# Phase 66: Secret Placeholder Proxy Module - Context

**Gathered:** 2026-03-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Extract the Phase 22 secret-proxy trick into a generic, well-tested NixOS module. Research ironclaw, gondolin, and all relevant secret management approaches first — best case: an existing tool solves this and we adopt it. Second best: clean Rust + NixOS implementation that's immediately useful for anyone running untrusted agents on NixOS. The module lets sandboxed agents use placeholder tokens while a proxy transparently re-injects real secrets, with per-secret domain allowlisting to block exfiltration.

</domain>

<decisions>
## Implementation Decisions

### Configuration surface
- Per-service grouping: each service declares its own proxy secrets inline (e.g., `services.secretProxy.services.claw-swap.secrets.anthropic-api-key = { ... }`)
- Placeholder tokens auto-generated by the module with user override option for cases needing specific formats
- Opinionated and simple: dedicated system user, reads from standard path (like `/run/secrets/secret-proxy/*`), systemd service with hardening built-in. User just declares secrets and allowed domains.

### Domain allowlisting
- Strict default-deny: every secret declaration MUST include `allowedDomains` — no exceptions
- Exact domain matching only — no wildcards, no subdomain patterns
- On violation: block the request and return an HTTP error (e.g., 403) with logging
- Validation at both Host header AND TLS SNI level to prevent IP-based bypasses

### Proxy architecture
- Rust implementation in a separate repo (`dangirsh/secret-proxy` or similar)
- CONNECT tunnel + TLS termination for transparent HTTPS interception
- Module auto-generates a local CA certificate and handles trust wiring
- Injection mode (headers-only vs body rewriting): research decides based on real-world API patterns
- Proxy mode (HTTP_PROXY vs BASE_URL): research decides based on what agents/SDKs actually support
- CA trust inside bwrap sandboxes: research decides the most practical approach

### Backend abstraction
- Purely backend-agnostic: module takes file paths for secrets, no sops-nix or agenix code in the module
- Secret source interface: standard NixOS pattern (research decides exact mechanism — file path or env, whichever is most standard/simple/secure for NixOS)
- Restart-on-change for secret rotation — no hot reload complexity

### Claude's Discretion
- Per-service proxy port vs shared proxy routing (whichever is more secure/simple)
- Environment wiring approach (EnvironmentFile vs Nix attrset output)
- Exact Rust crate choices for proxy implementation
- CA certificate storage location and trust injection method for bwrap

</decisions>

<specifics>
## Specific Ideas

- "Best case: it already exists and we don't need to build it!" — research existing tools (ironclaw, gondolin/earendil-works, mitmproxy, etc.) before building
- "Second best: clean NixOS + Rust implementation that would be immediately useful and easy to use for anyone running untrusted agents (all agents) on NixOS"
- User is unfamiliar with real-world protocol constraints — research must surface these and inform decisions left to "research decides"
- Plug-and-play with sops-nix + bwrap is the primary use case — should "just work" with minimal config
- Priority order: (1) cover majority of sandboxed agent use cases, (2) security, (3) simplicity
- Phase 22's Python stdlib proxy is the reference implementation to generalize from

</specifics>
