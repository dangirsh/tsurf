## Objective

Wire the new generic module to replace the Phase 22 consumer in the private overlay,
update eval checks, and update live tests. After this plan, the private overlay's
claw-swap service declares `services.secretProxy.services.claw-swap = { ... }` instead
of consuming the old hardcoded `anthropic-secret-proxy` service.

The public repo gets updated eval checks that validate the new module schema (with empty
default) and updated live tests that reference `secret-proxy-<name>` service name instead
of `anthropic-secret-proxy`.

## must_haves

- Private overlay has `services.secretProxy.services.claw-swap` declaration consuming
  `config.sops.secrets."anthropic-api-key".path` as the `secretFile`
- Old `sops.templates."secret-proxy-env"` declaration removed from private overlay (or never existed there)
- Old `systemd.services.anthropic-secret-proxy` no longer declared anywhere in public or private configs
- Eval checks updated: `expected-services-neurosys` no longer asserts `anthropic-secret-proxy` service;
  if it previously asserted that service, replace with a generic secretProxy check or remove the specific assertion
- Live test `api-endpoints.bats` updated: test for `secret-proxy-claw-swap` port 9091 instead of
  `anthropic-secret-proxy`
- Live test `service-health.bats` updated: assert `secret-proxy-claw-swap.service` is active
  instead of `anthropic-secret-proxy.service` (if such a test existed)
- `nix flake check` passes in public repo after all changes
- Private overlay evaluates cleanly (`nix flake check` or `nix eval .#nixosConfigurations.neurosys.config.system.build.toplevel`)

## Post-Plan: Deploy Sequence

After this plan's commits are complete, the deploy sequence is:
1. Deploy private overlay to neurosys: `cd /data/projects/private-neurosys && ./scripts/deploy.sh --node neurosys`
2. Verify on server: `systemctl status secret-proxy-claw-swap` is active
3. Verify claw-swap still works: check claw-swap service logs for successful API calls
4. Run live tests: `nix run .#test-live -- --host neurosys`

Note: During the deploy, the old `anthropic-secret-proxy` service stops and the new
`secret-proxy-claw-swap` service starts. There is a brief window where agents may fail
to reach the API. This is acceptable — agent tasks retry on connection failure.

## Risks

### Private overlay eval fails due to missing secretProxy module import (MEDIUM)
The private overlay imports public modules explicitly. If `modules/secret-proxy.nix` is
in the private overlay's import list (which it should be, since it's a public module),
the new schema is automatically available. If it was previously in `disabledModules`,
that must be removed (Task B). If the private overlay doesn't import the module at all,
add it to the imports list.

### bwrapArgs consumed before service declaration (LOW)
If the private overlay's claw-swap module references
`config.services.secretProxy.services.claw-swap.bwrapArgs` before that service is declared,
NixOS module system evaluation may fail with an infinite recursion. Solution: ensure the
`secretProxy` declaration and the `bwrapArgs` consumer are in the same module (or the
`bwrapArgs` is computed lazily via the option's `config` block, which it is — see Plan 66-02).

### anthropic-secret-proxy.service still running on server after deploy (LOW)
If the old service was managed by a separate `systemd.services.anthropic-secret-proxy`
declaration (which no longer exists after this plan), systemd will stop it and start
`secret-proxy-claw-swap`. No residual state issue — both services use ephemeral tmpfs.

### ANTHROPIC_BASE_URL still hardcoded somewhere in agent sandbox args (LOW)
If a bwrap wrapper script hardcodes `ANTHROPIC_BASE_URL=http://127.0.0.1:9091` as a
string rather than reading it from `bwrapArgs`, it will still work (port unchanged at 9091)
but won't benefit from future port changes. Track down all `ANTHROPIC_BASE_URL` references
in private overlay and ensure they derive from `bwrapArgs`.
