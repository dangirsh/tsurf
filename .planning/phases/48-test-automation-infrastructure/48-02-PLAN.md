# Plan 48-02: Deep Validation + Private Overlay Tests + Agent Integration Polish

## Goal

Extend the test suite from Plan 01 with deep validation tests for agentd, monitoring,
impermanence, sandbox, and networking. Add structured JSON output for agent consumption,
CI integration via GitHub Actions, guard hook integration, private overlay test
framework, and documentation. The result is a comprehensive, agent-optimized test
suite that gives maximal feedback for error-correction.

## Decisions Made (from Research)

- **Deep tests separate from core:** Tier 3 tests are more fragile (depend on
  specific runtime state) and slower. Separate files make it easy to run core
  tests quickly and deep tests when needed.
- **JSON output alongside TAP:** BATS TAP output is standard but hard to
  programmatically parse for agents. A `--json` flag in `run-tests.sh` produces
  a structured JSON summary alongside TAP output.
- **CI runs eval checks only:** GitHub Actions can run `nix flake check` without
  secrets or SSH access. Live tests require Tailscale and are not CI-runnable.
- **Private overlay tests in private repo:** Test files reference private
  services (nginx vhosts, openclaw containers, Matrix). The pattern is documented
  here but implementation is in private-neurosys.
- **Guard hook writes test-status automatically:** `scripts/run-tests.sh` already
  writes `.test-status` (Plan 01). Plan 02 documents the integration pattern.

## must_haves

1. 20+ additional BATS live tests in dedicated files: agentd (5+), monitoring (5+), impermanence (4+), sandbox (4+), networking (3+)
2. `scripts/run-tests.sh --json` produces JSON summary with test name, status, and error context for failed tests (extracted from BATS TAP `# ` comment lines after `not ok`)
3. `.github/workflows/test.yml` runs `nix flake check` on every PR (no secrets needed)
4. CLAUDE.md updated with testing conventions section (how to run tests, what passes mean)
5. Private overlay test pattern documented with example files (not implemented in public repo)
6. All existing tests from Plan 01 still pass after changes
7. `nix flake check` still passes

      # --- Retry helper ---
      # Retry a command up to N times with delay between attempts.
      # Usage: retry 3 2 ssh_cmd "$HOST" "systemctl is-active tailscaled"
      retry() {
        local max_attempts="$1"; shift
        local delay="$1"; shift
        local attempt=1
        while [[ $attempt -le $max_attempts ]]; do
          if "$@"; then
            return 0
          fi

      Add a function to convert BATS TAP output to JSON summary.
      In BATS TAP output, failure details appear as `# ` comment lines after
      a `not ok N - test name` line. The awk captures these and adds them as
      an `"error"` field in the JSON object for failed tests, giving agents
      maximum feedback for error-correction.

      ```bash
      tap_to_json() {
        local tap_output="$1"
        local host="$2"
        # Parse TAP output into JSON, capturing error context from # comment
        # lines that follow "not ok" entries (BATS failure details).
        echo "$tap_output" | awk -v host="$host" '
        BEGIN { print "["; first=1; in_fail=0; error_lines="" }
        # Flush previous failed test with accumulated error context
        function flush_fail() {
          if (in_fail) {
            gsub(/\\n$/, "", error_lines);
            if (error_lines != "") {
              printf ", \"error\": \"%s\"}", error_lines
            } else {
              printf "}"
            }
            in_fail=0;
            error_lines=""
          }
        }
        /^ok / {
          flush_fail();
          if (!first) print ",";
          first=0;
          sub(/^ok [0-9]+ /, "");
          # Strip trailing " # skip ..." for skipped tests
          skip_reason="";
          if (match($0, / # skip(.*)$/)) {
            skip_reason=substr($0, RSTART+7);
            $0=substr($0, 1, RSTART-1);
            gsub(/"/, "\\\"", skip_reason);
          }
          gsub(/"/, "\\\"");
          if (skip_reason != "") {
            printf "  {\"test\": \"%s\", \"status\": \"skip\", \"host\": \"%s\", \"reason\": \"%s\"}", $0, host, skip_reason
          } else {
            printf "  {\"test\": \"%s\", \"status\": \"pass\", \"host\": \"%s\"}", $0, host
          }
        }
        /^not ok / {
          flush_fail();
          if (!first) print ",";
          first=0;
          sub(/^not ok [0-9]+ /, "");
          gsub(/"/, "\\\"");
          printf "  {\"test\": \"%s\", \"status\": \"fail\", \"host\": \"%s\"", $0, host
          in_fail=1;
          error_lines=""
        }
        /^# / && in_fail {
          line=substr($0, 3);
          gsub(/"/, "\\\"", line);
          gsub(/\t/, " ", line);
          error_lines = error_lines line "\\n"
        }
        END {
          flush_fail();
          print "\n]"
        }
        '
      }
      ```

## Risks and Mitigations

### Sandbox isolation tests may behave differently across bwrap versions (LOW)

Different bubblewrap versions may have slightly different namespace behavior.

**Mitigation:** The sandbox tests use the exact same bwrap from the host's nix
store. Tests verify isolation behavior, not bwrap version-specific features.

### BATS file count may cause slow test-live execution (LOW)

9 BATS files with 60+ tests may take 2-3 minutes over SSH.

**Mitigation:** BATS runs tests sequentially within a file but files can be
targeted individually. The `test-live` wrapper supports passing specific BATS
files as arguments.

### GitHub Actions Nix eval may be slow (MEDIUM)

Full `nix flake check` downloads nixpkgs and evaluates both host configs. This
can take 5-10 minutes without cache.

**Mitigation:** Use `cachix/install-nix-action` which provides binary cache
integration. nixpkgs evaluation is cached after first run. Consider adding
Cachix cache in a future iteration.

### jq path differences between remote and local (LOW)

The `remote jq` pattern assumes jq is in the remote PATH. If a BATS helper
pipes through `remote jq` but jq is missing, tests fail confusingly.

**Mitigation:** jq is in `environment.systemPackages` via `base.nix` on both
hosts. Tests use `remote jq` consistently to ensure jq runs on the host where
the data is.
