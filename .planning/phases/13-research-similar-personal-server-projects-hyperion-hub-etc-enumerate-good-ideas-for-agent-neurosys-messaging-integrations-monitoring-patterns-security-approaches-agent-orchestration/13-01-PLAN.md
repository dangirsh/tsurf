**PRIORITY-RANKED IDEAS (present each with: what, effort, value, NixOS readiness, recommendation):**

**Tier 1 — High Value, Low/Medium Effort (recommended USE):**

1. **ntfy Push Notifications** [LOW effort, HIGH value]
   - Self-hosted HTTP pub-sub notification service. NixOS module: `services.ntfy-sh`.
   - Integrates with: deploy script, fail2ban, restic backups, agent-spawn, Home Assistant, Alertmanager.
   - Phone app available (iOS/Android). Single `curl` command to send.
   - Recommendation: USE — foundational notification layer for everything else.

2. **Prometheus + node_exporter + Grafana** [MEDIUM effort, HIGH value]
   - Declarative system monitoring. All have first-class NixOS modules.
   - Start minimal: node_exporter (CPU/memory/disk) + Prometheus (storage) + Grafana (dashboards).
   - Recommendation: USE — standard monitoring stack, well-documented NixOS setup.

3. **MCP-NixOS Server** [LOW effort, MEDIUM value]
   - MCP server providing real-time NixOS package/option lookup for Claude sessions.
   - Add to `.mcp.json` in project root. Prevents NixOS config hallucination.
   - Recommendation: USE — immediate productivity improvement for NixOS work.

4. **Claude Code Agent Teams** [LOW effort, MEDIUM value]
   - Native multi-agent orchestration in Claude Code. Enable via env var.
   - `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=true` — shared task list, parallel work.
   - Recommendation: EVALUATE — test in agent-spawn session before building workflows around it.

**Tier 2 — Medium Value, Low/Medium Effort (recommended CONSIDER):**

5. **Uptime Kuma** [LOW effort, MEDIUM value]
   - Lightweight service health dashboard. nixpkgs package exists.
   - Monitor Docker containers, Syncthing, Home Assistant, SSH.
   - Overlaps with Prometheus alerting but much simpler for "is it running?" checks.
   - Recommendation: CONSIDER — useful if you want a simple status page.

6. **CrowdSec** [MEDIUM effort, MEDIUM value]
   - Collaborative threat intelligence + intrusion prevention.
   - Community NixOS module (flake, not mainline). Complements existing fail2ban.
   - Adds shared blocklists — your bans help others, their bans help you.
   - Recommendation: CONSIDER — evaluate community NixOS module maturity first.

7. **endlessh-go SSH Tarpit** [LOW effort, LOW value]
   - SSH tarpit on port 22, real SSH on non-standard port. NixOS module: `services.endlessh-go`.
   - Wastes attacker resources. Minimal overhead. Feeds CrowdSec if deployed.
   - Less useful if SSH is primarily Tailscale-only access.
   - Recommendation: CONSIDER — low effort, low risk, pairs with CrowdSec.

**Tier 3 — Deferred (not needed now, conditions noted):**

8. **Caddy Reverse Proxy** [MEDIUM effort, LOW value currently]
   - Auto-HTTPS, reverse proxy for internal services. NixOS module exists.
   - Only needed when services need DNS-based routing (currently all Tailscale IP:port).
   - Recommendation: DEFER until services need internet-facing exposure or DNS names.

9. **Authelia SSO** [MEDIUM effort, LOW value currently]
   - Forward-auth single sign-on for web services behind reverse proxy.
   - Only valuable when multiple web services are exposed. Tailscale provides implicit auth.
   - Recommendation: DEFER until services are internet-facing.

10. **Headscale** [HIGH effort, LOW value currently]
    - Self-hosted Tailscale coordination server. Eliminates dependency on Tailscale's hosted control.
    - Tailscale free tier is sufficient. Adds operational complexity.
    - Recommendation: DEFER unless Tailscale pricing/policies change.

11. **Loki + Alloy Log Aggregation** [MEDIUM effort, LOW value currently]
    - Centralized log search in Grafana. Promtail is deprecated (use Alloy or Fluent Bit).
    - Single server means `journalctl` works fine. Most useful for Docker container + agent audit logs.
    - Check NixOS Alloy packaging before committing.
    - Recommendation: DEFER until specific log search needs arise.

**ARCHITECTURE PATTERNS (from research, for reference):**
- Layered monitoring: Layer 1 (metrics) -> Layer 2 (alerting+ntfy) -> Layer 3 (logs) — add layers incrementally
- Notification hierarchy: CRITICAL (phone push), WARNING (notification), INFO (badge only)
- vars.nix centralized config: single file for all ports/paths/hostnames used across modules

**PITFALLS TO AVOID (from research):**
- Over-engineering monitoring: more monitoring services than application services
- Notification fatigue: start with critical alerts only (disk >90%, service down >5min, backup failure)
- Agent orchestration complexity creep: Claude Code Agent Teams first, custom tooling only if insufficient
- Promtail is EOL March 2026: use Alloy or Fluent Bit for any new log collection
