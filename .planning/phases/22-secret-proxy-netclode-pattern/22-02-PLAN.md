  <action>
Create `modules/secret-proxy.nix`. This module declares the systemd service that runs secret-proxy outside all agent sandboxes.

```nix
# modules/secret-proxy.nix
# @decision PROXY-22-03: secret-proxy runs as dangirsh (not root) because it needs
#   /run/secrets/ access (sops-nix secrets are owned by dangirsh). Service is outside
#   all bubblewrap sandboxes — dangirsh's systemd user session manages it.
# @decision PROXY-22-04: RuntimeDirectory=secret-proxy-ca creates /run/secret-proxy-ca/
#   owned by dangirsh with mode 0755. The proxy writes ca.crt there at startup.
#   Sandboxed agents bind-mount this dir read-only to trust the ephemeral CA.
# @decision PROXY-22-05: PROXY_TOKEN loaded from sops-nix secret (proxy-token).
#   Prevents unauthenticated processes on localhost from triggering key injection.
#   Defense-in-depth: proxy listens only on 127.0.0.1, but token validation adds
#   a second layer so a compromised but sandboxed agent cannot spoof sessions.
{ config, pkgs, ... }:
let
  secret-proxy = pkgs.callPackage ../packages/secret-proxy.nix {};
in
{
  # Systemd service runs as dangirsh so it can read sops-nix secrets
  # (secrets are owned/readable by dangirsh, not root).
  systemd.services.secret-proxy = {
    description = "Secret proxy — MITM TLS proxy for agent API key injection";
    after = [ "network.target" ];
    wants = [ "multi-user.target" ];
    wantedBy = [ "multi-user.target" ];

    serviceConfig = {
      User = "dangirsh";
      Group = "users";
      ExecStart = "${secret-proxy}/bin/secret-proxy";
      Restart = "on-failure";
      RestartSec = "5s";

      # Creates /run/secret-proxy-ca/ owned by dangirsh (mode 0755).
      # Proxy writes ca.crt here at startup; agents bind-mount it read-only.
      RuntimeDirectory = "secret-proxy-ca";
      RuntimeDirectoryMode = "0755";

      # Load proxy-token from sops-nix rendered env file.
      # secret-proxy reads PROXY_TOKEN env var at startup.
      EnvironmentFile = config.sops.templates."proxy-token-env".path;
    };
  };
}
```

Note on the `after` dependency: sops-nix uses a NixOS activation script, not a dedicated systemd service named `sops-nix.service`. Therefore `after = [ "network.target" ]` is correct. The sops secrets are written during system activation (before multi-user.target), so the EnvironmentFile will be present by the time this service starts.

Note on `EnvironmentFile` vs direct env var: sops-nix renders the proxy-token as an env file (`proxy-token-env`) so it can be loaded by systemd's `EnvironmentFile` directive. The env file must contain `PROXY_TOKEN=<value>` (no export keyword). This is consistent with the sops.templates pattern used elsewhere in this codebase.

The module is intentionally small — it does not add the sops secret declaration (that stays in `modules/secrets.nix`).

Update `home/bash.nix` to wrap the real API key exports in an `AGENT_SANDBOX` guard. This prevents real keys from being visible in any shell context that sets AGENT_SANDBOX=1, providing defense-in-depth even if sandbox configuration changes.

**Current pattern in home/bash.nix:**
```bash
export GH_TOKEN="$(cat /run/secrets/github-pat 2>/dev/null)"
export ANTHROPIC_API_KEY="$(cat /run/secrets/anthropic-api-key 2>/dev/null)"
export OPENAI_API_KEY="$(cat /run/secrets/openai-api-key 2>/dev/null)"
```

**Replace with guarded pattern:**
```bash
# @decision BASH-SEC-01: Real API keys are only exported outside agent sandboxes.
#   bwrap uses --clearenv so sandboxed agents don't inherit these even without the guard.
#   This guard is defense-in-depth: if sandbox config changes and --clearenv is removed,
#   or if a shell is spawned inside the sandbox with bash.nix sourced, real keys won't leak.
#   agent-spawn sets AGENT_SANDBOX=1 before entering bwrap (cleared by --clearenv anyway,
#   but the intent is documented for future maintainers).
if [ -z "${AGENT_SANDBOX:-}" ]; then
  export GH_TOKEN="$(cat /run/secrets/github-pat 2>/dev/null)"
  export ANTHROPIC_API_KEY="$(cat /run/secrets/anthropic-api-key 2>/dev/null)"
  export OPENAI_API_KEY="$(cat /run/secrets/openai-api-key 2>/dev/null)"
fi
```

Additionally, in `modules/agent-compute.nix`, the agent-spawn script should set `AGENT_SANDBOX=1` in the environment before calling bwrap. Since bwrap uses `--clearenv`, this variable is cleared inside the sandbox automatically — but it documents intent and activates the guard for any shell started outside bwrap but within an agent-spawn process tree:

In the agent-spawn script, before the bwrap invocation:
```bash
# Signal to shell dotfiles that we are in an agent-spawn context.
# This activates the AGENT_SANDBOX guard in home/bash.nix (real keys not exported).
# Note: --clearenv in bwrap clears this for the sandboxed process itself.
export AGENT_SANDBOX=1
```
