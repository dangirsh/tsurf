<task type="auto">
  <name>Task 3: Declare cachix-auth-token in private secrets.nix</name>
  <files>/data/projects/private-neurosys/modules/secrets.nix</files>
  <action>
    Add the sops secret declaration to `/data/projects/private-neurosys/modules/secrets.nix`:

    ```nix
    sops.secrets."cachix-auth-token" = {
      sopsFile = ../secrets/neurosys.yaml;
    };
    ```

    This secret is Contabo-only (deploy.sh gates Cachix push on `$NODE == "neurosys"`), but declaring it in the shared secrets.nix with an explicit sopsFile pointing to neurosys.yaml means OVH will ignore it (sops-nix only decrypts if the host has the age key for the referenced sopsFile).

    WAIT -- sops-nix will FAIL if the secret can't be decrypted. Since secrets.nix is in commonModules (shared by both hosts), and cachix-auth-token is only in neurosys.yaml (which OVH can't decrypt), this will break OVH builds.

    **Fix:** Use a conditional based on what's available. Two options:

    Option A: Add cachix-auth-token to BOTH sops files (neurosys.yaml and ovh.yaml) with the same value. Simple but duplicates the secret.

    Option B: Guard the declaration with a host-specific condition. But NixOS modules shouldn't have hostname guards (that's what we just removed from matrix.nix).

    Option C: Declare it in a Contabo-only module. But there's no Contabo-specific secrets file.

    **Best approach: Option A.** Add to both sops files. The token is not security-sensitive beyond the repos themselves, and duplication is simpler than conditional logic.

    ```bash
    cd /data/projects/private-neurosys
    # Get the token value from neurosys.yaml
    TOKEN=$(sops --decrypt --extract '["cachix-auth-token"]' secrets/neurosys.yaml)
    # Add to ovh.yaml too
    sops --set '["cachix-auth-token"] "'"$TOKEN"'"' secrets/ovh.yaml
    ```

    Then declare in secrets.nix WITHOUT specifying sopsFile (it will use the default sops file for each host):
    ```nix
    sops.secrets."cachix-auth-token" = {};
    ```

    BUT the default sops file might not be set. Check what other secrets in secrets.nix use. Looking at the existing file, some secrets use explicit `sopsFile = ../secrets/neurosys.yaml` and others don't. The ones without explicit sopsFile rely on the host's default sopsFile.

    Check if `sops.defaultSopsFile` is set in the host configs. If so, use `{}`. If not, the safest approach is to declare two secrets with the correct sopsFile each.

    Actually, the SIMPLEST approach: just add it to neurosys.yaml only, and declare it in a way that only applies on Contabo. We can add it directly to one of the contaboModules (e.g., create a one-liner or add to an existing Contabo-only module).

    Looking at the private flake.nix, contaboModules is a list of module paths. We could add the declaration to any Contabo-only module. The cleanest spot: add it inline in the flake.nix contaboModules list:

    ```nix
    contaboModules = [
      ...existing modules...
      { sops.secrets."cachix-auth-token".sopsFile = ../secrets/neurosys.yaml; }
    ];
    ```

    Actually inline modules in a list is fine in Nix. But `../secrets/neurosys.yaml` path resolution might be tricky inline. Better to create a minimal `./modules/cachix.nix`:

    NO -- per simplicity conventions, no new module for <20 lines. Add it inline or to an existing Contabo module.

    **Final decision:** Add to private overlay's flake.nix as an inline module in contaboModules:
    ```nix
    { sops.secrets."cachix-auth-token".sopsFile = self + "/secrets/neurosys.yaml"; }
    ```

    Using `self + "/secrets/..."` resolves correctly in flake context.
