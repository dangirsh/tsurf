# nix-secret-proxy

A lightweight HTTP proxy that substitutes placeholder API keys with real credentials. Designed for sandboxed agents that need to call external APIs without ever seeing real keys.

## The pattern

Sandboxed agents receive a **placeholder token** and a local `BASE_URL`. The proxy intercepts every outbound API request, replaces the placeholder with the real credential, and forwards the request over HTTPS. The real key never enters the sandbox environment — not in env vars, logs, or the Nix store.

This pattern was independently described by Stanislas Polu in his [Netclode post](https://stanislas.blog/2026/02/netclode-self-hosted-cloud-coding-agent/#secret-proxy-api-keys-never-enter-the-sandbox). His implementation adds HTTPS MITM and per-caller identity validation on top. This repo provides a simpler variant: HTTP on loopback with OS-level isolation (loopback binding + bubblewrap) as the trust boundary.

See [docs/architecture.md](docs/architecture.md) for the full pattern description, security model, and comparison with Netclode.

## Quick start

### NixOS (recommended)

```nix
imports = [ inputs.nix-secret-proxy.nixosModules.default ];

services.secretProxy.services.my-agent = {
  port = 9091;
  placeholder = "sk-ant-api03-placeholder";
  secrets.anthropic = {
    headerName = "x-api-key";
    secretFile = config.sops.secrets."anthropic-api-key".path;
    allowedDomains = [ "api.anthropic.com" ];
  };
};
```

The module generates a systemd service, a dedicated system user, and a `bwrapArgs` attribute for sandbox wiring. See [docs/deployment-nixos.md](docs/deployment-nixos.md) for full usage.

### Docker Compose

```yaml
services:
  secret-proxy:
    image: ghcr.io/dangirsh/nix-secret-proxy:latest
    volumes:
      - /run/secrets/anthropic-api-key:/run/secrets/anthropic-api-key:ro
    environment:
      SECRET_PROXY_CONFIG: /etc/secret-proxy/config.toml
```

See [docs/deployment-docker.md](docs/deployment-docker.md) for a complete example with config file and agent wiring.

### Plain systemd

```ini
[Service]
ExecStart=/usr/local/bin/secret-proxy --config /etc/secret-proxy/config.toml
LoadCredential=anthropic-api-key:/run/secrets/anthropic-api-key
```

See [docs/deployment-systemd.md](docs/deployment-systemd.md) for a full unit and credential loading walkthrough.

## Configuration reference

The binary reads a single TOML file passed via `--config <path>`.

```toml
port = 9091
placeholder = "sk-ant-api03-placeholder"

[[secret]]
header = "x-api-key"
name = "anthropic"
file = "/run/secrets/anthropic-api-key"
allowed_domains = ["api.anthropic.com"]
```

| Field | Type | Description |
|-------|------|-------------|
| `port` | integer | Loopback port the proxy listens on |
| `placeholder` | string | Placeholder token value to match and replace |
| `[[secret]]` | array | One block per secret/upstream pair |
| `secret.header` | string | HTTP request header to inject the real key into |
| `secret.name` | string | Human-readable name for log messages |
| `secret.file` | string | Path to file containing the real secret (trimmed, must be non-empty) |
| `secret.allowed_domains` | array of string | Domains the proxy will forward to; first entry is the upstream host |

## Security

- Real keys are read from files at startup and held in memory; never stored in the Nix store, env vars, or CLI arguments
- Requests with a `Host` header not in `allowed_domains` receive HTTP 403 — agents cannot redirect traffic to arbitrary hosts
- The upstream destination is always `allowed_domains[0]`, fixed at config load time
- Each NixOS service runs as a dedicated system user with full systemd sandboxing
- Loopback-only binding (`127.0.0.1`) — not reachable from the network

Known limitations and degraded-mode issues are catalogued in [docs/known-issues.md](docs/known-issues.md).

```markdown
# Architecture: API Key Placeholder Substitution Proxy

## What problem does this solve?

Sandboxed agents (bubblewrap, containers, CI runners) need to call external APIs. Passing a real API key via env var defeats isolation — the agent can exfiltrate it, log it, or pass it to sub-processes. Removing the key entirely means the agent cannot call the API at all.

The placeholder substitution proxy threads the needle: the agent receives a fake key that looks syntactically valid but has no real value. The proxy, running outside the sandbox, replaces the fake key with the real one on every outbound request. The sandbox boundary is never crossed by a real credential.

## Prior art and attribution

This pattern was independently described by Stanislas Polu in his January 2026 Netclode post:
[https://stanislas.blog/2026/02/netclode-self-hosted-cloud-coding-agent/#secret-proxy-api-keys-never-enter-the-sandbox](https://stanislas.blog/2026/02/netclode-self-hosted-cloud-coding-agent/#secret-proxy-api-keys-never-enter-the-sandbox)

His Netclode implementation adds:
- **HTTPS MITM** with a local CA certificate distributed into the sandbox
- **ServiceAccount token identity validation** — the proxy authenticates the caller before substituting a key, enabling per-caller key pools
- Per-caller attribution and rate limiting

The nix-secret-proxy implementation is a simpler variant:
- **HTTP on loopback** — no CA certificate distribution, no TLS for the local leg
- **OS-level isolation** as the trust boundary (loopback binding + bubblewrap), not cryptographic identity
- No per-caller attribution — all agents sharing a proxy instance share the same upstream key

The tradeoff: lower setup complexity, no certificate management, but no per-caller attribution or key rotation.

## Request flow

```
┌─────────────────────────────────────────────┐
│                 Sandbox (bwrap)             │
│                                             │
│  Agent process                              │
│    ANTHROPIC_BASE_URL=http://127.0.0.1:9091 │
│    x-api-key: sk-ant-api03-placeholder      │──────────────────────────────────────────┐
│                                             │  HTTP POST http://127.0.0.1:9091/v1/...  │
└─────────────────────────────────────────────┘                                          │
                                                                                         ▼
                                              ┌──────────────────────────────────────────────────┐
                                              │             secret-proxy (outside sandbox)        │
                                              │                                                   │
                                              │  1. Extract Host header → "api.anthropic.com"     │
                                              │  2. Check against allowed_domains allowlist → OK  │
                                              │  3. Strip x-api-key and authorization headers     │
                                              │  4. Insert x-api-key: <real key from file>        │
                                              │  5. Forward as HTTPS to api.anthropic.com         │
                                              │  6. Stream response back to agent                 │
                                              └──────────────────────────────────────────────────┘
                                                                                         │
                                                       HTTPS with real key               │
                                                       ──────────────────────────────────▼
                                                       api.anthropic.com
```

The real key is loaded from a file at startup and held in process memory. It is never written to disk, logged, or passed via CLI arguments.

## Component responsibilities

### `src/main.rs` — startup and config loading

- Parses `--config <path>` from CLI
- Reads the TOML config file and each secret file (trims whitespace, fails fast on empty)
- Builds `ProxyState` (shared across all requests) and binds the axum listener
- Bind address is hardcoded to `127.0.0.1:<port>` (see [known-issues.md](known-issues.md) for the Docker implication)

### `src/config.rs` — TOML schema

Struct `Config { port, placeholder, secret: Vec<SecretConfig> }`. Each `SecretConfig` has `header`, `name`, `file`, `allowed_domains`.

### `src/proxy.rs` — request handler

- Extracts `Host` header (strips port suffix for matching)
- Finds the matching `LoadedSecret` by `allowed_domains` membership; returns 403 if none match
- Strips `x-api-key`, `authorization`, `host`, `content-length`, `transfer-encoding` from forwarded headers
- Injects the real key into the configured header
- Builds the upstream URL as `https://<allowed_domains[0]><path_and_query>` — the upstream host is always config-fixed, not runtime-user-supplied (SSRF prevention)
- Streams the request body using `wrap_stream` and the response using `bytes_stream` — no buffering, SSE passes through correctly

### `module.nix` — NixOS integration

- Declares `services.secretProxy.services.<name>` options
- Generates a TOML config file in the Nix store (with secret *paths* — not secret values)
- Creates one systemd unit and one dedicated system user per declared service
- Computes `bwrapArgs` as a read-only list attribute: `["--setenv" "ANTHROPIC_BASE_URL" "http://127.0.0.1:<port>"]`
- Applies full systemd hardening: `NoNewPrivileges`, `ProtectSystem=strict`, `MemoryDenyWriteExecute`, `CapabilityBoundingSet=""`, etc.

## Security model

### What the proxy protects against

- **Key exfiltration via env vars**: the real key is never in the process environment of the agent
- **Key exfiltration via Nix store**: config file in the Nix store contains only secret file paths, not values
- **SSRF / host redirect**: agents cannot redirect requests to arbitrary hosts by changing the URL — the upstream host is fixed from config at load time
- **Host header spoofing**: requests with an unknown `Host` header are rejected with 403 before any upstream connection is made

### What the proxy does NOT protect against

- **Compromised proxy process**: if the proxy process is compromised, the key in memory is accessible
- **Colocated processes on the same host**: any process on the host that can reach `127.0.0.1:<port>` can make API calls using the real key — OS-level process isolation is the trust boundary, not cryptographic identity
- **Sandbox escape**: if the agent escapes bubblewrap it can reach the loopback port and make requests
- **Per-caller attribution**: all agents sharing a proxy instance share one upstream key; individual agent activity cannot be attributed at the key level
- **Key rotation on compromise**: there is no mechanism to revoke access to a specific caller without restarting the proxy

### Trust boundary

The trust model is: "if a process can reach `127.0.0.1:<port>`, it is trusted." This is enforced by:
1. Loopback-only bind — the port is not reachable from the network
2. Bubblewrap sandbox — the agent process cannot reach loopback unless explicitly allowed

This is weaker than Netclode's per-caller ServiceAccount token validation but substantially simpler to deploy.

## Placeholder format and SDK compatibility

The Anthropic SDK validates that keys match the `sk-ant-api03-*` prefix before sending requests. A placeholder of `sk-placeholder-X` will be rejected client-side before the request reaches the proxy.

Use `sk-ant-api03-placeholder` (or any value matching `sk-ant-api03-*`) as the placeholder to pass SDK validation. This is a known limitation; see [known-issues.md](known-issues.md) for the full entry.

## Streaming and SSE

Both request and response bodies are fully streamed:
- Request body: `reqwest::Body::wrap_stream(body.into_data_stream())`
- Response body: `Body::from_stream(upstream_response.bytes_stream())`

No buffering occurs. Server-Sent Events (SSE) from the Anthropic streaming API pass through correctly.

## Multi-secret configuration

A single proxy instance can hold multiple `[[secret]]` blocks. The `Host` header determines which secret and upstream are used. This allows one proxy port to serve multiple API providers, or multiple credential tiers for a single provider.

## Options reference

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `services.secretProxy.services.<name>.port` | port | (required) | Loopback port |
| `services.secretProxy.services.<name>.placeholder` | string | `"sk-placeholder-<name>"` | Fake token for agent env |
| `services.secretProxy.services.<name>.baseUrlEnvVar` | string | `"ANTHROPIC_BASE_URL"` | Env var name in `bwrapArgs` |
| `services.secretProxy.services.<name>.secrets.<key>.headerName` | string | (required) | Header to inject real key into |
| `services.secretProxy.services.<name>.secrets.<key>.secretFile` | path | (required) | Runtime path to secret file |
| `services.secretProxy.services.<name>.secrets.<key>.allowedDomains` | list of string | (required) | Upstream domain allowlist |
| `services.secretProxy.package` | package | built from source | Override the binary |

`bwrapArgs` is a computed read-only attribute — do not set it manually.

## Wiring into bubblewrap

The module exposes `config.services.secretProxy.services.<name>.bwrapArgs`, a list of bwrap arguments that sets the base URL env var in the sandbox:

```nix
let
  proxyArgs = config.services.secretProxy.services.my-agent.bwrapArgs;
  # expands to: [ "--setenv" "ANTHROPIC_BASE_URL" "http://127.0.0.1:9091" ]
in
bwrap ${lib.escapeShellArgs proxyArgs} ...
```

The agent also needs the placeholder key. Inject it alongside `bwrapArgs`:

```nix
bwrap \
  ${lib.escapeShellArgs proxyArgs} \
  --setenv ANTHROPIC_API_KEY "sk-ant-api03-placeholder" \
  ...
```

The placeholder value must match the `placeholder` option exactly (case-sensitive). The proxy logs a warning if the incoming key does not match.

## Port conflicts

If two services declare the same port, the module emits a build-time assertion error:

```
services.secretProxy: duplicate ports detected: port 9091 used by [my-agent, other-agent]
```

## Systemd unit name

Each service creates a unit named `secret-proxy-<name>.service`. Check logs with:

```bash
journalctl -u secret-proxy-my-agent.service -f
```

## Access control

The proxy runs as a dedicated system user `secret-proxy-<name>` in group `secret-proxy`. The secret file must be readable by this user. For sops-nix, set:

```nix
sops.secrets."anthropic-api-key".owner = "secret-proxy-my-agent";
```

## Adding a new port to the firewall

The proxy binds to `127.0.0.1` — do not add proxy ports to the public firewall. If your NixOS config has an `internalOnlyPorts` assertion, add the proxy port there to make the intent explicit:

```nix
internalOnlyPorts = [ 9091 ];
```

# Deployment: Docker Compose

This guide covers deploying nix-secret-proxy in a Docker environment alongside a containerised agent.

## Limitation: loopback binding

The proxy binary binds to `127.0.0.1:<port>`. Docker containers cannot reach the host loopback by default. To use the proxy from a sibling container, you must either:

1. **Use `network_mode: host`** — both the proxy and agent share the host network stack (simplest)
2. **Run the proxy as a sidecar on a shared bridge network** and override bind address — requires a code change (not yet supported; tracked in [known-issues.md](known-issues.md))

The examples below use `network_mode: host`.

## Config file

Create `/etc/secret-proxy/config.toml`:

```toml
port = 9091
placeholder = "sk-ant-api03-placeholder"

[[secret]]
header = "x-api-key"
name = "anthropic"
file = "/run/secrets/anthropic-api-key"
allowed_domains = ["api.anthropic.com"]
```

## Docker Compose

```yaml
services:
  secret-proxy:
    image: ghcr.io/dangirsh/nix-secret-proxy:latest
    network_mode: host
    volumes:
      - ./config.toml:/etc/secret-proxy/config.toml:ro
      - /run/secrets/anthropic-api-key:/run/secrets/anthropic-api-key:ro
    command: ["--config", "/etc/secret-proxy/config.toml"]
    restart: unless-stopped

  my-agent:
    image: my-agent:latest
    network_mode: host
    environment:
      ANTHROPIC_BASE_URL: "http://127.0.0.1:9091"
      ANTHROPIC_API_KEY: "sk-ant-api03-placeholder"
    depends_on:
      - secret-proxy
```

## Secret file

The secret file at `/run/secrets/anthropic-api-key` must contain the raw API key (one line, no trailing newline required — the proxy trims whitespace). Mount it read-only into the proxy container. Do not mount it into the agent container.

## Docker Secrets (Swarm)

If using Docker Swarm secrets:

```yaml
services:
  secret-proxy:
    image: ghcr.io/dangirsh/nix-secret-proxy:latest
    network_mode: host
    secrets:
      - anthropic-api-key
    volumes:
      - ./config.toml:/etc/secret-proxy/config.toml:ro
    command: ["--config", "/etc/secret-proxy/config.toml"]

secrets:
  anthropic-api-key:
    external: true
```

Docker Swarm mounts secrets at `/run/secrets/<name>` by default, matching the config file path.

## Verification

After starting the stack, verify the proxy is running:

```bash
curl -s -o /dev/null -w "%{http_code}" \
  -H "Host: api.anthropic.com" \
  -H "x-api-key: sk-ant-api03-placeholder" \
  http://127.0.0.1:9091/v1/models
```

A 200 (or 401 from Anthropic if the key is invalid) confirms the proxy is forwarding requests. A 403 means the Host header was rejected by the allowlist.
```

# Deployment: Plain systemd

This guide covers deploying nix-secret-proxy on a non-NixOS Linux host using a plain systemd service unit.

## Prerequisites

- `secret-proxy` binary installed at `/usr/local/bin/secret-proxy` (or adjust `ExecStart`)
- A config file at `/etc/secret-proxy/config.toml`
- A secret file managed by your preferred secrets tool (Vault agent, SOPS, systemd credentials, etc.)

## Config file

```toml
# /etc/secret-proxy/config.toml
port = 9091
placeholder = "sk-ant-api03-placeholder"

[[secret]]
header = "x-api-key"
name = "anthropic"
file = "/run/credentials/secret-proxy.service/anthropic-api-key"
allowed_domains = ["api.anthropic.com"]
```

The `file` path above uses systemd credential injection (see below). Adjust to match your secrets delivery mechanism.

## Service unit

```ini
# /etc/systemd/system/secret-proxy.service
[Unit]
Description=Secret placeholder proxy
After=network.target

[Service]
Type=simple
User=secret-proxy
Group=secret-proxy
ExecStart=/usr/local/bin/secret-proxy --config /etc/secret-proxy/config.toml
Restart=on-failure
RestartSec=5s

# Systemd credential injection (systemd 247+)
# The key file is mounted at /run/credentials/secret-proxy.service/anthropic-api-key
LoadCredential=anthropic-api-key:/etc/secret-proxy/anthropic-api-key.encrypted
# Or with a plain file: LoadCredential=anthropic-api-key:/run/secrets/anthropic-api-key

# Hardening
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
PrivateTmp=true
MemoryDenyWriteExecute=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectKernelLogs=true
ProtectControlGroups=true
RestrictAddressFamilies=AF_INET AF_INET6
RestrictNamespaces=true
RestrictRealtime=true
RestrictSUIDSGID=true
CapabilityBoundingSet=
SystemCallArchitectures=native
LockPersonality=true
PrivateDevices=true

[Install]
WantedBy=multi-user.target
```

## Systemd credentials

`LoadCredential=` mounts the secret at `/run/credentials/<unit-name>/<name>` inside the service's private namespace. This avoids placing the raw secret in a world-readable location and integrates with systemd-creds for encrypted storage.

To encrypt a credential with the host's TPM or machine key:

```bash
systemd-creds encrypt --name=anthropic-api-key /path/to/key /etc/secret-proxy/anthropic-api-key.encrypted
```

Then reference it in the unit: `LoadCredentialEncrypted=anthropic-api-key:/etc/secret-proxy/anthropic-api-key.encrypted`

## User and group

```bash
useradd --system --no-create-home --shell /usr/sbin/nologin secret-proxy
```

## Enable and start

```bash
systemctl daemon-reload
systemctl enable --now secret-proxy.service
journalctl -u secret-proxy.service -f
```

## Wiring the agent

Set these environment variables in the agent's service unit or launch script:

```ini
[Service]
Environment=ANTHROPIC_BASE_URL=http://127.0.0.1:9091
Environment=ANTHROPIC_API_KEY=sk-ant-api03-placeholder
```

The agent process must be able to reach `127.0.0.1:9091`. If the agent runs in a network namespace or container, ensure loopback access or use `network_mode: host` (see [deployment-docker.md](deployment-docker.md)).

## Placeholder format

The Anthropic SDK validates that keys match `sk-ant-api03-*`. Use `sk-ant-api03-placeholder` (or another `sk-ant-api03-` prefixed value) as the placeholder to avoid client-side rejection before the request reaches the proxy.
