# Known Issues

This document catalogues known limitations, design constraints, and degraded-mode behaviours in nix-secret-proxy. Each entry is sourced from direct code inspection. Severities reflect actual impact on documented use cases.

Issues marked "Fix planned (Phase 72)" will be addressed in the next code phase. Issues marked "Accepted" are deliberate design tradeoffs documented here for transparency.

---

## BLOCKING

### BLOCK-01: Bind address hardcoded to 127.0.0.1

**Severity:** BLOCKING
**Status:** Fix planned (Phase 72)
**Affects:** Docker and container deployments

**Description:**
`src/main.rs` line 90 hardcodes the bind address:
```rust
let addr = SocketAddr::from(([127, 0, 0, 1], config.port));
```
Docker containers on a bridge network cannot reach the host loopback interface. The proxy is unreachable from sibling containers unless `network_mode: host` is used.

**Impact:** Any Docker Compose deployment where the proxy and agent are on different container network namespaces cannot use the proxy without `network_mode: host`. The NixOS deployment is unaffected (loopback is correct there).

**Current workaround:** Use `network_mode: host` for both the proxy and agent containers. See [deployment-docker.md](deployment-docker.md).

**Proposed mitigation (Phase 72):** Add an optional `bind` field to `src/config.rs` and the NixOS module option, defaulting to `127.0.0.1`. Docker configs can then set `bind = "0.0.0.0"` and rely on Docker network policy for isolation.

---

### BLOCK-02: Placeholder format rejected by Anthropic SDK

**Severity:** BLOCKING
**Status:** Document only
**Affects:** Any deployment using the Anthropic Python or Node SDK

**Description:**
The Anthropic SDK validates API keys client-side before sending requests. Keys must match the prefix `sk-ant-api03-`. The NixOS module's default placeholder is `sk-placeholder-<name>` (module.nix line 44):
```nix
default = "sk-placeholder-${name}";
```
This default fails SDK-side validation; the request never reaches the proxy.

**Impact:** Out-of-the-box NixOS module configuration silently fails when used with the Anthropic SDK. The agent process raises a validation error before making any HTTP request.

**Current workaround:** Explicitly set `placeholder = "sk-ant-api03-placeholder"` (or any `sk-ant-api03-` prefixed value) in the NixOS module configuration. This passes SDK validation; the proxy replaces the value before forwarding.

**Proposed mitigation:** Update the default in `module.nix` and documentation to use `sk-ant-api03-placeholder`. The module cannot enforce the correct prefix without coupling to Anthropic's format, so documentation is the primary mitigation.

---

## DEGRADED

### DEG-01: No upstream timeout

**Severity:** DEGRADED
**Status:** Fix planned (Phase 72)
**Affects:** All deployments

**Description:**
`src/main.rs` line 86 builds the reqwest client with no timeout:
```rust
client: reqwest::Client::new(),
```
A hung upstream connection (network partition, API outage, slow response) holds the connection open indefinitely. If the upstream never responds, the proxy handler never returns.

**Impact:** Under load, hung upstream requests accumulate open connections. This does not crash the proxy but degrades throughput and can exhaust OS connection limits under sustained API outages.

**Proposed mitigation (Phase 72):** Add a `timeout_secs` config field (default: 120). Build the reqwest client with `.timeout(Duration::from_secs(cfg.timeout_secs))`.

---

### DEG-02: Axum default body limit (2 MB) blocks large multimodal prompts

**Severity:** DEGRADED
**Status:** Fix planned (Phase 72)
**Affects:** Multimodal (image/document) API calls

**Description:**
Axum applies a 2 MB default body size limit to all request bodies. This limit is not overridden anywhere in `src/proxy.rs` or `src/main.rs`. Anthropic's vision and document API calls regularly exceed 2 MB (base64-encoded images, PDFs).

**Impact:** Multimodal API calls with large attachments fail with a 413 Payload Too Large before the proxy forwards anything to the upstream. Text-only calls (including streaming) are unaffected.

**Proposed mitigation (Phase 72):** Add `.layer(DefaultBodyLimit::disable())` (or a configurable limit) to the axum router in `proxy.rs`. A config field `max_body_bytes` (default: unlimited) would allow operators to set a cap.

---

### DEG-03: Plain-text 502 on upstream failure confuses SDK error handling

**Severity:** DEGRADED
**Status:** Fix planned (Phase 72)
**Affects:** All deployments

**Description:**
`src/proxy.rs` line 144 returns a plain-text body on upstream connection failure:
```rust
return simple_response(StatusCode::BAD_GATEWAY, "Upstream request failed\n");
```
The Anthropic SDK (and most API clients) expect error responses to be JSON with `{"type": "error", "error": {...}}`. A plain-text 502 causes unhandled exceptions or opaque error messages in SDK consumers.

**Impact:** Upstream failures surface as JSON decode errors in the SDK layer rather than clean API errors. Debugging is harder than necessary.

**Proposed mitigation (Phase 72):** Return a JSON body on 502:
```json
{"type": "error", "error": {"type": "api_error", "message": "Proxy upstream request failed"}}
```
with `Content-Type: application/json`.

---

### DEG-04: No graceful shutdown — in-flight SSE streams truncated on SIGTERM

**Severity:** DEGRADED
**Status:** Fix planned (Phase 72)
**Affects:** Streaming / SSE deployments

**Description:**
`src/main.rs` has no signal handling. On SIGTERM (e.g., systemd stop, rolling deploy), the process exits immediately. Active streaming responses (SSE, long-running tool calls) are truncated mid-stream with a connection reset, not a clean close frame.

**Impact:** Any agent awaiting a streaming response when the proxy is restarted receives a broken stream. The agent must handle the error and retry. Systemd's `Restart=on-failure` ensures the proxy restarts but cannot save in-flight streams.

**Proposed mitigation (Phase 72):** Register a SIGTERM handler (tokio `signal::ctrl_c` or `signal::unix::signal`) that stops accepting new connections and waits for active handlers to complete before exiting. Add a `shutdown_timeout_secs` config field (default: 30) to bound the drain period.

## INFORMATIONAL

### INFO-01: No health endpoint

**Severity:** INFORMATIONAL
**Status:** Accepted

**Description:**
The proxy has no `/healthz` or equivalent endpoint. Container orchestrators (Kubernetes, Docker Compose `healthcheck`) cannot probe liveness without a TCP check.

**Rationale:** Health endpoint adds complexity. A successful TCP connect to `127.0.0.1:<port>` is a sufficient liveness check for the current deployment targets (NixOS systemd, Docker with restart policy). A future phase may add one if Kubernetes becomes a target.

**Workaround:** Use a TCP health check:
```yaml
healthcheck:
  test: ["CMD-SHELL", "nc -z 127.0.0.1 9091 || exit 1"]
  interval: 10s
```

---

### INFO-02: Shared key — no per-caller rate limiting or attribution

**Severity:** INFORMATIONAL
**Status:** Accepted

**Description:**
All agents sharing a proxy instance share one upstream API key. There is no per-caller tracking, rate limiting, or attribution at the proxy layer. If one agent exhausts rate limits, all agents are affected.

**Rationale:** This is a deliberate design choice. The trust model is OS-level isolation, not cryptographic per-caller identity. Stanislas Polu's Netclode implementation (see [architecture.md](architecture.md)) adds per-caller ServiceAccount token validation and per-caller key pools for stronger isolation at higher setup cost. That approach is not adopted here.

**Workaround:** Deploy one proxy instance per agent if per-caller isolation is required. Each instance can use a different API key.

---

### INFO-03: No per-request audit trail

**Severity:** INFORMATIONAL
**Status:** Accepted

**Description:**
The proxy logs `ALLOW method=<M> host=<H> secret_name=<N> upstream=<U>` per request (proxy.rs line 125–128) but does not log request bodies, response bodies, token counts, or timing. There is no persistent audit log.

**Rationale:** Request body logging would require buffering (defeating streaming) and would log sensitive prompt content. OS process isolation (bubblewrap) is the primary audit boundary — which process made which request is visible at the OS level. Detailed content logging is out of scope.

---

### INFO-04: `authorization` header unconditionally stripped

**Severity:** INFORMATIONAL
**Status:** Accepted

**Description:**
`proxy.rs` line 20 strips `authorization` from all forwarded requests regardless of which secret is configured:
```rust
const REQUEST_SKIP_HEADERS: &[&str] = &[
    "x-api-key",
    "authorization",
    ...
];
```
This means a configured `header = "authorization"` (for OpenAI-style `Authorization: Bearer <key>`) is injected correctly, but any `Authorization` header sent by the agent is also stripped. The net result is correct for OpenAI API calls, but the unconditional strip could confuse operators expecting to pass through an `Authorization` header for a different purpose.

**Rationale:** Both `x-api-key` and `authorization` are stripped to prevent the agent from injecting a real key or overriding the proxy's substitution. This is intentional and correct. Documented here because it is a non-obvious invariant.

---

### INFO-05: HTTP_PROXY / HTTPS_PROXY honored by reqwest (undocumented)

**Severity:** INFORMATIONAL
**Status:** Document only

**Description:**
reqwest respects the `HTTP_PROXY`, `HTTPS_PROXY`, `NO_PROXY` environment variables at startup. If these are set in the proxy's systemd unit environment (e.g., inherited from a corporate proxy config), the proxy will chain through that upstream proxy before reaching the API. This is standard reqwest behavior but is not documented in the module or README.

**Impact:** In most deployments (no corporate proxy), this is a non-issue. In environments with a mandatory outbound proxy, this is actually useful behavior.

**Note:** The NixOS module's systemd hardening (`ProtectSystem=strict`) restricts filesystem access but does not prevent environment variable inheritance. If proxy chaining is undesired, set `Environment=NO_PROXY=*` in the systemd unit's `serviceConfig`.

---

### INFO-06: IPv6 untested

**Severity:** INFORMATIONAL
**Status:** Document only

**Description:**
The bind address is hardcoded to `127.0.0.1` (IPv4). IPv6 loopback (`::1`) is not tested or supported. The `allowed_domains` allowlist matches against the `Host` header value (domain name, not IP), so the matching logic itself is address-family agnostic. However, an IPv6-only host would not be able to run the proxy in its current form.

**Workaround:** None without a code change. IPv4 loopback is available on all supported deployment targets (Linux systemd, Docker, NixOS).
