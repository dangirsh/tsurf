# Plan 35-02: mautrix-whatsapp + mautrix-signal Bridges

## Goal

Add WhatsApp and Signal bridges to the existing Matrix hub. Both bridges use
their respective NixOS modules (`services.mautrix-whatsapp`,
`services.mautrix-signal`) and connect to the Conduit homeserver deployed in
Plan 35-01. After deployment, the user links their WhatsApp and Signal accounts
via interactive QR code / device linking flows.

## Decisions Made (from Research)

- **mautrix-whatsapp:** Go-based bridge, uses unofficial WA Web protocol.
  Full NixOS module in nixpkgs-25.11. Medium ban risk from WhatsApp (documented,
  accepted). No extra secrets needed -- WhatsApp linking is interactive (QR code).
- **mautrix-signal:** Go-based bridge with signal-cli backend.
  Full NixOS module in nixpkgs-25.11. Requires `MemoryDenyWriteExecute = false`
  systemd override (libsignal JIT requirement). Signal linking is interactive
  (device linking QR code or phone number verification).
- **No new sops secrets:** WhatsApp and Signal bridges do not need API keys or
  tokens provisioned in sops. Session state is auto-generated and stored in
  the bridge's state directory. The as_token/hs_token in registration.yaml are
  auto-generated at first start.
- **Port assignments:** Already added in Plan 35-01 (29318 for WhatsApp, 29328
  for Signal -- both in internalOnlyPorts).
- **Impermanence:** Both bridges need state directory persistence:
  `/var/lib/mautrix-whatsapp` and `/var/lib/mautrix-signal` (or under
  `/var/lib/private/` if DynamicUser).

## must_haves

1. `services.mautrix-whatsapp` enabled in `modules/matrix.nix` with correct homeserver settings
2. `services.mautrix-signal` enabled in `modules/matrix.nix` with correct homeserver settings
3. `systemd.services.mautrix-signal.serviceConfig.MemoryDenyWriteExecute = false` override present
4. `@decision` annotation MTX-03 (MemoryDenyWriteExecute) present in module header
5. `@decision` annotation MTX-04 (E2E encryption bridge break) present in module header
6. `@decision` annotation MTX-05 (WhatsApp ban risk) present in module header
7. Impermanence entries for mautrix-whatsapp and mautrix-signal state directories
8. `nix flake check` passes for both host configurations
9. After deploy: `systemctl status mautrix-whatsapp` shows active
10. After deploy: `systemctl status mautrix-signal` shows active
11. After manual linking: WhatsApp DMs appear as Matrix rooms
12. After manual linking: Signal DMs appear as Matrix rooms

## Risks and Mitigations

### WhatsApp account ban (MEDIUM)

mautrix-whatsapp uses the unofficial WA Web protocol. Meta may detect and
disconnect the linked device. This is documented as accepted risk MTX-05.
Mitigation: maintain independent WhatsApp chat export backups. Re-linking
is usually possible after disconnection.

### Signal device linking limitations (LOW)

Signal allows a limited number of linked devices (typically 5). If the limit
is reached, an existing device must be unlinked first. Signal may also require
re-linking after extended periods of inactivity. Mitigation: monitor
mautrix-signal service logs for disconnection events.

### mautrix-signal crash on startup (LOW)

If the `MemoryDenyWriteExecute = false` override is not applied correctly,
mautrix-signal will crash immediately. The error in journalctl will mention
SIGKILL or memory protection. Fix: verify the systemd override is in place
with `systemctl show mautrix-signal | grep MemoryDeny`.

## Notes for Implementer

- Both mautrix-whatsapp and mautrix-signal are Go bridges. They handle Conduit
  appservice registration automatically (unlike the Python-based mautrix-telegram
  which needs the manual curl workaround). However, the appservice still needs
  to be registered in Conduit's admin room.
- The NixOS modules for mautrix-whatsapp and mautrix-signal have similar option
  structures. If one works, the other should follow the same pattern.
- Check whether the NixOS modules create static users or use DynamicUser. This
  affects impermanence paths and sops template owners.
- The `serviceDependencies` option adds `After=` and `Requires=` dependencies
  in the systemd unit. This ensures bridges don't start before Conduit is ready.
  If `serviceDependencies` is not a recognized option, use the fallback pattern
  shown in the task code blocks (`systemd.services.<name>.after/requires`).
- QR code scanning cannot be automated. The user MUST be available with their
  phone to complete the WhatsApp and Signal linking steps.
