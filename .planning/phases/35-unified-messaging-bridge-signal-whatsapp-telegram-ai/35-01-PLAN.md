# Plan 35-01: Conduit Homeserver + mautrix-telegram Bridge

## Goal

Deploy a self-hosted Matrix homeserver (Conduit) with the mautrix-telegram bridge
on neurosys. Conduit provides the Matrix backbone; mautrix-telegram bridges
Telegram DMs into Matrix rooms. Federation is disabled (Tailscale-only private
hub). All config is declarative NixOS modules -- no Docker containers.

## Decisions Made (from Research)

- **Homeserver:** Conduit (Rust, ~32MB RAM baseline) via `services.matrix-conduit`.
  Chosen over Synapse for resource efficiency. Federation disabled; this is a
  private messaging hub, not a public Matrix server.
- **server_name:** `neurosys.local` -- internal-only, Tailscale MagicDNS.
  Permanent once set (cannot change without recreating all rooms/users).
- **Database:** RocksDB (Conduit default, recommended for production).
- **Registration:** Disabled by default. Uses `registration_token` for manual
  one-time user creation. Token stored in sops.
- **Bridge:** mautrix-telegram via `services.mautrix-telegram` NixOS module.
  Full module exists in nixpkgs-25.11 with `enable`, `settings`, `environmentFile`.
- **Appservice registration:** Manual one-time step. Bridge auto-generates
  `registration.yaml` at first start. Admin pastes YAML into Conduit admin room.
  Cannot be automated in NixOS config (requires authenticated Matrix session).
- **Telegram bot user:** Python-based bridge (mautrix-telegram) requires manual
  bot user registration via curl when using Conduit. One-time step.
- **Module location:** Single `modules/matrix.nix` (one file per concern).
  Will grow to include WhatsApp/Signal in Plan 35-02.
- **Secrets approach:** All Matrix secrets use explicit
  `sopsFile = ../secrets/neurosys.yaml` per-secret. This means OVH does NOT
  need placeholder entries in `secrets/ovh.yaml` for Matrix secrets -- because
  the Matrix bridge stack runs only on neurosys, and explicit `sopsFile` overrides
  `defaultSopsFile`, so these secrets are never looked up in `ovh.yaml` during
  OVH evaluation.
- **Impermanence:** Conduit state at `/var/lib/private/matrix-conduit` (DynamicUser
  service, already covered by `/var/lib/private` persistence). mautrix-telegram
  at `/var/lib/mautrix-telegram` (new persistence entry needed).

## must_haves

1. `modules/matrix.nix` exists with Conduit + mautrix-telegram configuration
2. `modules/default.nix` imports `./matrix.nix`
3. Ports 6167, 29317, 29318, 29328 all added to `internalOnlyPorts` in `networking.nix` (all 4 pre-added in this plan to avoid revisiting networking.nix in 35-02)
4. `secrets.nix` declares `telegram-api-id`, `telegram-api-hash`, `matrix-registration-token`
5. `secrets/neurosys.yaml` contains the 3 new secrets (placeholder values OK)
6. `/var/lib/mautrix-telegram` added to impermanence persistence directories
7. If Conduit does NOT use DynamicUser, `/var/lib/matrix-conduit` must be explicitly added to impermanence
8. `@decision` annotations MTX-01 through MTX-02 present in module header
9. `nix flake check` passes for both `nixosConfigurations.neurosys` and `nixosConfigurations.ovh`
10. After deploy: `systemctl status conduit` shows active service
11. After deploy: `systemctl status mautrix-telegram` shows active service
12. After manual steps: Telegram DMs appear as Matrix rooms accessible via Conduit

      # modules/matrix.nix
      # Matrix messaging hub — Conduit homeserver + mautrix bridges

      # @rationale: One module per concern. All Matrix services are tightly coupled

      #   Conduit binds 0.0.0.0 — consistent with project pattern (Spacebot,

      #   Not a new decision — standard convention for internal services.
      #
      { config, pkgs, ... }:
      let
        # Conduit server name — permanent, cannot be changed after creation.
        serverName = "neurosys.local";
      in {

        # --- Conduit Matrix homeserver ---
        services.matrix-conduit = {
          enable = true;
          settings.global = {
            server_name = serverName;
            address = "0.0.0.0";         # Bind all interfaces; Tailscale-only via nftables
            port = 6167;
            database_backend = "rocksdb";
            allow_registration = true;    # Gated by registration_token below
            allow_federation = false;
            allow_encryption = true;
            trusted_servers = [];
            registration_token = "";      # Overridden by environmentFile
          };
          # NOTE: Conduit's NixOS module does not have environmentFile.
          # registration_token must be set via settings or at runtime.
          # If the NixOS module lacks environmentFile support, set registration_token
          # directly: read from sops secret path at eval time is NOT possible
          # (sops paths are runtime). Alternatives:
          #   a) Use allow_registration = true temporarily, create users, then set false
          #   b) Set registration_token in settings (leaked to /nix/store — acceptable
          #      since it's one-time use and the server is Tailscale-only)
          # Check `services.matrix-conduit` module options before implementing.
        };

        # --- mautrix-telegram bridge ---
        services.mautrix-telegram = {
          enable = true;
          environmentFile = config.sops.templates."mautrix-telegram-env".path;
          settings = {
            homeserver = {
              address = "http://localhost:6167";
              domain = serverName;
            };
            appservice = {
              address = "http://localhost:29317";
              hostname = "127.0.0.1";  # Localhost-only; Conduit pushes from same host
              port = 29317;
              database = "sqlite:////var/lib/mautrix-telegram/mautrix-telegram.db";
            };
            bridge = {
              permissions = {
                "*" = "relaybot";
                "@admin:${serverName}" = "admin";
              };
            };
            # Telegram API credentials injected via environmentFile
            # (MAUTRIX_TELEGRAM_TELEGRAM_API_ID, MAUTRIX_TELEGRAM_TELEGRAM_API_HASH)
          };
          serviceDependencies = [ "conduit.service" ];
          # FALLBACK: If `serviceDependencies` is not a recognized module option,
          # remove the line above and use instead:
          #   systemd.services.mautrix-telegram = {
          #     after = [ "conduit.service" ];
          #     requires = [ "conduit.service" ];
          #   };
          # Note: the Conduit systemd service name is `conduit.service`
          # (not `matrix-conduit.service`).
        };

        # --- Sops template for mautrix-telegram environment ---
        sops.templates."mautrix-telegram-env" = {
          content = ''
            MAUTRIX_TELEGRAM_TELEGRAM_API_ID=${config.sops.placeholder."telegram-api-id"}
            MAUTRIX_TELEGRAM_TELEGRAM_API_HASH=${config.sops.placeholder."telegram-api-hash"}
          '';
          owner = "mautrix-telegram";
          mode = "0400";
        };
      }

## Risks and Mitigations

### Conduit registration_token handling (MEDIUM)

The `services.matrix-conduit` NixOS module may not support `environmentFile` for
injecting the registration token at runtime. If the token must be in
`settings.global`, it will be written to the Nix store (world-readable).
Mitigation: the token is one-time use (create users, then disable registration),
and the server is Tailscale-only. Document as accepted risk if needed.

### mautrix-telegram environment variable naming (LOW)

The exact environment variable names for overriding YAML config fields in
mautrix bridges follow a specific convention. If the names don't match, the
bridge won't get the Telegram API credentials. Mitigation: check the NixOS
module source or mautrix docs at implementation time. The `settings` approach
(direct config) can be used as fallback, with secrets injected via activation
script into the config file (same pattern as automaton.nix).

### Conduit + Python bridge compatibility (LOW)

Conduit may have edge cases with Python-based bridges (mautrix-telegram).
The manual bot registration workaround is well-documented upstream. If other
issues arise, the Conduit Matrix room and mautrix GitHub issues are the
reference points.
