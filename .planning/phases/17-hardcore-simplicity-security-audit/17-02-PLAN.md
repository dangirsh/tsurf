**SEC4 — Fix token leak in repos.nix:** The current clone command embeds the GitHub PAT directly in the URL: `git clone "https://${GH_TOKEN}@github.com/$repo.git"`. This leaks the token in:
- Process arguments (visible in `/proc`)
- Systemd journal logs
- The resulting `.git/config` remote URL

Replace the token-in-URL pattern with `GIT_ASKPASS`:

```nix
system.activationScripts.clone-repos = {
  deps = [ "users" ];
  text = ''
    repos=(
      "dangirsh/parts"
      "dangirsh/claw-swap"
      "dangirsh/global-agent-conf"
    )
    CLONE_DIR="/data/projects"
    GH_TOKEN="$(cat ${config.sops.secrets."github-pat".path} 2>/dev/null || true)"

    # Create a GIT_ASKPASS helper that provides the token without embedding it in URLs.
    # This prevents the token from appearing in process arguments, journal logs, or .git/config.
    ASKPASS_SCRIPT=$(mktemp)
    chmod 700 "$ASKPASS_SCRIPT"
    cat > "$ASKPASS_SCRIPT" <<'ASKEOF'
    #!/bin/sh
    echo "$GH_TOKEN"
    ASKEOF

    mkdir -p "$CLONE_DIR"

    for repo in "''${repos[@]}"; do
      name="$(basename "$repo")"
      target="$CLONE_DIR/$name"
      if [ ! -d "$target" ]; then
        echo "Cloning $repo to $target..."
        GH_TOKEN="$GH_TOKEN" GIT_ASKPASS="$ASKPASS_SCRIPT" \
          ${pkgs.git}/bin/git clone "https://github.com/$repo.git" "$target" \
          || echo "WARNING: Failed to clone $repo (will retry on next activation)"
        chown -R dangirsh:users "$target" 2>/dev/null || true
      fi
    done

    rm -f "$ASKPASS_SCRIPT"
  '';
};
```

Key changes:
- Clone URL is `https://github.com/$repo.git` (no token)
- Token provided via `GIT_ASKPASS` script (not in process args)
- `.git/config` will have clean URL (no embedded token)
- Cleanup: temp script deleted after use

Wait, `GIT_ASKPASS` approach for HTTPS needs the credential to be provided as username (token as username with blank password for GitHub PATs). Let me use a more reliable approach with `git -c credential.helper`:

Actually, the simplest correct approach for GitHub PATs is to use the `Authorization` header via `git -c http.extraHeader`. But that also shows in process args. Let me use the `GIT_ASKPASS` approach properly:

For GitHub HTTPS with a PAT, the PAT goes as the password (username can be anything non-empty, conventionally `x-access-token`). The `GIT_ASKPASS` script is called twice: once for "Username for..." and once for "Password for...".

Revised approach — use `git -c credential.helper` with a store file:

```bash
# Write token to a temporary credential store (not in process args, not in .git/config)
CRED_FILE=$(mktemp)
chmod 600 "$CRED_FILE"

for repo in "''${repos[@]}"; do
  name="$(basename "$repo")"
  target="$CLONE_DIR/$name"
  if [ ! -d "$target" ]; then
    echo "Cloning $repo to $target..."
    # Write credential in git credential store format
    printf 'https://x-access-token:%s@github.com\n' "$GH_TOKEN" > "$CRED_FILE"
    GIT_TERMINAL_PROMPT=0 ${pkgs.git}/bin/git \
      -c credential.helper="store --file=$CRED_FILE" \
      clone "https://github.com/$repo.git" "$target" \
      || echo "WARNING: Failed to clone $repo (will retry on next activation)"
    chown -R dangirsh:users "$target" 2>/dev/null || true
  fi
done

rm -f "$CRED_FILE"
```

This is cleaner: the credential helper reads from a temp file, the clone URL is clean (no token), the .git/config remote URL has no token, and the credential file is deleted after use.
