<task type="auto">
  <name>Task 2: Rewrite parts flake.nix with NixOS module and package outputs</name>
  <files>/data/projects/parts/flake.nix</files>
  <action>
Replace the parts `flake.nix` entirely. The current flake has:
- `inputs`: nixpkgs 24.11, agenix
- `outputs`: nixosConfigurations.parts-vps (full system config)

Rewrite to:
- `inputs`: nixpkgs 25.11 (matching agent-neurosys), sops-nix (replacing agenix)
- `outputs`: nixosModules.default (portable NixOS module), packages.x86_64-linux.{parts-agent,parts-tools} (standalone image builds for testing)
- Remove the nixConfig block (cachix substituter not needed for this pattern)
- Remove the nixosConfigurations output entirely (agent-neurosys owns system config)

The nixosModules.default should import `./nix/module.nix` passing `self` and `sops-nix` for secret path resolution and sops-nix integration.

The packages outputs should callPackage `./nix/parts-agent.nix` and `./nix/parts-tools.nix` with `{ src = self; }`. These files don't exist yet (created in Plan 02), but the flake.nix should reference them now — the flake will fail `nix flake check` until Plan 02 and 03 create the referenced files, which is expected.

Use the pattern from the research document's "Pattern 1: Cross-Flake Module Export" section. Key points:
- Do NOT set `inputs.nixpkgs.follows` in the parts flake itself (that's done from the consumer side in agent-neurosys)
- DO export both nixosModules and packages
- The module function signature should be: `{ self, sops-nix, ... }: { config, lib, pkgs, ... }: { ... }` (curried — outer args from flake, inner args from NixOS module system)

After writing flake.nix, run `nix flake lock` (in the parts repo) to generate/update flake.lock with sops-nix pinned. The lock will succeed even if nix/module.nix doesn't exist yet (flake.lock only resolves inputs, not evaluates outputs).
