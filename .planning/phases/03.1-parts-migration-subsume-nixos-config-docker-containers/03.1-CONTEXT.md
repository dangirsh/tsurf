# Phase 3.1: Parts Migration — Subsume NixOS Config + Docker Containers - Context

**Gathered:** 2026-02-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Agent-neurosys fully owns the server's NixOS system definition. Parts repo exports a NixOS module (via flake) declaring its own containers, networks, and secrets. Agent-neurosys imports parts as a flake input. Parts becomes self-describing but delegates system-level NixOS config to agent-neurosys.

**Simplified scope:** Parts is not live (no users, no data). This is greenfield integration, not production migration. No cutover window, no rollback plan, no canary system needed.

**Deferred:** Syncthing native service moves to Phase 6 (User Services).

</domain>

<decisions>
## Implementation Decisions

### Module architecture
- Parts repo adds a `flake.nix` with `nixosModules.default` output alongside existing app code
- Agent-neurosys adds parts as a flake input and imports `inputs.parts.nixosModules.default`
- Parts' NixOS module declares its own Docker containers, networks, and sops-nix secrets
- Agent-neurosys owns system-level config (boot, firewall, Docker engine, Tailscale, users)
- Parts owns app-level config (its containers, its networks, its secrets)

### Secrets format
- Parts switches from agenix to sops-nix (same toolchain as agent-neurosys)
- Same age host key (same server) — parts' `.sops.yaml` references the acfs host key
- Parts' NixOS module declares `sops.secrets` entries for its own secrets (Telegram bot token, API keys, OAuth creds)
- Encrypted secret files live in the parts repo

### Docker image builds
- Images built via Nix `dockerTools.buildImage` — fully reproducible, no external registry
- No GHCR, no Watchtower — images are built as part of `nixos-rebuild`
- Existing Dockerfiles in parts will be rewritten as Nix expressions

### Deploy trigger
- Manual `nixos-rebuild switch` from agent-neurosys (or its CI)
- Parts repo has no `nixos-rebuild` in its CI — it's consumed as a flake input
- After parts code changes: update flake input in agent-neurosys, rebuild

### Claude's Discretion
- Exact `dockerTools.buildImage` structure for parts containers
- How to structure parts' `flake.nix` outputs (just `nixosModules` or also `packages`)
- Whether parts module uses `virtualisation.oci-containers` or raw systemd units for containers
- Network creation approach (docker-network systemd units vs activation scripts)

</decisions>

<specifics>
## Specific Ideas

- Parts currently has 2 Docker containers: `parts-agent` and `parts-tools`
- Two Docker networks: `agent_net` (172.20.0.0/24, internal) and `tools_net` (172.21.0.0/24, external)
- 7+ secrets to migrate: tailscale-authkey, telegram-bot-token, anthropic-api-key, openai-api-key, google-oauth-client-id, google-oauth-client-secret, google-oauth-refresh-token
- Parts currently deploys via GitHub Actions with `nixos-rebuild switch --flake .#parts-vps --target-host root@parts-vps`
- Parts is on NixOS 24.11; agent-neurosys targets 25.11 — parts module must be compatible with 25.11

</specifics>
