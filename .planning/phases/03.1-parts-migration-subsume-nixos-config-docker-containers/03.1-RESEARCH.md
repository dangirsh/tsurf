## Summary

This phase makes the parts repo self-describing as a NixOS module while agent-neurosys remains the system owner. The parts repo currently has a full NixOS configuration (`flake.nix` with `nixosConfigurations.parts-vps`) that must be refactored: the system-level config (boot, firewall, Docker engine, Tailscale, SSH) is discarded (agent-neurosys owns those), and the app-level config (two containers, two networks, seven secrets) becomes a portable `nixosModules.default` output. The existing Dockerfiles for `parts-agent` (Node.js 22, TypeScript build, simple entrypoint) and `parts-tools` (Node.js 22 + Java 21 + signal-cli + git + ripgrep, multi-stage) must be rewritten as `dockerTools.buildLayeredImage` expressions. Secrets migrate from agenix (`.age` files with `secrets.nix` encryption rules) to sops-nix (YAML files with `.sops.yaml` creation rules), using the same host key since both repos deploy to the same server.

The key technical challenges are: (1) building npm workspace monorepo apps with `buildNpmPackage` since parts uses npm workspaces with a shared types package, (2) the tools container's complex multi-runtime requirements (Node.js + Java + signal-cli + git + ripgrep), and (3) ensuring sops-nix `sopsFile` paths resolve correctly when the module is evaluated from agent-neurosys but the encrypted files live in the parts flake input. All three have well-documented solutions in the Nix ecosystem.

**Primary recommendation:** Use `dockerTools.buildLayeredImage` (not `buildImage`) for better layer caching, `virtualisation.oci-containers` with `imageFile` for declarative container management, and per-secret `sopsFile` overrides pointing to `self + "/secrets/..."` paths so encrypted files resolve from the parts flake source.

## Standard Stack

### Core

| Component | Version/Source | Purpose | Why Standard |
|-----------|---------------|---------|--------------|
| NixOS flake `nixosModules` output | built into Nix flakes | Export NixOS module from parts | Standard pattern for cross-repo module composition; used by sops-nix, home-manager, disko, etc. |
| `dockerTools.buildLayeredImage` | nixpkgs 25.11 | Build Docker images reproducibly | Creates multi-layer images for better caching; preferred over `buildImage` for applications |
| `buildNpmPackage` | nixpkgs 25.11 | Package Node.js/TypeScript apps | Standard nixpkgs builder for npm projects; handles npm cache reproducibly via `npmDepsHash` |
| `virtualisation.oci-containers` | NixOS module | Run containers as systemd services | Declarative container management with systemd integration; supports `imageFile` for local images |
| `sops-nix` | Mic92/sops-nix (follows main) | Secrets management | Already in use by agent-neurosys; supports per-secret `sopsFile` overrides for cross-flake paths |

### Supporting

| Component | Version/Source | Purpose | When to Use |
|-----------|---------------|---------|-------------|
| `pkgs.signal-cli` | nixpkgs 25.11 | Signal messaging CLI | For parts-tools container; available in nixpkgs, avoids manual download |
| `pkgs.temurin-jre-bin-21` | nixpkgs 25.11 | Java 21 runtime | Required by signal-cli; Eclipse Temurin JRE matches current Dockerfile |
| `dockerTools.binSh` | nixpkgs helper | Provides /bin/sh in container | Needed if any container entrypoint uses shell scripts |
| `dockerTools.caCertificates` | nixpkgs helper | CA cert bundle | Required for HTTPS connections from containers |
| `dockerTools.fakeNss` | nixpkgs helper | /etc/passwd, /etc/group | Required for non-root user resolution in containers |
| `pkgs.buildEnv` | nixpkgs | Compose multiple packages into one root | Combines app + runtime deps into single derivation for `copyToRoot`/`contents` |

### Alternatives Considered (Claude's Discretion Recommendations)

| Decision Area | Recommendation | Rationale |
|--------------|----------------|-----------|
| `buildImage` vs `buildLayeredImage` | **Use `buildLayeredImage`** | Multi-layer output means shared base layers between parts-agent and parts-tools; better `docker load` caching across rebuilds. `buildImage` creates a single monolithic layer. |
| `oci-containers` vs raw systemd units | **Use `virtualisation.oci-containers`** | Parts already uses this pattern in `nixos/containers.nix`. The `imageFile` parameter integrates perfectly with `dockerTools.buildLayeredImage`. Raw systemd units would duplicate what oci-containers already provides. |
| Network creation: systemd oneshot vs activation scripts | **Use systemd oneshot services** | Parts already uses this pattern in `nixos/containers.nix`. Systemd services provide logging, ordering, and `systemctl` management. Activation scripts lack `$PATH` and are harder to debug. |
| Flake outputs: just `nixosModules` vs also `packages` | **Export both `nixosModules.default` and `packages.x86_64-linux.{parts-agent,parts-tools}`** | Exporting image packages enables standalone testing (`nix build .#parts-agent && docker load < result`) without needing the full NixOS config. The module references these packages internally. |

### Pattern 1: Cross-Flake Module Export

**What:** Parts exports a NixOS module that agent-neurosys imports as a flake input.
**When to use:** When one repo owns app-level config but another repo owns system-level config.

Parts `flake.nix`:
```nix
{
  description = "Parts - Personal Agent Runtime Systems";

  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-25.11";
    sops-nix = {
      url = "github:Mic92/sops-nix";
      inputs.nixpkgs.follows = "nixpkgs";
    };
  };

  outputs = { self, nixpkgs, sops-nix, ... }:
  let
    system = "x86_64-linux";
    pkgs = nixpkgs.legacyPackages.${system};
  in {
    # NixOS module for agent-neurosys to import
    nixosModules.default = import ./nix/module.nix { inherit self sops-nix; };

    # Standalone image packages for testing
    packages.${system} = {
      parts-agent = pkgs.callPackage ./nix/parts-agent.nix { src = self; };
      parts-tools = pkgs.callPackage ./nix/parts-tools.nix { src = self; };
    };
  };
}
```

Agent-neurosys `flake.nix` addition:
```nix
{
  inputs = {
    # ... existing inputs ...
    parts = {
      url = "github:dangirsh/parts";  # or git+ssh://...
      inputs.nixpkgs.follows = "nixpkgs";
      inputs.sops-nix.follows = "sops-nix";
    };
  };

  outputs = { self, nixpkgs, home-manager, sops-nix, disko, parts, ... } @ inputs: {
    nixosConfigurations.acfs = nixpkgs.lib.nixosSystem {
      system = "x86_64-linux";
      specialArgs = { inherit inputs; };
      modules = [
        # ... existing modules ...
        inputs.parts.nixosModules.default
      ];
    };
  };
}
```

### Pattern 2: dockerTools.buildLayeredImage with buildNpmPackage

**What:** Build a Node.js TypeScript app into a Docker image using pure Nix.
**When to use:** When replacing Dockerfiles with reproducible Nix builds.

```nix
# nix/parts-agent.nix
{ pkgs, src, ... }:

let
  parts-agent-app = pkgs.buildNpmPackage {
    pname = "parts-agent";
    version = "0.1.0";
    src = src;  # entire repo root (npm workspaces need root)

    npmDepsHash = "sha256-PLACEHOLDER";  # computed via prefetch-npm-deps
    npmWorkspace = "agent";

    # TypeScript build (npm run build in agent workspace)
    npmBuildScript = "build";

    # Custom install: copy dist + production node_modules
    installPhase = ''
      runHook preInstall

      mkdir -p $out/lib/parts-agent
      cp -r agent/dist $out/lib/parts-agent/dist
      cp -r agent/node_modules $out/lib/parts-agent/node_modules

      # Copy shared dist (rootDir: ".." puts shared types under dist/shared/)
      cp -r shared $out/lib/parts-agent/shared

      # Copy non-TS assets
      mkdir -p $out/lib/parts-agent/dist/agent/src/config
      cp agent/src/config/SOUL.md $out/lib/parts-agent/dist/agent/src/config/

      runHook postInstall
    '';
  };
in
pkgs.dockerTools.buildLayeredImage {
  name = "parts-agent";
  tag = "latest";

  contents = with pkgs.dockerTools; [
    caCertificates
    fakeNss
  ];

  config = {
    Cmd = [ "${pkgs.nodejs_22}/bin/node" "${parts-agent-app}/lib/parts-agent/dist/agent/src/index.js" ];
    Env = [
      "NODE_ENV=production"
      "GATEWAY_URL=http://parts-tools:8080"
      "AGENT_NAME=Jarvis"
      "THINKING_BUDGET=10000"
      "SESSION_DIR=/data/sessions"
      "SESSION_LOG=/data/session-log.jsonl"
    ];
    WorkingDir = "${parts-agent-app}/lib/parts-agent";
  };
}
```

### Pattern 3: oci-containers with imageFile and sops-nix Secrets

**What:** NixOS module that wires locally-built images into systemd-managed containers with decrypted secrets.
**When to use:** When containers need secrets injected as environment variables at activation time.

```nix
# nix/module.nix
{ self, sops-nix, ... }:

{ config, lib, pkgs, ... }:

{
  imports = [ sops-nix.nixosModules.sops ];

  # Secrets - encrypted file lives in parts repo, referenced via self
  sops.secrets."telegram-bot-token" = {
    sopsFile = self + "/secrets/parts.yaml";
    restartUnits = [ "docker-parts-tools.service" ];
  };
  sops.secrets."anthropic-api-key" = {
    sopsFile = self + "/secrets/parts.yaml";
    restartUnits = [ "docker-parts-tools.service" ];
  };
  # ... more secrets ...

  # Docker networks
  systemd.services."docker-network-agent_net" = {
    description = "Create agent_net Docker network (internal)";
    after = [ "docker.service" ];
    wantedBy = [ "multi-user.target" ];
    serviceConfig = {
      Type = "oneshot";
      RemainAfterExit = true;
    };
    script = ''
      ${pkgs.docker}/bin/docker network inspect agent_net >/dev/null 2>&1 || \
      ${pkgs.docker}/bin/docker network create agent_net \
        --driver bridge --internal --subnet 172.20.0.0/24
    '';
  };

  # Container with local image
  virtualisation.oci-containers.backend = "docker";
  virtualisation.oci-containers.containers.parts-tools = {
    image = "parts-tools:latest";
    imageFile = pkgs.callPackage (self + "/nix/parts-tools.nix") { src = self; };
    extraOptions = [ "--network=agent_net" "--network=tools_net" ];
    environment = {
      NODE_ENV = "production";
      AGENT_URL = "http://parts-agent:3000";
    };
    environmentFiles = [
      config.sops.templates."parts-tools-env".path
    ];
    volumes = [
      "/var/lib/parts/data:/app/data"
    ];
  };

  # sops template for environment file (alternative to environmentFiles)
  sops.templates."parts-tools-env" = {
    content = ''
      TELEGRAM_BOT_TOKEN=${config.sops.placeholder."telegram-bot-token"}
      ANTHROPIC_API_KEY=${config.sops.placeholder."anthropic-api-key"}
    '';
  };
}
```

### Pattern 4: sops-nix Secret Path Resolution in Cross-Flake Modules

**What:** Using `self + "/path"` to reference encrypted files from the flake source.
**When to use:** When a NixOS module exported by a flake needs to reference files within that flake's source tree.

```nix
# In parts' module.nix, `self` is passed in from flake.nix outputs
# self + "/secrets/parts.yaml" resolves to the parts flake source,
# NOT to agent-neurosys' source tree.

# CORRECT: reference via flake self
sops.secrets."my-secret".sopsFile = self + "/secrets/parts.yaml";

# WRONG: relative path -- resolves from agent-neurosys evaluation context
# sops.secrets."my-secret".sopsFile = ./secrets/parts.yaml;

# WRONG: defaultSopsFile in module -- sets global default, conflicts with agent-neurosys' own secrets
# sops.defaultSopsFile = self + "/secrets/parts.yaml";
```

**Critical:** Do NOT set `sops.defaultSopsFile` in the parts module. Agent-neurosys already sets `sops.defaultSopsFile = ../../secrets/acfs.yaml` in its own secrets module. Parts must use per-secret `sopsFile` overrides to avoid conflicts.

### Anti-Patterns to Avoid

- **Setting `sops.defaultSopsFile` in the parts module:** Conflicts with agent-neurosys' own default. Use per-secret `sopsFile` overrides.
- **Using relative paths (`./secrets/...`) in the module:** These resolve relative to the importing flake (agent-neurosys), not the exporting flake (parts). Always use `self + "/path"`.
- **Declaring `virtualisation.docker.enable = true` in parts module:** Docker engine is system-level config owned by agent-neurosys. Parts module should only declare containers.
- **Including system-level config in parts module:** No boot, firewall, Tailscale, SSH, users, or Docker engine config. Parts only owns its app-level concerns.
- **Using `buildImage` instead of `buildLayeredImage`:** `buildImage` creates a single layer; any change rebuilds the entire layer. `buildLayeredImage` shares layers between images and across rebuilds.
- **Using `runAsRoot` in dockerTools:** Requires KVM, which may not be available in all build environments. Use `fakeRootCommands` or `extraCommands` instead.
- **Importing sops-nix module in parts when agent-neurosys already imports it:** NixOS modules can only be imported once. Since agent-neurosys already imports `sops-nix.nixosModules.sops`, the parts module should NOT re-import it. Instead, parts' module should just declare `sops.secrets` entries and rely on the parent config having sops-nix available.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Docker image building | Shell scripts running `docker build` | `dockerTools.buildLayeredImage` | Reproducible, cached, no Docker daemon needed at build time |
| npm dependency management in Nix | Manual `fetchurl` for node_modules | `buildNpmPackage` with `npmDepsHash` | Handles npm cache correctly, supports workspaces |
| Secret encryption/decryption | Custom age encrypt/decrypt scripts | `sops-nix` with `.sops.yaml` creation rules | Handles activation, permissions, restart triggers |
| Docker network creation | Activation scripts with raw commands | systemd oneshot services | Proper ordering, logging, `systemctl` management |
| Container-to-secret wiring | Manual environment file generation | `sops.templates` with `sops.placeholder` | Atomic secret injection, restart on change |
| Container lifecycle management | Raw `docker run` in systemd | `virtualisation.oci-containers` | Auto-start, auto-restart, journal logging, systemd integration |

**Key insight:** The entire parts NixOS integration (images, containers, networks, secrets) is expressible as pure Nix declarations. No imperative scripts, no external registries, no manual steps. The only imperative action is `nixos-rebuild switch`.

## Common Pitfalls

### Pitfall 1: npmDepsHash Mismatch
**What goes wrong:** `buildNpmPackage` fails with hash mismatch when package-lock.json changes.
**Why it happens:** `npmDepsHash` is a fixed-output derivation hash of the npm cache. Any dependency change invalidates it.
**How to avoid:** After any `npm install` or `package-lock.json` change, regenerate the hash with `prefetch-npm-deps package-lock.json`. Consider a CI check that verifies the hash matches.
**Warning signs:** Build failure with "hash mismatch" error mentioning `npmDeps`.

### Pitfall 2: npm Workspace Root vs Workspace Member
**What goes wrong:** `buildNpmPackage` can't find workspace dependencies or shared types.
**Why it happens:** The `src` must be the monorepo root (where root `package.json` with `workspaces` field lives), but `npmWorkspace` must point to the specific workspace member.
**How to avoid:** Set `src = self` (entire parts repo) and `npmWorkspace = "agent"` or `npmWorkspace = "gateway"`. The build will run in the workspace context but with access to root `node_modules` and the `shared/` package.
**Warning signs:** "Cannot find module '../shared/types'" or npm install errors about missing workspace.

### Pitfall 3: Missing Runtime Dependencies in dockerTools Images
**What goes wrong:** Container starts but crashes with "ENOENT" or certificate errors.
**Why it happens:** `dockerTools.buildLayeredImage` creates a minimal image from scratch -- no `/tmp`, no CA certs, no `/etc/passwd`, no shell.
**How to avoid:** Include `caCertificates` (for HTTPS), `fakeNss` (for user resolution), and `pkgs.dockerTools.binSh` (if shell access needed for debugging). For the tools container, also ensure `git` and `ripgrep` are in contents.
**Warning signs:** "unable to get local issuer certificate", "getpwuid: no such user", "ENOENT: no such file or directory, '/tmp/...'".

### Pitfall 4: sops-nix Double Module Import
**What goes wrong:** NixOS evaluation fails with "option 'sops' already declared".
**Why it happens:** Both agent-neurosys and parts try to import `sops-nix.nixosModules.sops`.
**How to avoid:** Only agent-neurosys imports the sops-nix module. Parts' module just declares `sops.secrets` entries without importing the module. Use `inputs.sops-nix.follows = "sops-nix"` in agent-neurosys' parts input to ensure they share the same sops-nix instance.
**Warning signs:** Error during `nix flake check` about option declaration conflicts.

### Pitfall 5: imageFile Name/Tag Mismatch with oci-containers
**What goes wrong:** Container service tries to pull from Docker Hub instead of using local image.
**Why it happens:** The `image` attribute in oci-containers must exactly match the `name:tag` in the `imageFile` derivation.
**How to avoid:** If `dockerTools.buildLayeredImage` has `name = "parts-agent"` and `tag = "latest"`, then `virtualisation.oci-containers.containers.parts-agent.image` must be `"parts-agent:latest"`.
**Warning signs:** Container service logs show "pulling image" followed by registry errors.

### Pitfall 6: nixpkgs Version Mismatch Between Flakes
**What goes wrong:** Parts module evaluates against a different nixpkgs than agent-neurosys, causing subtle breakage.
**Why it happens:** Parts' flake has its own nixpkgs input. Without `follows`, two different nixpkgs versions coexist.
**How to avoid:** In agent-neurosys' flake: `inputs.parts.inputs.nixpkgs.follows = "nixpkgs"` and `inputs.parts.inputs.sops-nix.follows = "sops-nix"`. This ensures parts uses agent-neurosys' nixpkgs 25.11.
**Warning signs:** Warning about "follows" during `nix flake lock`, or unexpected package versions in built images.

### Pitfall 7: Secrets Path Resolution in Cross-Flake Modules
**What goes wrong:** sops-nix can't find the encrypted secrets file at evaluation time.
**Why it happens:** Relative paths (`./secrets/parts.yaml`) resolve relative to the importing flake (agent-neurosys), not the exporting flake (parts).
**How to avoid:** Use `self + "/secrets/parts.yaml"` where `self` is the parts flake's self reference, passed as a parameter to the module function.
**Warning signs:** "No such file or directory" during `nix flake check` or `nixos-rebuild build`.

### Pitfall 8: Docker Network Ordering
**What goes wrong:** Container fails to start because its network doesn't exist yet.
**Why it happens:** systemd starts container services before network-creation services.
**How to avoid:** Add explicit `after` and `requires` dependencies on network services in the container systemd units. Parts' current `containers.nix` already does this correctly -- preserve this pattern.
**Warning signs:** "network agent_net not found" in container service logs.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|-----------------|--------------|--------|
| agenix for secrets | sops-nix | Already used by agent-neurosys | More flexible; supports YAML, per-secret file overrides, templates |
| `dockerTools.buildImage` (single layer) | `dockerTools.buildLayeredImage` (multi-layer) | Available since nixpkgs 21.05+ | Better caching, smaller incremental updates |
| `docker build` from Dockerfile | `dockerTools.buildLayeredImage` from Nix | Standard for NixOS deployments | Reproducible, no Docker daemon needed at build time |
| Manual `docker load` + `docker run` | `oci-containers` with `imageFile` | NixOS module, stable for years | Declarative, systemd-managed, auto-restart |
| node2nix for npm packages | `buildNpmPackage` | nixpkgs 23.05+ | Simpler, no code generation step, npm cache-based |

**Deprecated/outdated:**
- `node2nix`: Replaced by `buildNpmPackage` for most use cases. Still works but requires regenerating Nix files on dependency changes.
- `dream2nix`: Experimental alternative that never reached stability for Node.js workspaces.
- `runAsRoot` in dockerTools: Still works but requires KVM; prefer `fakeRootCommands` or `extraCommands`.

## Open Questions

1. **Google OAuth credentials format in sops**
   - What we know: Currently stored as `.json.age` files (entire JSON files encrypted by agenix). sops-nix supports `format = "json"` and `format = "binary"` for non-YAML files.
   - What's unclear: Should these be stored as individual key-value pairs in the main `parts.yaml`, or as separate sops-encrypted JSON files? The current format has full JSON objects (client ID, client secret, tokens) in single files.
   - Recommendation: Store individual values (client-id, client-secret, refresh-token) as separate keys in `parts.yaml` rather than encrypting entire JSON files. This is simpler and avoids needing `format = "json"` complexity. The container can reconstruct any needed JSON at runtime.

2. **signal-cli package in nixpkgs 25.11**
   - What we know: `signal-cli` is available in nixpkgs. The current Dockerfile pins version 0.13.24.
   - What's unclear: Whether nixpkgs 25.11 has a compatible version, and whether it packages signal-cli with the correct JNI/glibc compatibility (the current Dockerfile uses Eclipse Temurin for glibc-based libsignal JNI compat).
   - Recommendation: Verify `pkgs.signal-cli` version and runtime compatibility. If incompatible, the nixpkgs signal-cli derivation can be overridden or a custom derivation can fetch the specific version.

3. **npmDepsHash for workspaces**
   - What we know: `prefetch-npm-deps` works on a single `package-lock.json`. Parts has a root `package-lock.json` covering both workspaces.
   - What's unclear: Whether `buildNpmPackage` with `npmWorkspace` correctly handles workspace-level dependencies when the hash is computed from the root lockfile.
   - Recommendation: Use the root `package-lock.json` for `npmDepsHash` computation. Set `src = self` and `npmWorkspace = "agent"` or `"gateway"`. Test with a simple build first.

4. **Tailscale authkey secret ownership**
   - What we know: Currently in parts' agenix secrets. But tailscale is system-level config owned by agent-neurosys.
   - What's unclear: Should the tailscale-authkey secret stay in parts' sops file or move to agent-neurosys' secrets?
   - Recommendation: Move tailscale-authkey to agent-neurosys' `secrets/acfs.yaml`. It's a system-level secret, not an app-level secret. Parts' module should not manage it.
