**Section 1: Overview**
- What this document covers: full VPS recovery from catastrophic loss
- Target recovery time: < 2 hours (30min deploy, 30min restore, 30min verify, 30min buffer)
- RPO: 24 hours (daily backups to B2)
- Recovery sources: git repo (NixOS config), Backblaze B2 (stateful data), manual (re-auth)

**Section 2: Prerequisites**
Before starting recovery, you need:
1. Local machine with Nix installed (for `nixos-anywhere`)
2. The `agent-neurosys` git repo cloned locally
3. SSH access to the new VPS (Contabo dashboard → rescue console or new VPS provisioning)
4. **Restic credentials** — needed BEFORE sops-nix works on the new server:
   - Restic repository password: decrypt locally with admin age key: `sops -d secrets/acfs.yaml | grep restic-password`
   - B2 application key ID and key: `sops -d secrets/acfs.yaml | grep b2-`
   - Admin age private key must be at `~/.config/sops/age/keys.txt` on local machine
5. Fresh Tailscale auth key (if `/var/lib/tailscale/` restore is stale): generate at https://login.tailscale.com/admin/settings/keys
6. New VPS IP address (may differ from old: 161.97.74.121)

**Section 3: What's Where**
Table with three columns:
| Data | Source | Notes |
|------|--------|-------|
| NixOS configuration | Git repo | Fully declarative, `nix flake check` validates |
| SSH host ed25519 key | B2 backup | Critical: sops-nix age key derivation depends on this |
| Docker bind mount data (claw-swap, parts) | B2 backup | PostgreSQL data, Caddy certs, parts sessions |
| Tailscale device state | B2 backup | Avoids re-auth if state is fresh |
| Home Assistant state | B2 backup | Database, integrations, automations |
| User home + Syncthing | B2 backup | dotfiles, Syncthing config/data |
| Code repos (/data/projects/) | B2 backup + git remotes | B2 has latest uncommitted work |
| sops-nix secrets | Git repo (encrypted) | Decrypt using SSH host key (from B2 restore) |
| Tailscale auth | Manual re-auth | Only if /var/lib/tailscale/ restore fails or is stale |
| Home Assistant device pairing | Manual re-setup | Only if /var/lib/hass/ restore fails |

**Section 4: Phase 1 — Deploy NixOS from Git (est. 30 min)**

Step 1.1: Extract SSH host key from B2 backup
```bash
# On local machine — set restic credentials
export AWS_ACCESS_KEY_ID="<from sops -d secrets/acfs.yaml>"
export AWS_SECRET_ACCESS_KEY="<from sops -d secrets/acfs.yaml>"
export RESTIC_REPOSITORY="s3:s3.eu-central-003.backblazeb2.com/SyncBkp"
export RESTIC_PASSWORD="<from sops -d secrets/acfs.yaml>"

# List snapshots, pick latest
restic snapshots

# Restore only the SSH host key to a temp dir
mkdir -p /tmp/host-keys/etc/ssh
restic restore latest --target /tmp/host-keys \
  --include /etc/ssh/ssh_host_ed25519_key \
  --include /etc/ssh/ssh_host_ed25519_key.pub
chmod 600 /tmp/host-keys/etc/ssh/ssh_host_ed25519_key
chmod 644 /tmp/host-keys/etc/ssh/ssh_host_ed25519_key.pub
```

Verify: `ls -la /tmp/host-keys/etc/ssh/ssh_host_ed25519_key` shows non-empty file with 600 permissions.

Step 1.2: Update NixOS config if IP changed
If the new VPS has a different IP than 161.97.74.121, update `hosts/acfs/hardware.nix` (networking.interfaces or equivalent static IP config).

Step 1.3: Deploy with nixos-anywhere
```bash
cd /path/to/agent-neurosys
nixos-anywhere --extra-files /tmp/host-keys \
  --flake .#acfs root@<new-vps-ip>
```

Verify:
- SSH works: `ssh root@<new-vps-ip> hostname` returns "acfs"
- sops secrets decrypt: `ssh root@<new-vps-ip> ls /run/secrets/` shows 15+ secrets
- Clean up local: `rm -rf /tmp/host-keys`

**Section 5: Phase 2 — Restore Stateful Data from B2 (est. 30 min)**

Step 2.1: Set restic credentials on the VPS
```bash
ssh root@<new-vps-ip>
# sops-nix should have decrypted these — check:
cat /run/secrets/restic-password
ls /run/secrets/rendered/restic-b2-env
# If available, source them:
source /run/secrets/rendered/restic-b2-env
export RESTIC_PASSWORD=$(cat /run/secrets/restic-password)
export RESTIC_REPOSITORY="s3:s3.eu-central-003.backblazeb2.com/SyncBkp"
```

Step 2.2: Verify repository integrity
```bash
restic check
restic snapshots
```

Step 2.3: Stop services before restore
```bash
systemctl stop docker.service
systemctl stop home-assistant.service
systemctl stop tailscaled.service
```

Step 2.4: Restore stateful data
```bash
# Restore Docker bind mount data
restic restore latest --target / \
  --include /var/lib/claw-swap \
  --include /var/lib/parts

# Restore Tailscale state
restic restore latest --target / \
  --include /var/lib/tailscale

# Restore Home Assistant
restic restore latest --target / \
  --include /var/lib/hass

# Restore user data and projects
restic restore latest --target / \
  --include /data/projects \
  --include /home/dangirsh
```

Step 2.5: Fix ownership (restic restores may set wrong uid/gid)
```bash
chown -R dangirsh:users /home/dangirsh/
chown -R dangirsh:users /data/projects/
# Docker data should be root-owned (container bind mounts)
```

Step 2.6: Start services
```bash
systemctl start docker.service
systemctl start home-assistant.service
systemctl start tailscaled.service
```

Verify:
- `docker ps` shows claw-swap and parts containers running
- `systemctl status home-assistant` shows active
- `ls /data/projects/agent-neurosys/flake.nix` exists
- `ls /home/dangirsh/.config/syncthing/config.xml` exists (if Syncthing was configured)

**Section 6: Phase 3 — Manual Re-Authentication (est. 15 min)**

Step 3.1: Tailscale
If `/var/lib/tailscale/` restore was fresh enough, Tailscale may reconnect automatically. Check:
```bash
tailscale status
```
If it shows "NeedsLogin" or is not connected:
1. Generate a fresh auth key at https://login.tailscale.com/admin/settings/keys
2. Update the Tailscale auth key in sops:
   ```bash
   # On local machine:
   sops secrets/acfs.yaml  # Edit tailscale-authkey value
   git add secrets/acfs.yaml && git commit -m "fix: update Tailscale auth key for recovery"
   scripts/deploy.sh --target root@<vps-ip>
   ```
3. Or manually authenticate: `ssh root@<vps-ip> tailscale up --authkey=<new-key>`

Step 3.2: Home Assistant
If `/var/lib/hass/` restored successfully, HA should work as-is. Check:
```bash
curl -s http://localhost:8123/api/ | head
```
If it returns the HA API, you're good. If HA is freshly initialized (onboarding screen), the restore failed -- device pairing must be redone manually in the HA web UI.
