# Stack Research

**Domain:** NixOS declarative server configuration (VPS)
**Researched:** 2026-02-13
**Confidence:** HIGH

## Recommended Stack

### Core System

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| NixOS | 25.11 "Xantusia" | Base operating system | Current stable release (Nov 30, 2025). Supported until June 30, 2026. Ships Linux 6.17 kernel, LLVM 21, GCC 14. 7,002 new packages added over prior release. Use `nixos-25.11` channel for stability. |
| Nix (package manager) | 2.33.x | Package manager / build tool | Latest active release line (2.33.2 as of Feb 2, 2026). Ships with NixOS 25.11. Flakes support is stable enough for production use despite "experimental" label -- the entire ecosystem depends on it. |
| Flakes | (built into Nix) | Reproducible configuration | De facto standard for NixOS configuration. Provides lock files, hermetic builds, and composable inputs. Every serious NixOS configuration in 2025/2026 uses flakes. Enable with `nix.settings.experimental-features = [ "nix-command" "flakes" ];` |

### Flake Inputs

| Input | URL | Purpose | Why This Source |
|-------|-----|---------|-----------------|
| nixpkgs | `github:NixOS/nixpkgs/nixos-25.11` | Package repository | Pin to stable channel for server reliability. `nixos-25.11` receives security patches until June 2026. Do NOT use `nixpkgs-unstable` for a production server -- breakage risk is real. |
| home-manager | `github:nix-community/home-manager/release-25.11` | User environment management | Use the `release-25.11` branch that matches the nixpkgs channel. Master branch tracks unstable and will break with stable nixpkgs. Add `inputs.nixpkgs.follows = "nixpkgs";` to deduplicate. |
| sops-nix | `github:Mic92/sops-nix` | Secrets management | The standard for NixOS secrets. Decrypts at activation time, stores secrets in `/run/secrets` (tmpfs). Uses age encryption derived from SSH host keys -- no separate key management needed. Add `inputs.nixpkgs.follows = "nixpkgs";` |
| disko | `github:nix-community/disko` | Declarative disk partitioning | v1.13.0 (Jan 2026). Integrates directly with nixos-anywhere. Declares partition layout as Nix code, enabling reproducible disk setup. Use the `single-disk-ext4` template for VPS simplicity. Add `inputs.nixpkgs.follows = "nixpkgs";` |
| nixos-anywhere | (CLI tool, not a flake input) | Remote NixOS installation | v1.13.0 (Nov 2025). Run directly via `nix run github:nix-community/nixos-anywhere`. Does NOT need to be a flake input -- it consumes your flake as an argument. Handles kexec boot, disko partitioning, and NixOS install in one command. |

### Secrets Management

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| sops-nix | latest (follows main) | NixOS secret provisioning | Atomic secret provisioning. Secrets encrypted in repo, decrypted at activation time to `/run/secrets`. Supports per-secret ownership/permissions. Integrates with both NixOS modules and home-manager. |
| age | 1.x (from nixpkgs) | Encryption backend for sops | Simpler than GPG. Derive age keys from existing SSH ed25519 keys via `ssh-to-age`. Host keys auto-generated by sshd provide machine identity. No separate key distribution needed. |
| sops | 3.x (from nixpkgs) | Secret file editing | CLI tool for editing encrypted YAML/JSON files. Supports YAML (recommended), JSON, dotenv, INI formats. Diffs are readable in cleartext. |

### Container Runtime

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| Docker (via NixOS) | from nixpkgs 25.11 | Container runtime | Use `virtualisation.docker.enable = true;` for Docker daemon. The project already uses Docker Compose patterns (Caddy, app, PostgreSQL, MCP server). Staying with Docker avoids migration complexity. |
| virtualisation.oci-containers | (NixOS module) | Declarative container management | Manages Docker containers as systemd services. Declare images, ports, volumes, environment in Nix. Containers auto-start, auto-restart, and integrate with systemd journal logging. Set `backend = "docker";` explicitly. |

### Services (NixOS Native)

| Service | NixOS Option | Purpose | Why Native Over Docker |
|---------|-------------|---------|----------------------|
| Tailscale | `services.tailscale` | VPN / mesh networking | First-class NixOS module. `enable = true; useRoutingFeatures = "server";`. Needs firewall config: `networking.firewall.trustedInterfaces = [ "tailscale0" ];`. Use `services.tailscale.authKeyFile` with sops-nix for automated auth. |
| Syncthing | `services.syncthing` | File synchronization | Full declarative config via `services.syncthing.settings`. Define devices and folders in Nix. Set `key` and `cert` options for stable node ID across rebuilds (encrypt with sops-nix). |
| Restic | `services.restic.backups` | Backup to Backblaze B2 | NixOS module creates systemd timers. Use S3-compatible backend (NOT native B2 backend -- restic's B2 connector is unreliable). Configure with `s3:s3.us-west-000.backblazeb2.com/<bucket>`. Wrap credentials in sops-nix secrets. |
| Ollama | `services.ollama` | Local LLM inference | Native NixOS service module. `enable = true;` is sufficient for CPU-only on AMD EPYC. No GPU drivers needed for this VPS (no GPU). Set `services.ollama.host = "127.0.0.1";` to bind locally, expose via Tailscale only. |
| OpenSSH | `services.openssh` | Remote access | Always enabled. Harden: `PasswordAuthentication = false; PermitRootLogin = "prohibit-password";`. The ed25519 host key auto-generates and doubles as the sops-nix age key source. |

### Reverse Proxy

| Technology | Approach | Purpose | Why |
|------------|----------|---------|-----|

Configure via Caddyfile volume mount. Alternative: NixOS native `services.caddy` module offers tighter integration but requires migrating existing Caddyfile to Nix expressions. |

### Database

| Technology | Approach | Purpose | Why |

Use sops-nix for password injection via environment file. Alternative: NixOS native `services.postgresql` is more integrated but requires data migration planning. |

## Reference flake.nix Structure

```nix
{
  description = "NixOS server configuration for Contabo VPS";

  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-25.11";

    home-manager = {
      url = "github:nix-community/home-manager/release-25.11";
      inputs.nixpkgs.follows = "nixpkgs";
    };

    sops-nix = {
      url = "github:Mic92/sops-nix";
      inputs.nixpkgs.follows = "nixpkgs";
    };

    disko = {
      url = "github:nix-community/disko";
      inputs.nixpkgs.follows = "nixpkgs";
    };
  };

  outputs = { self, nixpkgs, home-manager, sops-nix, disko, ... }: {
    nixosConfigurations.neurosys = nixpkgs.lib.nixosSystem {
      system = "x86_64-linux";
      modules = [
        disko.nixosModules.disko
        sops-nix.nixosModules.sops
        home-manager.nixosModules.home-manager
        ./hosts/neurosys/configuration.nix
        ./hosts/neurosys/disko-config.nix
        {
          home-manager.useGlobalPkgs = true;
          home-manager.useUserPackages = true;
          home-manager.users.ubuntu = import ./home/ubuntu.nix;
          home-manager.sharedModules = [
            sops-nix.homeManagerModules.sops
          ];
        }
      ];
    };
  };
}
```

## Deployment Commands

```bash
# Initial installation (from local machine with Nix):
nix run github:nix-community/nixos-anywhere -- \
  --flake .#neurosys \
  --target-host root@<contabo-ip>

# Subsequent updates (from local machine):
nixos-rebuild switch --flake .#neurosys \
  --target-host root@<contabo-ip> \
  --use-remote-sudo

# Or from the server itself:
sudo nixos-rebuild switch --flake /etc/nixos#neurosys

# Update flake inputs:
nix flake update

# Update single input:
nix flake update home-manager
```

## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| Secrets | sops-nix (age) | agenix | Both are solid. sops-nix has better docs, supports YAML editing with sops CLI, and the 2025 Michael Stapelberg guide is definitive. agenix is simpler but less flexible. |
| Secrets encryption | age (via SSH keys) | GPG | GPG is complex to manage. age keys derived from SSH ed25519 keys mean zero extra key management. |
| Disk partitioning | disko | Manual partitioning | disko is required for nixos-anywhere. Manual partitioning is not reproducible. |
| Container runtime | Docker | Podman | Project already uses Docker. Podman is the NixOS default for oci-containers but Docker has better compatibility with existing compose files and images. |
| Rust toolchain | rustup | fenix / rust-overlay | fenix and rust-overlay are better for building Nix packages from Rust source. For a developer workstation where `rustup` manages toolchains imperatively, rustup is simpler and matches the project requirement. |
| Channel | nixos-25.11 (stable) | nixpkgs-unstable | Server stability matters. Unstable gets newer packages but risks breakage. Pin overlays for specific packages that need newer versions. |
| Home-manager mode | NixOS module | Standalone | Module mode means `nixos-rebuild switch` applies both system and user config atomically. No separate `home-manager switch` command. Single deployment path. |
| Reverse proxy | Caddy (Docker) | Caddy (NixOS native) | Native `services.caddy` is cleaner long-term but requires rewriting Caddyfile as Nix expressions. Docker preserves existing config. Can migrate later. |
| Database | PostgreSQL (Docker) | PostgreSQL (NixOS native) | Native `services.postgresql` is better integrated but Docker preserves existing data patterns. Evaluate migration after initial NixOS setup is stable. |
| Backup target | Restic via S3 API to B2 | Restic via native B2 API | Restic maintainers recommend S3 over native B2 due to reliability issues with the B2 connector. B2 is fully S3-compatible. |
| Installation method | nixos-anywhere | nixos-infect | nixos-anywhere is the modern approach: kexec + disko + install in one command. nixos-infect is older, less reliable, and doesn't support declarative disk layout. |

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| nixos-infect | Legacy tool. Modifies existing OS in-place, fragile on many providers, no declarative disk management. | nixos-anywhere with disko |
| NixOS channels (non-flake) | Legacy package management. No lock files, no reproducibility guarantees, no composability. | Flakes with `flake.lock` |
| GPG for sops | Complex key management, key distribution headaches, expired keys break decryption. | age encryption with SSH-derived keys |
| nixpkgs-unstable for server | Package breakage, no security patch guarantees, evaluation failures. | nixos-25.11 stable channel |
| nixpkgs-mozilla (Rust overlay) | Deprecated and unmaintained. Replaced by fenix and rust-overlay years ago. | rustup (for dev) or fenix (for Nix builds) |
| Imperative Nix commands | `nix-env -i` installs are not declarative, not reproducible, and conflict with NixOS config. | Declare packages in configuration.nix or home.nix |
| Password-protected age keys | sops-nix decrypts at boot/activation time. Password-protected keys require interactive input, breaking automated activation. | Unprotected age key derived from SSH host key |
| Manual partitioning on VPS | Not reproducible. If the VPS disk fails or you migrate, you cannot recreate the exact layout. | disko declarative partitioning |
| NixOS containers (systemd-nspawn) | Only supports rootful containers. Less ecosystem support than Docker. Fewer pre-built images. | Docker via `virtualisation.docker` + oci-containers |

## Git Integration Notes

- **All files must be git-tracked**: Nix flakes only see files that git tracks. Run `git add .` before builds. Files in `.gitignore` are invisible to Nix.
- **Secrets are safe to commit**: sops-encrypted YAML files contain only ciphertext. Commit `secrets/secrets.yaml` directly.
- **flake.lock is committed**: This is the reproducibility guarantee. Always commit lock file changes.
- **Do NOT commit**: Age private keys, SSH private keys, `.sops.yaml` key files. The `.sops.yaml` config file itself (which maps keys to files) IS committed.
