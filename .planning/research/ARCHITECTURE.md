# Architecture Research: NixOS Flake-Based Server Configuration

**Domain:** NixOS server configuration (single-host VPS, flake-based)
**Researched:** 2026-02-13
**Confidence:** HIGH (multiple official sources, well-established community patterns)

## Standard Architecture

### System Overview

```
┌─────────────────────────────────────────────────────────────────────┐
│                          flake.nix                                  │
│  (Entry point: inputs, nixosConfigurations, specialArgs)            │
```

### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| `flake.nix` | Entry point: declares inputs (nixpkgs, home-manager, sops-nix, disko), wires nixosConfigurations | Single file at repo root; uses `specialArgs` to pass inputs to modules |
| `flake.lock` | Pins exact versions of all inputs | Auto-generated by `nix flake update`; committed to git |
| Host config (`hosts/acfs/`) | Machine-specific settings: hardware, disk layout, networking, boot | `default.nix` + `hardware-configuration.nix` + `disko-config.nix` |
| System modules (`modules/`) | Reusable, composable NixOS configuration units | One `.nix` file per concern (networking, docker, services, shell, dev-tools) |
| Secrets (`secrets/`) | Encrypted secret files (YAML), `.sops.yaml` rules | sops-encrypted YAML, age keys derived from SSH host key |
| Home config (`home/`) | User-level packages, dotfiles, shell, user systemd services | Imported via `home-manager.users.<name>` from host config |
| Overlays (`overlays/`) | Package customizations and additions | `default.nix` exporting overlay list |

## Recommended Project Structure

```
.
├── flake.nix                    # Entry point: inputs + nixosConfigurations
├── flake.lock                   # Pinned dependency versions (auto-generated)
├── .sops.yaml                   # sops encryption rules (age keys, path regexes)
│
├── hosts/
│   └── acfs/                    # Host: Contabo VPS
│       ├── default.nix          # Host entry: imports hardware, disko, modules
│       ├── hardware-configuration.nix  # Generated by nixos-generate-config
│       └── disko-config.nix     # Disk layout: GPT, ESP, ext4 root, swap
│
├── modules/                     # Reusable NixOS modules (system-level)
│   ├── base.nix                 # Nix settings, garbage collection, flake config
│   ├── boot.nix                 # Bootloader (GRUB), kernel settings
│   ├── networking.nix           # Firewall, static IP, SSH hardening
│   ├── users.nix                # User accounts, groups, sudo
│   ├── secrets.nix              # sops-nix configuration, secret declarations
│   ├── docker.nix               # Docker daemon + OCI container declarations
│   ├── services.nix             # System-level services (fail2ban, etc.)
│   └── default.nix              # Imports all modules in this directory
│
├── home/                        # Home-manager configuration (user-level)
│   ├── default.nix              # User entry point, imports sub-modules
│   ├── shell.nix                # Zsh/Bash, starship, aliases, environment
│   ├── git.nix                  # Git configuration
│   ├── dev-tools.nix            # CLI tools (ripgrep, fd, jq, etc.)
│   ├── services/                # User systemd services
│   │   ├── syncthing.nix        # Syncthing file sync
│   │   └── cass-indexer.nix     # CASS indexer service
│   └── programs/                # Per-program config
│       └── neovim.nix           # Editor configuration
│
├── secrets/                     # Encrypted secret files
│   └── acfs.yaml                # Per-host secrets (encrypted with sops)
│
├── overlays/                    # Package overlays (if needed)
│   └── default.nix              # Overlay definitions
│
└── lib/                         # Helper functions (if needed)
    └── default.nix              # Shared Nix utility functions
```

### Structure Rationale

- **`hosts/` per-machine:** Each host gets its own directory. Adding a second server later means adding `hosts/newhost/` with its own `default.nix`, `hardware-configuration.nix`, and `disko-config.nix`. The flake.nix gains one more `nixosConfigurations` entry. No restructuring needed.
- **`modules/` by concern, not by host:** Modules are host-agnostic. A module like `docker.nix` declares OCI containers; the host's `default.nix` imports it. This makes modules reusable across hosts and keeps each file focused on one concern.
- **`home/` separate from `modules/`:** System-level NixOS config (modules/) and user-level home-manager config (home/) are distinct domains. Mixing them creates confusion about what runs as root vs. user. Keeping them separate aligns with the NixOS/home-manager boundary.
- **`secrets/` at repo root:** Encrypted secrets are committed to git. The `.sops.yaml` at repo root defines which age keys can decrypt which files. Per-host secret files (e.g., `secrets/acfs.yaml`) keep secrets compartmentalized.
- **`modules/default.nix` aggregator:** A single file that imports all modules in the directory. Host configs import `../modules` instead of listing individual files. Adding a new module means adding the file and one import line in `default.nix`.

## Architectural Patterns

### Pattern 1: Flake Input Wiring with specialArgs

**What:** Pass flake inputs (sops-nix, home-manager, etc.) through to all modules via `specialArgs`, avoiding the need to thread them manually.
**When to use:** Always in a flake-based NixOS config. This is the standard mechanism.
**Trade-offs:** Clean access to inputs everywhere, but can make module dependencies implicit.

**Example:**
```nix
# flake.nix
{
  inputs = {
    nixpkgs.url = "github:nixos/nixpkgs/nixos-unstable";
    home-manager = {
      url = "github:nix-community/home-manager";
      inputs.nixpkgs.follows = "nixpkgs";
    };
    sops-nix = {
      url = "github:Mic92/sops-nix";
      inputs.nixpkgs.follows = "nixpkgs";
    };
    disko = {
      url = "github:nix-community/disko";
      inputs.nixpkgs.follows = "nixpkgs";
    };
  };

  outputs = { self, nixpkgs, home-manager, sops-nix, disko, ... } @ inputs: {
    nixosConfigurations.acfs = nixpkgs.lib.nixosSystem {
      system = "x86_64-linux";
      specialArgs = { inherit inputs; };
      modules = [
        ./hosts/acfs
        sops-nix.nixosModules.sops
        disko.nixosModules.disko
        home-manager.nixosModules.home-manager
        {
          home-manager.useGlobalPkgs = true;
          home-manager.useUserPackages = true;
          home-manager.extraSpecialArgs = { inherit inputs; };
          home-manager.users.admin = import ./home;
        }
      ];
    };
  };
}
```

### Pattern 2: Module-Per-Concern Decomposition

**What:** Each `.nix` file in `modules/` owns exactly one concern (networking, docker, secrets). The module declares NixOS options for that concern only.
**When to use:** Always. Even for a single host, this prevents configuration files from growing into unmanageable monoliths.
**Trade-offs:** More files to navigate, but each file is self-contained and testable. The `default.nix` aggregator keeps imports simple.

**Example:**
```nix
# modules/docker.nix
{ pkgs, config, ... }: {
  # Enable Docker daemon
  virtualisation.docker.enable = true;

  # Declarative OCI containers (replaces docker-compose)
  virtualisation.oci-containers = {
    backend = "docker";
    containers = {
      watchtower = {
        autoStart = true;
        image = "containrrr/watchtower:latest";
        volumes = [ "/var/run/docker.sock:/var/run/docker.sock" ];
        environment = {
          WATCHTOWER_CLEANUP = "true";
          WATCHTOWER_POLL_INTERVAL = "86400";
        };
      };
      # More containers declared here
    };
  };

  # Docker users
  users.users.admin.extraGroups = [ "docker" ];
}
```

```nix
# modules/default.nix
{
  imports = [
    ./base.nix
    ./boot.nix
    ./networking.nix
    ./users.nix
    ./secrets.nix
    ./docker.nix
    ./services.nix
  ];
}
```

### Pattern 3: Host as Composition Root

**What:** The host's `default.nix` is purely a composition point. It imports hardware config, disko config, and the shared modules directory. Host-specific overrides (hostname, IP, timezone) live here; everything else is in modules.
**When to use:** Always. This keeps host files thin and modules reusable.
**Trade-offs:** Host files become very small (often under 30 lines), which is the goal.

**Example:**
```nix
# hosts/acfs/default.nix
{ config, pkgs, inputs, ... }: {
  imports = [
    ./hardware-configuration.nix
    ./disko-config.nix
    ../../modules
  ];

  # Host-specific settings only
  networking.hostName = "acfs";
  time.timeZone = "America/New_York";
  system.stateVersion = "24.11";

  # Host-specific sops default file
  sops.defaultSopsFile = ../../secrets/acfs.yaml;
}
```

### Pattern 4: Secrets with Owner Assignment

**What:** Each secret in sops-nix gets explicit `owner` and `group` settings matching the service that needs it. Services reference secrets by their `/run/secrets/` path.
**When to use:** For any service that needs credentials, API keys, or tokens.
**Trade-offs:** Requires knowing which user each service runs as; but prevents permission errors at runtime.

**Example:**
```nix
# modules/secrets.nix
{ config, ... }: {
  sops.age.sshKeyPaths = [ "/etc/ssh/ssh_host_ed25519_key" ];

  sops.secrets."docker/registry-auth" = {
    owner = "root";
    group = "docker";
    mode = "0440";
  };

  sops.secrets."syncthing/api-key" = {
    owner = "admin";
  };

  sops.secrets."cass/api-token" = {
    owner = "admin";
  };
}
```

### Pattern 5: User Systemd Services via Home-Manager

**What:** User-level long-running services (CASS indexer, Syncthing) are declared as `systemd.user.services` in home-manager, not as system-level services.
**When to use:** For services that should run as a specific user, not root. Home-manager's `services.syncthing` is a first-class option.
**Trade-offs:** User services start after user login (or with lingering enabled). System services start at boot. For a server, enable lingering (`loginctl enable-linger admin`) so user services start at boot without login.

**Example:**
```nix
# home/services/cass-indexer.nix
{ pkgs, config, ... }: {
  systemd.user.services.cass-indexer = {
    Unit = {
      Description = "CASS Indexer Service";
      After = [ "network-online.target" ];
    };
    Service = {
      ExecStart = "${pkgs.writeShellScript "cass-indexer" ''
        export API_TOKEN=$(cat /run/secrets/cass/api-token)
        exec /home/admin/cass/indexer --config /home/admin/cass/config.toml
      ''}";
      Restart = "on-failure";
      RestartSec = 10;
    };
    Install = {
      WantedBy = [ "default.target" ];
    };
  };
}
```

### Pattern 6: Disko for Declarative Disk Layout

**What:** Disk partitioning is declared in Nix and applied by nixos-anywhere during initial deployment. The same config generates fstab entries for subsequent boots.
**When to use:** Always for new deployments. Disko makes disk layout reproducible and version-controlled.
**Trade-offs:** Disko config is only applied during initial install (or explicitly). Changing partitions later requires careful migration.

**Example:**
```nix
# hosts/acfs/disko-config.nix
{ ... }: {
  disko.devices = {
    disk.main = {
      device = "/dev/sda";  # Verify with lsblk on Contabo
      type = "disk";
      content = {
        type = "gpt";
        partitions = {
          boot = {
            size = "1M";
            type = "EF02";   # BIOS boot partition
          };
          ESP = {
            name = "ESP";
            size = "512M";
            type = "EF00";
            content = {
              type = "filesystem";
              format = "vfat";
              mountpoint = "/boot";
              mountOptions = [ "umask=0077" ];
            };
          };
          swap = {
            size = "8G";
            content = {
              type = "swap";
              randomEncryption = true;
            };
          };
          root = {
            size = "100%";
            content = {
              type = "filesystem";
              format = "ext4";
              mountpoint = "/";
            };
          };
        };
      };
    };
  };
}
```

## Data Flow

### Deployment Flow (Initial Install via nixos-anywhere)

```
[Local machine]
    │
    │  nix run github:nix-community/nixos-anywhere -- \
    │    --flake .#acfs root@<contabo-ip>
    │
    ├── 1. SSH into Contabo VPS (running Ubuntu/rescue)
    ├── 2. kexec into NixOS installer environment
    ├── 3. disko: partition + format /dev/sda per disko-config.nix
    ├── 4. nixos-install: build system closure from flake
    ├── 5. Copy closure to target
    ├── 6. Reboot into NixOS
    │
    └── [NixOS activation on first boot]
         ├── Generate SSH host keys (age key source for sops)
         ├── Create users/groups
         ├── Decrypt sops secrets to /run/secrets/
         ├── Run activation scripts
         └── Start systemd services (including Docker containers)
```

**Critical note:** On first deploy, the host has no SSH host key yet (disko wipes the disk). sops-nix cannot decrypt secrets until host keys exist. Two strategies:
1. **Pre-seed the host key** by passing `--extra-files` to nixos-anywhere with the host's SSH key
2. **Two-phase deploy:** First deploy without secrets, then add the host's public key to `.sops.yaml`, re-encrypt, and `nixos-rebuild switch`

### Configuration Rebuild Flow (Ongoing Updates)

```
[Local machine: edit Nix files]
    │
    │  git commit + push
    │
    │  nixos-rebuild switch --flake .#acfs --target-host root@<ip>
    │  (or SSH in and run locally)
    │
    └── [On target: NixOS activation]
         ├── Evaluate flake (lazy: only touched options computed)
         ├── Build changed derivations
         ├── Switch system profile
         ├── sops-nix: decrypt secrets to /run/secrets.d/<N>/
         │    └── Atomic symlink: /run/secrets/ -> /run/secrets.d/<N>/
         ├── Run activation scripts
         ├── Restart changed systemd units
         │    ├── System services (Docker containers, fail2ban, etc.)
         │    └── User services (CASS indexer, Syncthing)
         └── Done (previous generation preserved for rollback)
```

### Secrets Data Flow

```
[Developer machine]
    │
    │  ~/.config/sops/age/keys.txt  (user's private age key)
    │
    │  sops secrets/acfs.yaml       (edit secrets in cleartext)
    │
    ├── .sops.yaml routes encryption to:
    │    ├── User's age public key (so developer can edit)
    │    └── Host's age public key (so server can decrypt)
    │
    │  git commit secrets/acfs.yaml  (encrypted at rest in git)
    │
    └── [On NixOS host at activation time]
         │
         │  /etc/ssh/ssh_host_ed25519_key  (host private key)
         │    └── Converted to age key by sops-nix
         │
         ├── sops-nix decrypts secrets/acfs.yaml
         ├── Creates /run/secrets/<secret-name> for each secret
         ├── Sets owner/group/mode per sops.secrets.<name> config
         │
         └── Services read from /run/secrets/<path>
              ├── Docker containers via environment files
              ├── Systemd services via EnvironmentFile or ExecStart
              └── User services via /run/secrets/ paths
```

### Module Dependency Flow (What Must Exist Before What)

```
flake.nix (inputs)
    │
    ├── disko (disk layout)
    │     └── Must exist BEFORE first nixos-anywhere deploy
    │
    ├── hosts/acfs/default.nix (composition root)
    │     ├── hardware-configuration.nix
    │     │     └── Generated by nixos-generate-config or nixos-anywhere
    │     └── disko-config.nix
    │
    ├── modules/base.nix (nix settings)
    │     └── No dependencies; foundational
    │
    ├── modules/boot.nix (bootloader)
    │     └── Depends on: disko (must know partition layout)
    │
    ├── modules/users.nix (user accounts)
    │     └── Depends on: base (nix settings)
    │     └── Must exist BEFORE secrets (sops needs user for owner)
    │     └── Must exist BEFORE home-manager (needs user to exist)
    │
    ├── modules/networking.nix (firewall, SSH)
    │     └── Depends on: base
    │     └── Must exist BEFORE services (services need network)
    │
    ├── modules/secrets.nix (sops-nix declarations)
    │     └── Depends on: users (for owner assignment)
    │     └── Must exist BEFORE docker (containers need secrets)
    │     └── Must exist BEFORE home/services (user services need secrets)
    │
    ├── modules/docker.nix (containers)
    │     └── Depends on: networking (port exposure)
    │     └── Depends on: secrets (container credentials)
    │     └── Depends on: users (docker group membership)
    │
    ├── modules/services.nix (system services)
    │     └── Depends on: networking, secrets
    │
    └── home/* (user config + user services)
          └── Depends on: users, secrets
          └── Depends on: networking (for Syncthing)
```

**Important:** NixOS module evaluation is lazy -- it does not follow a strict "build order." All modules are merged into a single attribute set and evaluated on demand. The dependency arrows above represent *logical* dependencies (what must be configured for something else to work), not evaluation order. The NixOS module system handles merging automatically.

The activation-time ordering is where real sequencing matters:
1. Users/groups created
2. sops-nix decrypts secrets (with `neededForUsers = true` for password secrets)
3. Activation scripts run
4. Systemd services start (respecting `After=` and `Requires=` directives)

## Scaling Considerations

| Scale | Architecture Adjustments |
|-------|--------------------------|
| 1 host (current: acfs) | Single `nixosConfigurations.acfs` in flake. All modules in flat `modules/` directory. Simple and maintainable. |
| 2-5 hosts | Add `hosts/<name>/` per host. Shared modules stay in `modules/`. Host-specific modules go in `hosts/<name>/modules/` if needed. Add host keys to `.sops.yaml`. Consider `colmena` or `deploy-rs` for multi-host deployment. |
| 5-20 hosts | Introduce role-based composition: `profiles/` directory with `profiles/webserver.nix`, `profiles/database.nix`. Hosts compose from profiles. Consider splitting secrets into per-host files with separate `.sops.yaml` creation rules. |

### Scaling Priorities

1. **First bottleneck: secrets management at 3+ hosts.** Each host needs its age key in `.sops.yaml`, and re-encrypting after key changes becomes tedious. Mitigation: per-host secret files from the start (already recommended above).
2. **Second bottleneck: deployment orchestration at 5+ hosts.** Running `nixos-rebuild` per-host doesn't scale. Mitigation: adopt `colmena` or `deploy-rs` when adding a second host, not after five.

## Anti-Patterns

### Anti-Pattern 1: Monolithic configuration.nix

**What people do:** Put everything in one giant `configuration.nix` file.
**Why it's wrong:** Files over 200 lines become hard to navigate, review, and debug. Changes to Docker config require scrolling past unrelated networking config.
**Do this instead:** Module-per-concern in `modules/` directory. Each file under 100 lines. `default.nix` aggregates imports.

### Anti-Pattern 2: Using activation scripts for one-time setup

**What people do:** Put `git clone` or large downloads in `system.activationScripts`.
**Why it's wrong:** Activation scripts run on every `nixos-rebuild switch`. They must be idempotent and fast. Cloning a repo is neither fast nor reliably idempotent.
**Do this instead:** Use a oneshot systemd service with `RemainAfterExit=true` and a state file check. Or use `tmpfiles.d` rules for directory creation and symlinks, reserving activation scripts only for truly trivial operations (mkdir, chmod).

### Anti-Pattern 3: Hardcoding secrets in Nix expressions

**What people do:** Put API keys, passwords, or tokens directly in `.nix` files.
**Why it's wrong:** Nix expressions are copied to the Nix store, which is world-readable (`/nix/store/`). Secrets in Nix expressions are visible to any user on the system.
**Do this instead:** Use sops-nix. Declare secrets, reference them by `/run/secrets/<path>`. Secrets never enter the Nix store.

### Anti-Pattern 4: Running home-manager standalone on NixOS

**What people do:** Use `home-manager switch` separately from `nixos-rebuild switch`.
**Why it's wrong:** Two separate commands to keep in sync. System and user config can drift. Forgetting one means partial configuration.
**Do this instead:** Import `home-manager.nixosModules.home-manager` as a NixOS module. Both system and user config deploy atomically with `nixos-rebuild switch`.

### Anti-Pattern 5: Forgetting git add for new files

**What people do:** Create a new `.nix` file, run `nixos-rebuild`, get a confusing error.
**Why it's wrong:** Nix flakes only see git-tracked files. Untracked files are invisible to the evaluator.
**Do this instead:** Always `git add` new files before building. Make this a habit or add a pre-build check.

### Anti-Pattern 6: Using docker-compose alongside NixOS

**What people do:** Install docker-compose and manage containers outside of NixOS configuration.
**Why it's wrong:** Creates two sources of truth for container state. NixOS doesn't know about compose-managed containers; they don't benefit from NixOS rollback, declarative management, or systemd integration.
**Do this instead:** Use `virtualisation.oci-containers` in NixOS. Each container becomes a systemd service. If you have an existing `docker-compose.yaml`, use `compose2nix` to convert it.

## Integration Points

### External Services

| Service | Integration Pattern | Notes |
|---------|---------------------|-------|
| Contabo VPS (Ubuntu rescue) | nixos-anywhere via SSH | Initial deploy wipes disk; need SSH access to rescue/Ubuntu. Contabo hostname uses dots (RFC 1035 violation) -- set valid hostname before install. |
| Docker Hub / registries | `virtualisation.oci-containers` pulls images | Registry auth secrets via sops-nix if using private registries |
| Syncthing relay/discovery | Home-manager `services.syncthing` | Opens ports 22000/TCP and 21027/UDP; add to `networking.firewall.allowedTCPPorts` |
| Git remotes | Activation scripts or oneshot services | Clone repos for CASS and other tools; use SSH keys managed by sops-nix |

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| System modules <-> Home-manager | `specialArgs` / `extraSpecialArgs` to pass inputs; no direct config sharing | System creates user; home-manager configures user's environment. They don't share options directly. |
| sops-nix <-> Services | Filesystem path (`/run/secrets/<name>`) | Services read secret files. sops-nix sets permissions. Use `restartUnits` option to restart services when secrets change. |
| sops-nix <-> Users | `neededForUsers = true` for password secrets | Password secrets must decrypt before user creation. Other secrets decrypt after users exist. |
| Docker containers <-> Host | Port mappings, volume mounts, environment variables | Containers are isolated. Use `extraOptions = ["--network=host"]` only when necessary. Prefer explicit port mapping. |
| User systemd services <-> System | `loginctl enable-linger <user>` | Required for user services to start at boot without login session. Set in `users.users.<name>.linger = true`. |

## Build Order for Implementation

Based on dependency analysis, implement in this order:

### Phase 1: Skeleton (must exist before anything else)
1. `flake.nix` -- inputs + empty nixosConfigurations
2. `hosts/acfs/disko-config.nix` -- disk layout
3. `modules/base.nix` -- nix settings, flake config
4. `modules/boot.nix` -- bootloader config
